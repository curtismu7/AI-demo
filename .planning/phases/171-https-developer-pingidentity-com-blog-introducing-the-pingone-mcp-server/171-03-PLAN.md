## Deploy to Production: Security Hardening Guide

> **Important:** This section covers the BX Finance demo deployment. For production deployments, add compliance layers (HIPAA, PCI-DSS, SOC 2) and conduct a security audit.

### Pre-Deployment Checklist

- [ ] **PingOne Configuration**
  - [ ] OAuth app created with proper redirect URIs (no localhost in production)
  - [ ] PKCE enabled (auto-enforced by PingOne, but verify in admin console)
  - [ ] Refresh token rotation enabled
  - [ ] Audience (aud) claim set correctly for MCP server resource
  - [ ] Scopes properly scoped (banking:read, banking:write, not wildcard)

- [ ] **Session Management**
  - [ ] Redis session store configured (Upstash Redis on Vercel, or equivalent for on-prem)
  - [ ] Session TTL set (recommend 15 minutes for access, 7 days for refresh)
  - [ ] Session cleanup job running (automatic Redis key expiration)

- [ ] **Cookie Security**
  - [ ] httpOnly flag enabled (no JavaScript access)
  - [ ] Secure flag enabled (HTTPS only)
  - [ ] SameSite=Strict or SameSite=Lax (prevent CSRF)
  - [ ] Domain set correctly (no overly broad domain)

- [ ] **Token Handling**
  - [ ] Tokens NEVER sent to frontend JavaScript
  - [ ] Tokens stored in httpOnly cookie only
  - [ ] Token validation performed server-side on all API calls
  - [ ] Signature validation enabled (see: SKIP_TOKEN_SIGNATURE_VALIDATION in REGRESSION_PLAN.md — set to false in production)

- [ ] **CIBA / HITL Flow**
  - [ ] Push notification service configured (PingOne Notifications)
  - [ ] Polling timeout set (e.g., 300 seconds max wait)
  - [ ] Fallback to inline auth if notifications fail

- [ ] **Logging & Monitoring**
  - [ ] Audit logs enabled: who initiated action, what action, approval status
  - [ ] No sensitive data in logs (no tokens, passwords, PII)
  - [ ] Centralized logging (e.g., CloudWatch, Datadog, ELK stack)
  - [ ] Alerting: Monitor for auth failures, token exchange errors, HITL timeouts

### On-Premises Deployment (Docker/Kubernetes)

For on-prem or self-hosted Kubernetes:

**Key changes from Vercel:**

1. **Session store:** Deploy Redis separately (not Vercel KV)
2. **HTTPS:** Use Let's Encrypt or corporate CA; ensure TLS 1.3+
3. **Database:** Swap SQLite for PostgreSQL or managed database
4. **Monitoring:** Add Prometheus/Grafana for metrics, Jaeger for tracing
5. **Secrets:** Use HashiCorp Vault, AWS Secrets Manager, or Kubernetes Secrets (not env files)

**Example Kubernetes setup:**

See banking-demo repository, `/docker-kubernetes-deployment` phase — Helm charts and examples for full containerized setup.

### Known Production Issues & Hard Guards (from REGRESSION_PLAN.md)

**hard guard: SKIP_TOKEN_SIGNATURE_VALIDATION disabled**

In `banking_api_server/middleware/tokenValidation.js`, ensure:

```javascript
if (process.env.SKIP_TOKEN_SIGNATURE_VALIDATION === 'true') {
  console.error('FATAL: SKIP_TOKEN_SIGNATURE_VALIDATION enabled in production! Exiting.');
  process.exit(1);  // Hard exit, not just warning
}
```

**Issue:** If an operator accidentally enables SKIP_TOKEN_SIGNATURE_VALIDATION in production (e.g., for debugging), the server will accept invalid tokens. The hard guard (process.exit) ensures the server fails fast rather than silently running insecurely.

### Monitoring Checklist

After deployment, monitor:

```
1. **Auth Success Rate:** OAuth login, CIBA approval, HITL approval rates
2. **Token Exchange Performance:** Latency from exchange request to MCP token issued (target: <100ms)
3. **Session Errors:** "readonly database" errors (indicates session store misconfiguration)
4. **MCP Tool Invocation:** Track which tools are called, success rate, performance
5. **HITL Approval Times:** Average time for user to approve transaction (should be <2 mins)
```

**Alert thresholds:**
- Auth success rate < 95% → investigate
- Token exchange mean latency > 200ms → check Redis/database
- MCP tool errors > 5% → check MCP server logs
```

## Best Practices & Next Steps

### Key Takeaways

1. **Session Custodian Pattern (BFF)**
   - API tokens never leave the backend
   - Frontend receives session cookie (httpOnly); all API calls authenticated via session
   - BFF validates every request, then calls MCP server with properly scoped token
   - ✅ Prevents XSS from leaking tokens
   - ✅ Allows token rotation without frontend coordination

2. **RFC 8693 Token Exchange**
   - Use 1-exchange for rapid prototyping (user token → resource token)
   - Use 2-exchange for production (user token + agent token → resource token with `act` claim)
   - `act` claim enables audit trail: "User X approved, Agent Y executed"
   - Always validate `sub` (subject) and scopes on resource side

3. **HITL (Human-In-The-Loop) Strategy**
   - HITL is not a UI afterthought; design it into your transaction model
   - Use PingOne CIBA for out-of-band approval (mobile notification)
   - Use inline auth for frictionless HITL (user authenticates without page nav)
   - Set HITL thresholds based on risk (Phase 170: all transfers, regardless of amount)

4. **Delegation vs Role-Based Access Control (RBAC)**
   - **Delegation** (RFC 8693): "User A authorizes User/Agent B to act on their behalf"
   - **RBAC**: "What is User B allowed to do?" (separate from delegation)
   - Combine both: Delegation (who acts) + RBAC (what they can act on) = strong auth model
   - Example: Agent B can read spending (RBAC scope banking:read) and act on behalf of User A (delegation `act` claim)

### Common Pitfalls (and How to Avoid Them)

| Pitfall | Why It's Bad | Solution |
|---------|------------|----------|
| Storing tokens in browser localStorage | Vulnerable to XSS; attacker steals token | Use httpOnly cookies, serverside session store |
| Hardcoding API keys in frontend code | Keys committed to git, leaked in browser | Use environment variables on backend only; never expose to frontend |
| Skipping token signature validation | Attacker can forge tokens | Always validate (never skip in production; see SKIP_TOKEN_SIGNATURE_VALIDATION hard guard) |
| Using same token for user and agent | No audit trail; can't distinguish user-initiated vs agent-initiated | Use RFC 8693 2-exchange; get separate `act` claim |
| HITL as an afterthought | UI gets complicated; approvals are slow or forgotten | Design HITL into transaction model from day 1 (use CIBA or inline auth natively) |

### Resources & Next Steps

**RFC References (for further reading):**
- [RFC 8693: OAuth 2.0 Token Exchange](https://tools.ietf.org/html/rfc8693) — The delegation standard we used
- [RFC 7519: JSON Web Tokens (JWT)](https://tools.ietf.org/html/rfc7519) — Token format deep-dive
- [RFC 9396: OAuth 2.0 Rich Authorization Requests (RAR)](https://tools.ietf.org/html/rfc9396) — Advanced consent/HITL
- [CIBA Core](https://openid.net/specs/openid-client-initiated-backchannel-authentication-core-1_0.html) — Out-of-band authentication spec

- [Local development](./README.md) — Run the demo locally with `./run-bank.sh`

### Try It Yourself

1. **Clone:** `git clone https://github.com/curtismu7/banking-demo.git`
2. **Install:** `npm install` in UI and server directories
3. **Configure:** Set PingOne environment variables (see `.env.example`)

   - Try a transfer (2-exchange: user + agent tokens → MCP token with `act` claim)
   - Approve via CIBA notification or inline auth
   - Check token chain visualization (tokens decoded and claims displayed)

### Call to Action

**Secure AI agent integration is not a nice-to-have — it's a must-have.** As AI becomes embedded in banking, fintech, healthcare, and other sensitive domains, the ability to safely delegate access while maintaining user control and audit trails is critical.

The BX Finance demo and this post show that implementing RFC 8693 token exchange, CIBA, and HITL is not only possible but can be done transparently — users don't see the complexity, implementers can follow clear patterns, and auditors can trace every action to its authorization.
