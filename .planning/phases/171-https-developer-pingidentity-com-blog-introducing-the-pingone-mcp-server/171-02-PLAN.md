**Use case:** Agent requests approval for a sensitive operation; user approves on a second device (mobile phone) without returning to the browser.

## RFC 8693 Token Exchange in Action: Safe Delegation for AI Agents

### The Problem: How Do We Give an AI Agent Access to a Banking API Safely?

Traditional answer: "Give the agent an API key." Problems:
- No user context: API key is not tied to the user who authorized the agent
- Revocation is global: Revoking one agent's access revokes all agents
- Auditability: Hard to distinguish user-initiated vs agent-initiated actions
- Token lifetime: API keys often live forever

Modern answer (RFC 8693): Exchange the user's OAuth token for an agent-specific token with delegation claims.

### The Safe Flow: RFC 8693 Token Exchange

**Diagram placeholder:** [DIAGRAM: Side-by-side comparison — Traditional API Key vs RFC 8693 exchange]

RFC 8693 defines a standard HTTP endpoint for token exchange:

```
POST /token HTTP/1.1
grant_type=urn:ietf:params:oauth:grant-type:token-exchange
subject_token={user_token}
subject_token_type=urn:ietf:params:oauth:token-type:access_token
resource={MCP_SERVER_RESOURCE_ID}
audience={MCP_SERVER_AUDIENCE}
requested_token_use=access_token
```

**What happens:**
1. App holds user's access token (from OAuth login)
2. App wants to delegate to agent / MCP server
3. App calls token exchange endpoint with:
   - `subject_token`: User's access token (proof: user is authenticated)
   - `resource`: MCP server's resource identifier
   - Optional `actor_token`: Agent's service credentials (for 2-exchange)
4. Authorization server validates both tokens
5. Issues new access token with delegation claims

**Code example:** Show exchange endpoint from `banking_api_server/routes/auth/oauth/tokenExchange.js` (POST /token) (20-25 lines)

### 1-Exchange: User Stream Only

Use case: Agent acts directly on behalf of the user (no intermediate agent identity).

```
User Token ─── Exchange ──→ MCP Token
                |
                └─ MCP token has `sub` (user), no `act` claim
```

**Example:**

```json
{
  "subject": "user-id-123",
  "scope": "banking:read banking:write",
  "aud": "https://mcp.banking.api"
}
```

**When to use:** Simple scenarios where agent is purely delegating user actions. No separate agent context needed.

**Code walkthrough:** Show actual exchange request/response from `banking_api_server/` test logs or curl example (10-15 lines)

### 2-Exchange: User + Agent Context (RFC 8693 with Authorization Claims)

Use case: Agent acts on user's behalf, but we want to track that an agent was involved and ensure the agent is authorized.

```
User Token ──┐
             ├─ Exchange ──→ MCP Token (with delegation info)
Agent Token ─┤
             └─ MCP token has `sub` (user), `act` (agent), optional `may_act` (agent's permissions)
```

**Example MCP token (decoded JWT):**

```json
{
  "sub": "user-id-123",
  "act": {
    "sub": "agent-service-account-456"
  },
  "may_act": {
    "policies": ["banking:read", "banking:write", "limits:check"]
  },
  "scope": "banking:read banking:write",
  "aud": "https://mcp.banking.api",
  "exp": 1704067200
}
```

**Key claims:**
- `sub`: The actual user (subject of the operation)
- `act`: The agent performing the operation (authorization chain)
- `may_act`: What the agent is authorized to do

**When to use:** Production environments where you need clear delegation tracking, audit trails, and agent authorization control.

**Code walkthrough:** Show 2-exchange request with `actor_token` from test or log (15-20 lines)

### Comparison: 1-Exchange vs 2-Exchange

| Aspect | 1-Exchange | 2-Exchange |
|--------|-----------|-----------|
| **Token components** | User only | User + Agent |
| **Use case** | Simple delegation | Tracked, auditable delegation |
| **Privacy** | `sub` identifies user, no agent tracking | `act` claim reveals agent identity |
| **Auditability** | User actions indistinguishable from direct | Agent actions clearly marked |
| **Agent authorization** | Inherits user's permissions | Can enforce additional constraints via `may_act` |
| **Revocation** | Revoke user token → all flows stop | Can revoke agent without affecting user auth |

**When implementing:**
- Use 1-exchange for demo/prototype
- Use 2-exchange for production (compliance, audit trails, separate agent revocation)

```

## BX Finance Case Study: Putting It All Together

### The Scenario

BX Finance is a fictional online bank building an AI-powered banking assistant. Users can:
1. Ask the agent to analyze spending (read-only)
2. Request the agent to recommend transfers (advisory, requires HITL)
3. Ask the agent to execute transfers (requires HITL + explicit approval)

The challenge: How do we let the agent invoke MCP tools safely without sharing raw API credentials?

### The Architecture

**Three-tier stack:**

```
┌─────────────────────────────────────────────────────────┐
│ Browser (SPA)                                          │
│ • User logs in (Authorization Code + PKCE)             │
│ • Token stored in httpOnly cookie via BFF              │
│ • UI presents agent chat interface                      │
└────────────────────┬────────────────────────────────────┘
                     │ HTTPS
┌────────────────────▼────────────────────────────────────┐
│ BFF (Backend For Frontend / banking_api_server)        │
│ • Guard: All user requests validated                   │
│ • Token custodian: Holds user's access token           │
│ • Delegation: Exchanges user token for MCP token (2-ex)│
│ • Consent: Checks transaction amount, HITL threshold   │
│ • Proxy: Forwards validated requests to MCP server     │
└────────────────────┬────────────────────────────────────┘
                     │ WebSocket
┌────────────────────▼────────────────────────────────────┐
│ MCP Server (banking_mcp_server)                         │
│ • Tool registry: banking:read, banking:write tools     │
│ • Auth check: Validates MCP token claims (sub, act)    │
│ • Tools: Account analysis, transaction simulation      │
│ • Challenge: Returns step_up_required if HITL needed   │
└─────────────────────────────────────────────────────────┘
```

### The Flows in Action

**Scenario 1: Analyze Spending (1-Exchange, Read-Only)**

1. User asks: "Analyze my spending this month"
2. Browser sends message to BFF (authenticated session)
3. BFF checks operation: "banking:read only" → no HITL needed
4. BFF exchanges user token for MCP token (1-exchange)
5. BFF calls MCP server: `analyze_spending` tool with MCP token
6. MPC server decodes token, checks `sub` (user) and scopes
7. MCP server queries banking dataset for user's transactions
8. MCP server returns: "You spent $5,000 this month: dining $1,200, groceries $800, ..."
9. Browser displays result in chat

**Tokens used:**
- User's access token (held by BFF)
- MCP token (1-exchange, scope: banking:read)

---

**Scenario 2: Execute Transfer with HITL Approval**

1. User asks: "Transfer $5,000 to savings account"
2. Browser sends message to BFF
3. BFF checks operation: "transfer $5000" → hits HITL threshold (Phase 170)
4. BFF returns: `step_up_required: "HITL"` with challenge_id
5. UI presents inline login form ("Verify your identity")
6. User enters password + biometric
7. BFF verifies approval with Core (PingOne)
8. Core confirms: user approved transfer
9. BFF exchanges user + agent tokens for MCP token (2-exchange)
10. BFF calls MCP: `execute_transfer` with MCP token + HITL proof
11. MCP server decodes token, checks `sub` (user) + `act` (agent) + constraint (HITL approved)
12. MCP executes transfer
13. Browser shows: "Transfer submitted for processing"

**Tokens used:**
- User's access token (core auth)
- Agent's service token (proves agent is within policy)
- MCP token (2-exchange, subject: user, actor: agent, constraints: HITL approved)

---

### Key Security Principles in Action

1. **Token Custody:** BFF holds user's token; agent never sees it
2. **Delegation:** User token → MCP token via RFC 8693 exchange
3. **Tracking:** `act` claim in MCP token attributes action to agent
4. **Auditability:** HITL approval is tracked in token request; audit logs show "User + Agent executed transfer"
5. **Revocation:** BFF can revoke 1-exchange or 2-exchange access independently

Clone the repo and run locally:

```bash

cd banking-demo
npm install
./run-bank.sh
open https://localhost:4000
```

Then:
1. Log in as user: `bankuser` / prompt for password
2. Navigate to dashboard
3. Open agent chat
4. Try analyze spending 
5. Try a transfer (you'll hit HITL)
6. Approve on notification / inline
7. Look at token chain visualization to see the exchanged tokens

**Public demo (no PingOne setup needed):** https://banking-demo-puce.vercel.app (limited features, but shows the flows)
