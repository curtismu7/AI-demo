# RFC 8693 Token Exchange in Action

## The Problem: How Do Tokens Get to the AI Agent?

Consider this scenario: a user is logged into the banking app (they have an access token in their BFF session). They ask the AI agent to check account balances. The MCP server needs a token to call the banking API on the user's behalf.

**The wrong answer:** Pass the user's access token directly to the MCP server. This violates the principle of least privilege — the user's token has all their scopes, and the MCP server only needs `banking:accounts:read`.

**The right answer:** RFC 8693 Token Exchange. The BFF exchanges the user's token for a new, narrowly-scoped token targeted at the MCP server's audience. The original token never leaves the BFF.

## What Is RFC 8693?

[RFC 8693](https://datatracker.ietf.org/doc/html/rfc8693) defines a standard OAuth 2.0 grant type for exchanging one security token for another. The key parameters:

| Parameter | Value | Purpose |
|-----------|-------|---------|
| `grant_type` | `urn:ietf:params:oauth:grant-type:token-exchange` | Identifies the exchange grant |
| `subject_token` | User's access token | Who the action is for |
| `subject_token_type` | `urn:ietf:params:oauth:token-type:access_token` | Type of subject token |
| `actor_token` | Agent's client-credentials token (optional) | Who performs the action |
| `audience` | MCP server resource URI | Where the token will be used |
| `scope` | `banking:accounts:read` | What the token can do |

## Pattern 1: Single Exchange (User Context Only)

The simplest pattern. The BFF exchanges the user's access token for an MCP-scoped token. No agent identity in the resulting token.

**When to use:** Read-only operations, user-initiated actions, simple tool calls where you don't need to audit which agent performed the action.

```javascript
// From banking_api_server/services/oauthService.js — performTokenExchange()
async performTokenExchange(subjectToken, audience, scopes) {
  const body = new URLSearchParams({
    grant_type: 'urn:ietf:params:oauth:grant-type:token-exchange',
    subject_token: subjectToken,
    subject_token_type: 'urn:ietf:params:oauth:token-type:access_token',
    requested_token_type: 'urn:ietf:params:oauth:token-type:access_token',
    audience: audience,         // e.g. 'https://mcp-server.pingdemo.com'
    scope: scopes.join(' '),    // e.g. 'banking:accounts:read'
    client_id: this.config.clientId,
  });

  const response = await axios.post(
    this.config.tokenEndpoint,
    body.toString(),
    { headers: { 'Content-Type': 'application/x-www-form-urlencoded' } }
  );
  
  return response.data.access_token;  // Narrowly-scoped MCP token
}
```

**Resulting token claims:**

```json
{
  "sub": "user-uuid-1234",
  "aud": "https://mcp-server.pingdemo.com",
  "scope": "banking:accounts:read",
  "iss": "https://auth.pingone.com/{envId}/as",
  "exp": 1700000000
}
```

The `sub` claim is the user. The `aud` is the MCP server. The scope is narrowed to exactly what the tool needs. No `act` claim because there's no actor token.

## Pattern 2: Double Exchange (User + Agent Context)

The full delegation pattern. Two tokens go in — the user's access token (subject) and the agent's client-credentials token (actor). The resulting token represents the user but carries `act` claims identifying which agent is operating.

**When to use:** Write operations, transfers, any action where you need to audit *which agent* performed *which action* on behalf of *which user*.

```javascript
// From banking_api_server/services/oauthService.js — performTokenExchangeWithActor()
async performTokenExchangeWithActor(subjectToken, actorToken, audience, scopes) {
  const body = new URLSearchParams({
    grant_type: 'urn:ietf:params:oauth:grant-type:token-exchange',
    subject_token: subjectToken,
    subject_token_type: 'urn:ietf:params:oauth:token-type:access_token',
    actor_token: actorToken,                                        // NEW
    actor_token_type: 'urn:ietf:params:oauth:token-type:access_token', // NEW
    requested_token_type: 'urn:ietf:params:oauth:token-type:access_token',
    audience: audience,
    scope: scopes.join(' '),
    client_id: this.config.clientId,
  });

  const response = await axios.post(
    this.config.tokenEndpoint,
    body.toString()
  );
  
  return response.data.access_token;
}
```

**Resulting token claims:**

```json
{
  "sub": "user-uuid-1234",
  "aud": "https://mcp-server.pingdemo.com",
  "scope": "banking:accounts:read banking:transactions:write",
  "act": {
    "sub": "agent-client-id-5678",
    "client_id": "banking-ai-agent"
  },
  "iss": "https://auth.pingone.com/{envId}/as",
  "exp": 1700000000
}
```

The critical difference: the `act` claim. This means:
- **`sub`** = the user (whose data is being accessed)
- **`act.sub`** = the agent (who is performing the action)
- Every API call is auditable: "Agent X accessed User Y's accounts at time Z"

## ID Token Exchange Variant

There's a third variant worth noting: exchanging the user's **ID token** (not access token) as the subject. This is useful when the agent should never hold the user's access token — even transiently within the BFF.

```javascript
// From banking_api_server/services/oauthService.js — performTokenExchangeFromIdToken()
const body = new URLSearchParams({
  grant_type: 'urn:ietf:params:oauth:grant-type:token-exchange',
  subject_token: idToken,
  subject_token_type: 'urn:ietf:params:oauth:token-type:id_token',  // ID token, not access token
  requested_token_type: 'urn:ietf:params:oauth:token-type:access_token',
  audience: audience,
  scope: scopeStr,
});
```

**Why?** The ID token proves *who the user is* but has no API access itself. PingOne issues a fresh access token scoped to the MCP server based on the user's identity — not by copying scopes from an existing access token.

## Comparison: When to Use Which

| Pattern | Subject Token | Actor Token | `act` Claim | Use Case |
|---------|--------------|-------------|-------------|----------|
| **1-Exchange** | User access token | None | No | Read-only tools, simple queries |
| **2-Exchange** | User access token | Agent CC token | Yes | Write operations, auditable actions |
| **ID Token Exchange** | User ID token | None or Agent | Varies | Zero-trust: agent never holds user AT |

**Rule of thumb:** If the operation modifies data or moves money, use 2-Exchange. If it reads data, 1-Exchange is sufficient. If you want maximum isolation, use ID Token Exchange.
