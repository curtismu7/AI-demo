  <action>
Update agent service to:
- Read fallback chain from req.session.langchain_config.fallback_order (e.g., ['lm_studio', 'groq', 'anthropic'])
- When initializing agent LLM, iterate through chain:
  - Try first provider
  - If provider initialization fails (timeout, unreachable, bad key), log warning and try next
  - Continue until one succeeds or all exhaust
  - If all exhaust, return error (don't silently fall through to hardcoded default)
- Log which provider actually initialized: `console.log('[agent] LLM initialized: ' + providerUsed + ' (' + reason + ')')`
- Expose chosen provider in agent response metadata for debugging

Example logic:
```javascript
const fallbackOrder = req.session.langchain_config?.fallback_order || ['lm_studio', 'groq'];
let llm = null;
for (const provider of fallbackOrder) {
  try {
    llm = await initializeLLM(provider, req.session.langchain_config);
    console.log(`[agent] LLM initialized: ${provider}`);
    break;
  } catch (err) {
    console.warn(`[agent] Provider ${provider} failed, trying next: ${err.message}`);
  }
}
if (!llm) throw new Error('All LLM providers exhausted');
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API | Untrusted API key input (could contain secrets intended for different service) |
| API → Third-party LLM | External provider connectivity + error messages could leak config details |
| Session → File | Fallback chain order is user-scoped (not system-wide) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-176-01 | Spoofing | API key input field | Mitigate | Validate API key format before storing (check length, charset); never echo key in responses |
| T-176-02 | Information Disclosure | Provider status checks | Mitigate | Health check timeouts at 3s to prevent time-based enumeration; don't leak "key is invalid" vs "service down" distinction in error message (just return status) |
| T-176-03 | Tampering | Fallback chain reordering | Accept | Session-scoped order; user themselves can trivially inspect/modify via developer tools anyway |

</threat_model>
