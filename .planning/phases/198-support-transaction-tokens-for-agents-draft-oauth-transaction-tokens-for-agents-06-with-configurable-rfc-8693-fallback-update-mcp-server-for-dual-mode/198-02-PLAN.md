**Token Format Detection Strategy:**
RFC 8693 JWT characteristics:
- Has `act` claim (actor/agent identity)
- Has `sub` claim (user/subject)
- Standard JWT format

Transaction Tokens JWT characteristics:
- Has `txn_id` claim (transaction identifier)
- Has `agent_id` or `act` claim (agent identity)
- May have `txn_scope` or `intent` claim
- Standard JWT format (can add custom claims)

**Patterns to Follow:**
- Error handling: Throw errors from validation; authMiddleware catches and closes connection
- Logging: Use consistent format (timestamp, operation, status)
- TypeScript: Use proper types (Token interfaces for JWT payload)

**`function detectTokenType(payload: any): 'rfc_8693' | 'transaction_tokens' | 'unknown'`**:
- Check for Transaction Tokens indicators first:
  - If has `txn_id` claim → 'transaction_tokens'
  - If has `txn_scope` claim → 'transaction_tokens'
- Check for RFC 8693 indicators:
  - If has `act` claim with `sub` field → 'rfc_8693'
- If neither → 'unknown'
- Add logic:
  ```typescript
  if (payload.txn_id || payload.txn_scope) {
    return 'transaction_tokens';
  }
  if (payload.act && typeof payload.act === 'object' && payload.act.sub) {
    return 'rfc_8693';
  }
  return 'unknown';
  ```

**`function validateRfc8693Token(payload: any): Rfc8693TokenPayload | null`**:
- Check required fields: `sub`, `act.sub`
- Check expiration: `exp > current_time`
- Return payload or null if validation fails

**`function validateTransactionToken(payload: any): TransactionTokenPayload | null`**:
- Check required fields: `sub`, `txn_id`
- Check expiration: `exp > current_time`
- Validate txn_id format (should start with `txn-` or be UUID)
- Return payload or null if validation fails

3. **Implement main validation function**:
```typescript
export async function validateAndDecodeToken(
  token: string,
  publicKey: string // PingOne public key for signature verification
): Promise<ValidatedToken | null> {
  try {
    // Decode JWT (without verification first) to inspect payload
    const decoded = jwt.decode(token); // or parse if custom format
    if (!decoded || typeof decoded !== 'object') {
      console.error('[TokenValidation] Failed to decode token');
      return null;
    }

    // Detect token type
    const tokenType = detectTokenType(decoded);
    console.log(`[TokenValidation] Detected token type: ${tokenType}`);

    let payload: TokenPayload | null = null;

    // Validate according to type
    if (tokenType === 'rfc_8693') {
      payload = validateRfc8693Token(decoded);
    } else if (tokenType === 'transaction_tokens') {
      payload = validateTransactionToken(decoded);
    } else {
      console.error('[TokenValidation] Unknown token type');
      return null;
    }

    if (!payload) {
      console.error(`[TokenValidation] ${tokenType} validation failed`);
      return null;
    }

    // Extract common fields
    const userId = payload.sub;
    const actorId = payload.act?.sub || (payload as TransactionTokenPayload).agent_id;
    const transactionId = (payload as TransactionTokenPayload).txn_id;
    const transactionScope = (payload as TransactionTokenPayload).txn_scope;

    // Verify JWT signature with PingOne public key (standard JWT verification)
    // jwt.verify(token, publicKey, { algorithms: ['RS256'] });

    return {
      type: tokenType as any,
      payload,
      userId,
      actorId,
      transactionId,
      transactionScope
    };
  } catch (err) {
    console.error('[TokenValidation] Validation error:', err.message);
    return null;
  }
}
```

   - Use standard Node.js crypto for signature verification

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External JWT Tokens | Bearer tokens from PingOne; must validate signature before trusting claims |
| Token Type Detection | Heuristic based on JWT claims; vulnerability if type detection logic is weak |
| Transaction Metadata | Claims within token; not trusted until JWT signature verified |
| MCP Tool Invocation | Tools receive context with user/actor identity; both token types provide same field types |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-198-06 | Tampering | Token Validation | Mitigate | Verify JWT signature before trusting any claims (using PingOne public key) |
| T-198-07 | Spoofing | Token Type Detection | Mitigate | Detection based on JWT claims; attacker can't add claims without valid signature. Signature verification gates claim trust |
| T-198-08 | Elevation of Privilege | Context Population | Mitigate | Context fields extracted from validated JWT only; no privilege escalation possible within single transaction scope |
| T-198-09 | Information Disclosure | Transaction ID in Context | Accept (low sensitivity) | TransactionId is UUID (no sensitive data); visible in logs and context. Acceptable for audit trail purposes |
| T-198-10 | Denial of Service | Token Parsing | Mitigate | Timeouts on JWT parsing/validation; if validation takes >5s, fail fast and close connection |

</threat_model>
