## Page Shell Layout Contract

Each monitoring page uses the **standard admin shell** pattern already established in App.js:

```
<AdminSideNav user={user} />
<TopNav user={user} onLogout={logout} />
<main className="main-content">
  <{PanelComponent} />
</main>
```

### Page Wrapper Component

Create a thin wrapper page component for each route. The wrapper:
- Adds the `app-page-shell` class to the outermost div (matches existing admin pages)
- Renders the panel component full-width with no additional chrome (no modal, no floating)
- Passes no extra props except what the panel already accepts in DevToolsDashboard

Specific panel prop contracts (extracted from DevToolsDashboard.jsx):

| Panel | Props when used as page |
|-------|------------------------|
| `TokenChainDisplay` | `hideHeader` omitted (show header in page context) |
| `TokenDiffPanel` | no props |
| `UnifiedTokenFlowInspector` | `floatingByDefault={false}` `showToggle={false}` |
| `McpTrafficPage` | no props |
| `ApiExplorerPanel` | no props |

Note: `TokenChainDisplay` in Dashboard.js uses no `hideHeader` prop ‚Äî include the header on the page. In the DevToolsDashboard floating context it used `hideHeader` to save vertical space; on a full page the header is appropriate.

### Active State Rules

Active state is determined by `isActive()` in AdminSideNav.jsx (line 265‚Äì271):

```js
const isActive = (path) => {
  if (path === "/admin" || path === "/dashboard")
    return location.pathname === path;
  return (
    location.pathname === path || location.pathname.startsWith(path + "/")
  );
};
```

## Route Registration Contract

### In App.js ‚Äî sidebarRoutePatterns array (lines 585‚Äì606)

Add `/monitoring` to the `sidebarRoutePatterns` array so the sidebar activates for all `/monitoring/*` routes:

```js
const sidebarRoutePatterns = [
  // ... existing entries ...
  "/monitoring",   // ADD THIS
];
```

### New Route Definitions

Add inside the authenticated `<Routes>` block (within the `path="*"` catch-all's inner `<Routes>`), alongside existing routes like `/agent-flow-inspector`:

```jsx
<Route
  path="/monitoring/token-chain"
  element={
    user ? (
      <TokenChainDisplay />
    ) : (
      <Navigate to="/" replace />
    )
  }
/>
<Route
  path="/monitoring/token-diff"
  element={
    user ? (
      <TokenDiffPanel />
    ) : (
      <Navigate to="/" replace />
    )
  }
/>
<Route
  path="/monitoring/flow-inspector"
  element={
    user ? (
      <UnifiedTokenFlowInspector floatingByDefault={false} showToggle={false} />
    ) : (
      <Navigate to="/" replace />
    )
  }
/>
<Route
  path="/monitoring/mcp-traffic"
  element={<McpTrafficPage />}
/>
<Route
  path="/monitoring/api-explorer"
  element={
    user ? (
      <ApiExplorerPanel />
    ) : (
      <Navigate to="/" replace />
    )
  }
/>
```

Auth guard pattern: follow the same `user ? <Component /> : <Navigate to="/" replace />` guard used by `/agent-flow-inspector` (App.js line 1131‚Äì1140). `McpTrafficPage` already has a public route at `/mcp-traffic` with no guard; keep `/monitoring/mcp-traffic` consistent with the existing `/mcp-traffic` route (no guard).

Import additions required in App.js: `TokenDiffPanel` and `ApiExplorerPanel` are not yet imported in App.js (they are only used inside DevToolsDashboard.jsx). Add:

```js
import TokenDiffPanel from "./components/TokenDiffPanel";
import ApiExplorerPanel from "./components/ApiExplorerPanel";
```

`TokenChainDisplay`, `UnifiedTokenFlowInspector`, and `McpTrafficPage` are already imported (lines 27, 84, 55).
