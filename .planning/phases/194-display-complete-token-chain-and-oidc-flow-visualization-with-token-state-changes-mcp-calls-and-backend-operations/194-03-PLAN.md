<objective>
Display backend banking operations in the flow timeline. Show which API endpoint was called, with what scopes/token, and what result was returned.

**Purpose:** Users see the complete end-to-end flow: token acquisition → MCP tool call → banking API invocation → result delivery.

**Output:** BackendOperationIndicator.js + addBackendOperation() hook + audit trail integration
</objective>

Banking API reference:
- `/api/banking/accounts` — list account summaries
- `/api/banking/balance?accountId=...` — get balance
- `/api/banking/transactions?accountId=...` — list transactions
- `/api/banking/transfer` — POST with { from, to, amount }
- `/api/banking/delegate` — manage delegation

These are called from banking agent tools (Get Balance, Transfer Money, Create Delegate).
</execution_context>

<context>
## Current Flow Architecture
1. **BankingAgent** initiates tool call with user intent
2. **bankingAgentService.callMcpTool()** invokes MCP server
3. **banking_mcp_server/tools** (Get Balance, Transfer, etc.) call BFF endpoints
4. **banking_api_server/routes/banking.js** handles actual banking operations

Currently, the UI shows:
- MCP tool call in AgentFlowDiagramPanel
- Token exchange in TokenChainDisplay
- But NOT the backend API calls

Missing link: MCP tool result → actual banking operation

## Solution Approach
BFF already audits operations (or can easily add audit trail). The BFF returns operation metadata in:
- Response headers (e.g., `X-Banking-Operation`, `X-Operation-Duration`)
- Or response body (OperationAuditLog entry)

The UI polls /api/mcp/tool-call-audit (if it exists, or we create /api/mcp/operation-audit) to fetch completed operations and display them.

Alternatively, BankingAgent receives operation details in the tool result and can surface them locally.
</context>

<threat_model>
## Trust Boundaries
Backend operations displayed include response data (balances, transaction IDs, etc.).

## STRIDE Analysis
**Spoofing:** Operations are fetched from trusted BFF; cannot be spoofed locally
**Tampering:** Operations in localStorage are read-only UI state; malicious modification doesn't affect backend
**Repudiation:** Audit trail on BFF side (not UI; out of scope)
**Information Disclosure:** Response data (balance, transaction ID) is already visible to the authenticated user; UI just displays what BFF returns
**Denial of Service:** Operations array capped at 30 entries to prevent growth
**Elevation of Privilege:** N/A

No new security risks. Operations displayed are already disclosed by BFF to the authenticated user.
</threat_model>
