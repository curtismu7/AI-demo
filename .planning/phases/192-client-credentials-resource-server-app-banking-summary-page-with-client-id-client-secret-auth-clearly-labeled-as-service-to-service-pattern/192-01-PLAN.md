<objective>
Create a Client Credentials resource server page — same banking summary layout as Phase 191 but authenticated using client_id/client_secret (no user context). The page is clearly labeled "Client ID / Client Secret — Service-to-Service" throughout. Contrasts with Phase 191's OIDC user-delegated flow to demonstrate the difference between user-context and machine-context access.

Purpose: Show that the same banking API can be accessed two ways: (1) user-delegated via OIDC + token exchange (Phase 191), (2) machine-to-machine via Client Credentials. The agent's dual token exchange produces tokens targeting the OIDC resource server, NOT this one. This page educates on why Client Credentials alone is insufficient for agentic delegation.
Output: ClientCredentialsResourcePage.jsx + CSS + resourceServerCC.js route + routing
</objective>

   ```json
     "accounts": [

       "scope": "...",
       "aud": "...",
       "iss": "...",
       "exp": ...,
       "iat": ...
     },
     "tokenMetadata": {
       "grantType": "client_credentials",
       "audience": "...",
       "scopes": [...],
       "expiresAt": "...",
       "issuer": "..."
     },
     "resourceServerInfo": {
       "name": "Banking API Resource Server (Client Credentials)",
       "type": "Client Credentials",
       "description": "Machine-to-machine access via client_id + client_secret — no user context",
       "authMethod": "Client Credentials Grant (OAuth 2.0 §4.4)",
       "note": "This token has NO sub claim — it represents the application, not a user"
     },
     "comparison": {
       "oidc": "Phase 191: User authenticates → access token with sub (user identity) + act (agent delegation) → resource server validates user context",
   ```

   configured), return status 200 with a structured error payload so the UI can display an informational
   message instead of a generic 500:

   ```js
   return res.json({
     error: 'cc_not_configured',
     message: err.message,
     configNeeded: ['PINGONE_WORKER_TOKEN_CLIENT_ID', 'PINGONE_WORKER_TOKEN_CLIENT_SECRET']
   });
   ```
4. Mount in server.js: `app.use('/api/resource-server-cc', require('./routes/resourceServerCC'));`

| Boundary | Description |
|----------|-------------|
| BFF → Client | CC token claims decoded server-side; raw secret never exposed |
| Admin gate | CC page requires admin role — prevents non-admin users from seeing CC token metadata |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-192-01 | Information Disclosure | CC token claims show client_id | Mitigate | Admin-only route; client_secret never included in response; only decoded claims |
| T-192-02 | Elevation of Privilege | Non-admin accesses CC page | Mitigate | AdminRoute wrapper on React side + admin session check on backend |
| T-192-03 | Spoofing | CC token impersonation | Accept | CC tokens are display-only for education; no authorization decisions made from displayed claims |
</threat_model>
