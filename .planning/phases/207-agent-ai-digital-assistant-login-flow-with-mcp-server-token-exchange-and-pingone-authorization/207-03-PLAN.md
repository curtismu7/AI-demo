<objective>
Implement HITL (Human-in-the-Loop) async decision flow and session correlation for concurrent MCP tool calls.

When PingOne Authorize signals that a tool call requires HITL approval (e.g., transaction >$500), the agent needs to:
1. Show the user an approval modal
2. Wait for user approval (polling or SSE)
3. Retry the tool call after approval

This also adds `taskId` correlation so concurrent tool calls on different Vercel Lambda instances route responses correctly.

**This is a DESIGN + BUILD plan.** Some design decisions need confirmation before implementation because:
- PingOne Authorize HITL signal shape is not yet confirmed (how does PA express "requires HITL"?)
- Polling vs SSE vs WebSocket for the async wait is a team decision
- The existing HITL flow in BankingAgent uses `consent_challenge_required` + `setHitlPendingIntent` — need to confirm this extends to PA-driven HITL

**Output:**
- `banking_api_server/routes/mcpDecisionPolling.js` — New polling endpoint
- Updated route registration in `server.js`
- Updated `BankingAgent.js` with HITL polling loop
</objective>

Before implementation, confirm these design decisions:

**Q1: How does PingOne Authorize express "HITL required"?**
Current `pingOneAuthorizeService.js` returns `{ decision, stepUpRequired, raw, decisionId }`.
- Is HITL a separate field in the PA response (`requiresHitl: true`)?
- Or is HITL signaled via `stepUpRequired: true` with a specific obligation in the `raw` response?
- Or is HITL a custom attribute in the PA policy that appears in the raw response body?

**Q2: Polling vs SSE?**
Options:
- **Polling:** `GET /api/mcp/decision/:taskId` every 2s — simpler, works with Vercel serverless
- **SSE:** Server-Sent Events — lower latency, but harder with Vercel Lambda (10s timeout)
- **Recommendation:** Polling — simpler, Vercel-compatible, matches existing patterns

**Q3: Where is HITL approval stored?**
- Upstash session store: `hitl:${taskId}` → `{ status: 'pending'|'approved'|'denied', approvedAt, approvedBy }`
- The approval modal POSTs to a new `POST /api/mcp/decision/:taskId/approve` endpoint
- The polling endpoint reads from Upstash

  <action>
Create `banking_api_server/routes/mcpDecisionPolling.js`:

1. **GET /decision/:taskId** — Poll for decision status
   - Requires authenticated session (session cookie)
   - Reads from Upstash: `hitl:${taskId}`
   - Returns: `{ status: 'pending'|'approved'|'denied'|'expired', taskId, updatedAt }`
   - If not found: HTTP 404

2. **POST /decision/:taskId/approve** — User approves HITL
   - Requires authenticated session
   - Validates: session user matches the taskId's user (prevent cross-user approval)
   - Writes to Upstash: `hitl:${taskId}` → `{ status: 'approved', approvedAt, approvedBy: session.user.sub }`
   - TTL: 5 minutes (auto-expire pending decisions)
   - Returns: HTTP 200 `{ status: 'approved' }`

3. **POST /decision/:taskId/deny** — User denies HITL
   - Same as approve but sets `status: 'denied'`

**Pattern:** Use the existing Upstash session store pattern from `banking_api_server/services/upstashSessionStore.js`.

// Poll every 2s for up to 60s
const pollInterval = setInterval(async () => {
  const resp = await fetch(`/api/mcp/decision/${taskId}`);
  const { status } = await resp.json();
  if (status === 'approved') { clearInterval(pollInterval); retryToolCall(); }
  if (status === 'denied' || status === 'expired') { clearInterval(pollInterval); showError(); }
