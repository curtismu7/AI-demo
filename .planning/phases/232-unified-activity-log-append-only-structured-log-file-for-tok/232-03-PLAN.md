<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| oauth callback → logAppEvent | authedUser.username/role written to metadata — comes from PingOne userInfo response |
| cibaService → logAppEvent | loginHint (user email or sub) written to metadata — operator-initiated CIBA flow |
| agentMcpTokenService → logAppEvent | mcpResourceUri and scope count written — no raw tokens |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-232-09 | Information Disclosure | oauth metadata.username | accept | Username is already stored in session and returned by /api/admin/app-events; same trust level as existing data |
| T-232-10 | Information Disclosure | ciba metadata.loginHint | mitigate | loginHint is an email or sub — not a secret, but a PII field. Log it because CIBA initiation is an admin-triggered event. Acceptable for a demo; in production would hash or redact. |
| T-232-11 | Information Disclosure | exchangeTokenRfc8693 metadata | accept | Only mcpResourceUri and scope count logged; no access tokens, no client secrets |
| T-232-12 | Repudiation | OAuth state mismatch warning | mitigate | Warning logged with timestamp and session/cookie state presence flags — provides audit trail for CSRF investigation |
| T-232-13 | Elevation of Privilege | logAppEvent called before redirect | accept | logEvent is fire-and-forget observability; does not change auth outcome; all redirects execute as before |
</threat_model>
