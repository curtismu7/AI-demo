<objective>
Fill the remaining D-04 instrumentation gaps by adding logEvent calls to routes/tokenChain.js, services/agentTokenService.js, and services/delegationService.js at key decision points (token issued, exchange attempted, delegation resolved).

Purpose: These three files were named in CONTEXT.md D-04 as known gaps. Without instrumentation the token chain and delegation flows are invisible in the activity log despite being central to the agent/MCP architecture.

Rationale for banking_mcp_server/ exclusion: banking_mcp_server/ is a separate TypeScript process (WebSocket-based MCP server) with its own package.json, tsconfig.json, and dist/ build. It does not import appEventService — that module lives in banking_api_server/. Instrumenting it would require either cross-process HTTP calls or duplicating the event service in a different runtime. This is out of scope for Phase 232 (server-side BFF instrumentation only). The MCP tool dispatch events are already captured via agentMcpTokenService.js (Plan 03) on the BFF side.

Output:
- tokenChain.js: logAppEvent on GET / success (chain length) and error
- agentTokenService.js: logAppEvent in validateAgentActorToken on valid/invalid result
- delegationService.js: logAppEvent in grantDelegation on success, provisioning failure, validation error; in revokeDelegation on success and not-found
</objective>

From banking_api_server/routes/tokenChain.js (full file — 44 lines):
```js
const { getTokenChain, getCurrentTokens, synthesizeFromSession, getMCPToolCalls } = require('../services/tokenChainService');

// GET /api/token-chain
router.get('/', async (req, res) => {
  try {
    let tokenChain = await getTokenChain(req.user.id);
    if (tokenChain.length === 0 && req.session?.oauthTokens?.accessToken) {
      tokenChain = synthesizeFromSession(req.session.oauthTokens.accessToken);
    }
    const mcpToolCallsChain = await getMCPToolCalls(req.user.id);
    res.json({ tokenChain, mcpToolCallsChain, metadata: { userId: req.user.id, totalEvents: tokenChain.length, ... } });
  } catch (err) {
    console.error('[tokenChain] GET error:', err.message);
    res.status(500).json({ error: 'internal_error', message: err.message });
  }
});

// GET /api/token-chain/current
router.get('/current', async (req, res) => {
  try {
    const currentTokens = await getCurrentTokens(req.user.id);
    res.json({ currentTokens });
  } catch (err) {
    console.error('[tokenChain] GET current error:', err.message);
    res.status(500).json({ error: 'internal_error', message: err.message });
  }
});
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| tokenChain route → logAppEvent | userId from req.user (set by authenticateToken middleware); chain length integer only — no token values |
| agentTokenService → logAppEvent | actorId and subject from introspection result — same data already returned to caller |
| delegationService → logAppEvent | delegatorEmail and delegateEmail written to metadata — operator-initiated action |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-232-14 | Information Disclosure | delegation metadata.delegateEmail | accept | Email is the grant target — already stored in delegations table and returned by listDelegations; no new disclosure boundary |
| T-232-15 | Information Disclosure | agentTokenService metadata.actorId | accept | actorId is extracted from an already-validated token's claims; same data already returned to the caller and visible in token chain UI |
| T-232-16 | Repudiation | delegation grant/revoke events | mitigate | Events logged with delegatorUserId, delegateEmail, delegationId, and timestamp — provides audit trail for delegation disputes |
| T-232-17 | Denial of Service | logAppEvent in delegationService hot path | accept | Delegation grant/revoke are low-frequency admin actions; appendFileSync is non-blocking for human-scale traffic; errors are try/catch in appEventService |
</threat_model>
