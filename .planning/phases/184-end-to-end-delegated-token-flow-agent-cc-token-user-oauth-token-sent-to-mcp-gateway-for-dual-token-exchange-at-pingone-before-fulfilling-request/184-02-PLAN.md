Phase 184 Token Exchange Model:
- **User token** (OAuth, subject): user@bank.example
- **Agent token** (client-credentials, actor): agent@bank
- **Exchange target**: MCP Gateway (unique resource for delegation)
- **Exchange mode**: RFC 8693 with act/may_act claims
- **Result**: Single token with act claim proving agent acted on behalf of user

Backend Endpoint Structure (pingoneTestRoutes.js):
- POST /api/pingone-test/token-exchange (two blocks, likely duplicate or intentional split)
  - Accepts mode ('single', 'dual', 'double')
  - Selects tokenUri based on mode (single → MCP Server + user token, dual → MCP Gateway + both tokens)
  - Returns normalized result so client knows what executed
- GET /api/pingone-test/exchange-user-agent-to-mcp
  - Phase 184 canonical path (user + agent → MCP Gateway)
  - Preferred method for testing dual-token
- GET /api/pingone-test/exchange-user-to-agent-to-mcp
  - Legacy educational flow (for comparison/testing only)
  - Two sequential exchanges

Current Issues:
- 'double' mode may not have explicit MCP Gateway targeting
- Audience routing ambiguous for POST endpoints
- Legacy vs canonical paths not explicitly documented

Phase 184 Requirements:

3. Mode normalization: 'double' → 'dual' with backward-compat alias
