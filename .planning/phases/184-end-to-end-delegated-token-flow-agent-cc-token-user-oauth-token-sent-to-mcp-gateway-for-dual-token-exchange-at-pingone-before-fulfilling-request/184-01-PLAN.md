Phase 184 Context:
- **Goal**: Make the Phase 184 delegated path explicit and verifiable across UI, backend routes, and docs so Exchange 2 consistently means user OAuth token + agent CC token exchanged for an MCP Gateway token before request fulfillment.
- **Scope**: Terminology and flow labeling only; no functional changes to test endpoints.
- **Audience**: Developers learning token exchange patterns; agents consuming MCP via Phase 184 dual-token path.

PingOne Test Page Role:
- Allows manual testing of all three token exchange patterns (1-exchange, 2-exchange, legacy 2-step)
- TEST_CONFIG.tests array defines each exchange with description and test function
- UI displays token panels showing user token, agent token, and final result token
- Form hints explain differences between patterns before test selection

Phase 184 Pattern:
- User is logged in (has OAuth token)
- Agent acts on behalf via client-credentials token
- Both tokens sent in single RFC 8693 exchange to MCP Gateway
- Result: single token with `act` claim proving delegation chain
- This is **canonical** for agents accessing MCP via Phase 184

Legacy Pattern (Exchange 3):
- User → agent token (separate access calls)
- User + agent → MCP token (separate exchange)
- Two separate calls; used for testing/educational purposes only
- Not the recommended path for Phase 184
