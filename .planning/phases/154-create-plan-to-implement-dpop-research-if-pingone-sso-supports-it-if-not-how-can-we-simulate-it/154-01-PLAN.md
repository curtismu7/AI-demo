<objective>
Research DPoP (Demonstrating Proof-of-Possession, RFC 9449) implementation feasibility with PingOne SSO, and create a plan for implementation or simulation.

Purpose: DPoP binds access tokens to a specific client key pair, preventing token theft/replay. This is increasingly important for AI agent scenarios where tokens are exchanged between services.

Output: 154-DPOP-RESEARCH.md with PingOne support assessment, implementation plan, and simulation approach if native support is absent.
</objective>

  <files>.planning/phases/154-create-plan-to-implement-dpop-research-if-pingone-sso-supports-it-if-not-how-can-we-simulate-it/154-DPOP-RESEARCH.md</files>
  <action>
1. Research DPoP (RFC 9449) fundamentals:
   - How DPoP proof JWTs work (client generates key pair, signs proof JWT)
   - DPoP header on token requests and resource requests
   - Token binding: `cnf.jkt` claim in access tokens
   - DPoP vs mTLS for proof-of-possession

2. Research PingOne SSO DPoP support:
   - Check PingOne documentation for DPoP support in authorization server
   - Check if PingOne token endpoint accepts DPoP headers
   - Check if PingOne issues tokens with `cnf.jkt` claims
   - If not supported: document the gap

3. Research simulation approaches (if PingOne doesn't support DPoP natively):
   - BFF-level DPoP simulation: generate DPoP proofs in banking_api_server, validate on resource access
   - Education-only mode: show DPoP proof generation and token binding in UI without actual enforcement
   - Proxy approach: BFF adds DPoP-like binding to tokens it issues to the MCP server

4. Create 154-DPOP-RESEARCH.md with:
   - **DPoP Overview**: What it is, why it matters for AI agents
   - **PingOne Support Assessment**: Supported / Not Supported / Partial (with evidence)
   - **Current Token Flow**: How tokens flow in this demo (authorize → BFF → MCP exchange)
   - **DPoP Integration Points**: Where DPoP proofs would be generated and validated
   - **Simulation Approach** (if needed): Detailed design for demo-grade DPoP
   - **Implementation Plan**: Phases/steps needed, estimated complexity per component
   - **Educational Value**: What this teaches demo viewers about sender-constrained tokens
  </action>
