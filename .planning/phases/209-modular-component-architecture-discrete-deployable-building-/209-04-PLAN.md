    Write a reference document covering:

    ## Overview
    - What: BX Finance MCP Server implements MCP spec 2025-11-25
    - Purpose: Exposes banking tools over MCP protocol; PingGateway and other MCP clients connect here

    ## Authentication
    - Auth model: `Authorization: Bearer <mcp-token>` header on every request
    - Token format: JWT or opaque token issued via RFC 8693 token exchange from the Banking API Server
    - Token validation: server calls `PINGONE_INTROSPECTION_ENDPOINT` (or `OIDC_INTROSPECTION_ENDPOINT`)
    - Audience: token `aud` claim must include `MCP_SERVER_RESOURCE_URI` when that env var is set
    - Error on missing/invalid token: `HTTP 401` with `WWW-Authenticate: Bearer realm="..."` header

    ## Transport Options
    Two transports operate simultaneously on the same port (default 8080):

    ### HTTP Streamable Transport (MCP 2025-11-25 spec, recommended for PingGateway)
    - Endpoint: `POST /mcp`
    - Content-Type: `application/json`
    - Session lifecycle: server returns `Mcp-Session-Id` header on first request; client must echo it on subsequent requests
    - Method: `initialize` → `tools/list` → `tools/call`
    - Session termination: `DELETE /mcp` with `Mcp-Session-Id` header

    ### WebSocket Transport (for Banking API UI)
    - Endpoint: `ws://<host>:8080/`
    - Used by the included React SPA via banking_api_server WebSocket proxy
    - Auth: token delivered via `session_init` message (not Authorization header)

    ### RFC 9728 Protected Resource Metadata
    - Endpoint: `GET /.well-known/oauth-protected-resource`
    - Returns: `{ resource, authorization_servers, bearer_methods_supported: ["header"] }`

    ## Available Tools
    Document each tool from BankingToolRegistry.ts:

    For each tool include:
    - Tool name
    - Description (copy from registry)
    - Input schema (copy from registry inputSchema)
    - Required scopes (copy from registry requiredScopes)
    - Read-only: yes/no
    - Requires user auth: yes/no

    Tools to document (read actual definitions from BankingToolRegistry.ts):
    1. `get_my_accounts`
    2. `get_account_balance`
    3. `get_my_transactions`
    4. `transfer_funds`
    5. `make_deposit`
    6. `make_withdrawal`

    ## Error Response Shapes
    ```json
    { "jsonrpc": "2.0", "id": <id>, "error": { "code": -32xxx, "message": "..." } }
    ```
    Application-level errors returned in `result.error`:
    - `{ "error": "authentication_required", "status": 401 }`
    - `{ "error": "insufficient_scope", "status": 403 }`
    - `{ "error": "step_up_required", "status": 428 }`

    ## PingGateway Integration Notes
    - Point PingGateway at `POST /mcp` endpoint
    - Configure Bearer token validation using the RFC 9728 discovery endpoint
    - Use `MCP-CONTRACT.md` tool definitions to configure PingGateway tool policies
    - The Banking API Server must also be running (tools call `BANKING_API_BASE_URL`)

    Document both adapter interfaces:

    ## banking-core Component Reference

    ### OIDCProviderAdapter Interface

    Both banking_api_server and banking_mcp_server implement this adapter independently
    (JS and TS respectively). The interface contract is:

    ```
    issuer                   → string: OIDC issuer URL
    authorization_endpoint   → string: /authorize endpoint
    token_endpoint           → string: /token endpoint
    introspection_endpoint   → string: /introspect endpoint
    userinfo_endpoint        → string: /userinfo endpoint
    jwks_uri                 → string: JWKS endpoint
    ```

    Env var precedence (both components):
    1. `OIDC_*` override vars (highest — non-PingOne IDP)
    2. `PINGONE_*` vars (default — existing PingOne deployments)
    3. Derived from PINGONE_BASE_URL or configStore region/environmentId

    `PINGONE_*` var names are NEVER renamed (D-05). Existing .env files work unchanged.

    ### AuthorizationAdapter Interface (banking_api_server only)

    To implement a custom authorization engine, create a class that extends
    `AuthorizationAdapter` and implements both methods:

    ```javascript
    class MyCustomAdapter extends AuthorizationAdapter {
      async evaluateTransaction(context) {
        // context: { userId, amount, type, acr }
        // Returns: { decision: 'PERMIT'|'DENY'|'INDETERMINATE',
        //            stepUpRequired: boolean, hitlRequired: boolean,
        //            path: string, decisionId: string, raw: object }
      }

      async evaluateMcpTool(toolName, context) {
        // context: { userId, tokenAudience, actClientId, nestedActClientId,
        //            mcpResourceUri, acr }
        // Returns: { allowed: boolean, reason: string,
        //            decision: 'PERMIT'|'DENY'|'INDETERMINATE',
        //            hitlRequired: boolean, stepUpRequired: boolean,
        //            decisionId: string, raw: object }
      }
    }
    ```

    Register by setting `AUTHORIZATION_ENGINE` env var or modifying the `getAdapter()`
    factory in `banking_api_server/services/authorization/AuthorizationAdapter.js`.

    **Security requirement:** Custom adapters MUST NOT fail-open. If configuration is
    incomplete, throw an error — do not return PERMIT or `allowed: true` silently.

    ### Shared Services

    | Service | Purpose | Notes |
    |---------|---------|-------|
    | `configStore` | Runtime config with SQLite backing + AES-256-GCM encryption | Singleton — must be the same instance across all require() calls |
    | `demoDataService` | Demo bank account data (SQLite) | Depends on configStore |
    | `exchangeAuditStore` | In-memory ring buffer for token exchange audit events | Self-contained, no external deps |

    Each service is consumed via `file:../banking-core` in package.json.
    After `npm install`, `require('banking-core/configStore')` and
    `require('./services/configStore')` return the SAME singleton object.
