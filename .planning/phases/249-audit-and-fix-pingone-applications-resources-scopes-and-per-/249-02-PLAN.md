# Plan 249-02 — BFF aud assertion + D-05 blacklist audit + regression tests

## Objective

After Plan 249-01 fixes the passthrough and MCP Server audience config, this plan:
1. Adds a BFF-side assertion that the token sent to the gateway has aud=mcpgateway.ping.demo
2. Audits and confirms the gateway D-05 blacklist covers all downstream URIs (including the new mcpserver.ping.demo)
3. Adds unit tests for both assertions
4. Logs Phase 249 in REGRESSION_PLAN.md §4

## Tasks

### Task 1 — BFF: assert gateway audience on resolved MCP token

<read_first>
- demo_api_server/services/agentMcpTokenService.js (lines 1620–1760 — _resolveFinalMcpAudience, two-exchange path)
- demo_api_server/services/configStore.js (lines around pingone_resource_mcp_gateway_uri, mcp_gateway_http_url)
- demo_api_server/.env
</read_first>

<action>
In `agentMcpTokenService.js`, find the point where the two-exchange path obtains the final
RFC 8693 result token (the token sent to the gateway). This is after Exchange #2 completes.

After the exchange response is parsed and before `tokenEvents` is assembled for the UI:

1. Decode the result access token using the existing `decodeJwtClaims()` function already
   defined in this file (line ~100).

2. Read the expected gateway audience:
   ```javascript
   const expectedGwAud =
     configStore.getEffective('pingone_resource_mcp_gateway_uri') ||
     configStore.getEffective('pingone_resource_two_exchange_uri') ||
     process.env.PINGONE_RESOURCE_MCP_GATEWAY_URI ||
     'mcpgateway.ping.demo';
   ```

3. Assert the decoded aud includes expectedGwAud:
   ```javascript
   const decodedResult = decodeJwtClaims(mcpAccessToken);
   const audList = decodedResult?.claims?.aud
     ? (Array.isArray(decodedResult.claims.aud) ? decodedResult.claims.aud : [decodedResult.claims.aud])
     : [];
   if (!audList.includes(expectedGwAud)) {
     warnLog('[AudAssertion] MCP token aud mismatch — expected:', expectedGwAud, 'got:', audList);
     throwTokenResolutionError(tokenEvents, 'aud_mismatch',
       `Exchange produced token with wrong audience: expected ${expectedGwAud}, got [${audList.join(', ')}]`,
       502);
   }
   warnLog('[AudAssertion] MCP token aud verified:', expectedGwAud);
   ```

4. Place the assertion AFTER the token is obtained but BEFORE the result is returned to the caller.
   Do not add this assertion on the single-exchange path — that path targets the BFF's own resource
   and the audience depends on the legacy exchange mode.

5. Do not add new require() statements. All helpers (decodeJwtClaims, warnLog, throwTokenResolutionError,
   tokenEvents, configStore) are already in scope in agentMcpTokenService.js.
</action>

<acceptance_criteria>

- If aud does not include expectedGwAud, throwTokenResolutionError is called with code='aud_mismatch'
- warnLog('[AudAssertion] MCP token aud verified: ...') is logged when assertion passes
- No new require() statements in agentMcpTokenService.js
- `cd demo_api_ui && npm run build` exits 0 (no import regressions — BFF is not imported by UI, but the build check catches any server.js reference chain issues)
</acceptance_criteria>

<action>
Create `demo_api_server/tests/agentMcpTokenService.audAssertion.test.js`.

Use the mock patterns from the closest existing test file (likely agentMcpTokenService.test.js).

Test cases:

1. **correct_gw_aud_passes** — when Exchange #2 result has aud=mcpgateway.ping.demo,
   no aud_mismatch error is thrown, token is returned successfully.

2. **wrong_aud_throws** — when Exchange #2 result has aud=enduser.ping.demo,
   throwTokenResolutionError is called with code='aud_mismatch', httpStatus=502.

3. **mcpserver_aud_rejected** — when Exchange #2 result has aud=mcpserver.ping.demo
   (a bypass-attempt shape — server aud must never appear on the gateway-bound token),
   throwTokenResolutionError is called with code='aud_mismatch'.

4. **missing_aud_rejected** — when Exchange #2 result token is missing aud claim entirely,
   throwTokenResolutionError is called with code='aud_mismatch'.

5. **multi_aud_with_gw_passes** — when aud is an array containing mcpgateway.ping.demo
   plus another value, assertion passes (aud.includes() check is correct).

Mock setup:
- jest.mock configStore to return 'mcpgateway.ping.demo' for 'pingone_resource_mcp_gateway_uri'
- Mock the Exchange #2 HTTP call to return a synthetic JWT with the specified aud
- Use a synthetic base64url JWT: `btoa(JSON.stringify({aud: 'mcpgateway.ping.demo', sub: 'test-user', exp: 9999999999}))` for valid tokens

Note: if the two-exchange function is complex to isolate, test the aud assertion logic as a
unit by extracting the aud-check into a named helper `assertGatewayAudience(claims, expected, tokenEvents)`
and testing that helper directly — then call it from the two-exchange path. Only extract the
helper if the two-exchange function is not directly unit-testable.
</action>
