**Current flow (assumed):**
- Receives OAuth token from PingOne
- Validates token
- Creates session
- Redirects to `/dashboard` or `/marketing` (or a configured `redirectURI`)

**New flow:**
- Keep all existing validation and session creation logic UNCHANGED
- Change the final redirect destination from `/dashboard` to `/oauth/token-display`
- OR: Add query parameter option: if `?showToken=true`, redirect to token display; otherwise, redirect to dashboard

**Implementation (two options, pick one):**

**Option A: Always redirect to token display (simplest)**
```javascript
// In /api/auth/oauth/user/callback handler, at the end:
return res.redirect(302, '/oauth/token-display');
```

**Option B: Conditional redirect (preserve existing flow)**
```javascript
// In /api/auth/oauth/user/callback handler:
const redirectTo = req.query.redirectTo || req.body.redirectTo || '/oauth/token-display';
return res.redirect(302, redirectTo);
```
This allows clients to specify `?redirectTo=/dashboard` if they want old behavior.

**CRITICAL: DO NOT BREAK existing flows:**
- Admin OAuth callback (`/api/auth/oauth/callback`) — should redirect to `/admin` as before
- User logout: should still work (not affected)
- Session cookies: should still be set correctly
- PKCE validation: should still work (not touched)

**Test points:**
- After callback, user session is created (check `req.session` or Cookie)
- User is redirected to `/oauth/token-display`
- Token display page has access to the user's session token

**Security:**
- Only redirect to relative URLs (use `/oauth/token-display`, not full URLs)
- Verify that session is established before redirect
- Do NOT include token in redirect URL (use session)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OAuth callback ↔ Token display | Redirect uses session only (no token in URL) |
| User ↔ OAuth service | PingOne authentication (PingOne responsibility) |
| BFF ↔ PingOne userinfo | Bearer token used; result returned to frontend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-169-08 | Tampering | OAuth callback redirect | Mitigate | Only relative URLs allowed; use res.redirect (automatic safe redirects) |
| T-169-09 | Repudiation | Token display access | Mitigate | Require user session; log access in activity log if available |
| T-169-10 | Information Disclosure | Redirect chain | Mitigate | Token only in session; never in URL/query params; session over HTTPS |
| T-169-11 | Elevation of Privilege | Admin vs User flow | Mitigate | Keep admin callback separate; verify callback handler doesn't confuse roles |
</threat_model>
