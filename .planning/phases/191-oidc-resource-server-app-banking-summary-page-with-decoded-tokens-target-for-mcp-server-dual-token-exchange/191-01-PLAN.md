<objective>
Create an OIDC-authenticated resource server page that looks like the Banking summary page (account cards, balances, transaction summary) and displays decoded access/ID tokens with full claim details. This page represents the real resource server endpoint — the target audience for the MCP server's dual token exchange.

Purpose: Demonstrate what the MCP server is protecting. When an agent exchanges tokens via RFC 8693, the resulting token's `aud` claim points to THIS resource server. The page makes the OIDC flow visible: user authenticates → sees their banking data + decoded tokens proving delegated access.
Output: ResourceServerPage.jsx + ResourceServerPage.css + resourceServer.js route + App.js/AdminSideNav wiring
</objective>

|----------|-------------|
| BFF → Client | Only decoded claims sent; raw tokens never leave server |
| Session → Route | Route requires authenticated session (401 if missing) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-191-01 | Information Disclosure | Decoded token claims sent to browser | Mitigate | agentMcpTokenService.sanitizeClaims strips sensitive fields; only display-safe claims forwarded |
| T-191-02 | Spoofing | Unauthenticated access to banking summary | Mitigate | Route checks req.session.oauthTokens.accessToken and rejects '_cookie_session' placeholder; server.js mounts with authenticateToken middleware; React route requires `user` prop |
| T-191-03 | Tampering | Client-side claim modification | Accept | Claims are display-only; no server-side decisions based on client-displayed claims |
</threat_model>
