<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → BFF (/api/banking/...) | User passes sessionId in cookie; BFF validates session + token scope before calling resource server |
| BFF → Resource Server (PingOne) | BFF uses MCP-exchanged tokens with correct `aud` and `act` claims; Phase 187 validates token payload before send |
| Client ← BFF (401 response) | BFF returns `need_auth: true` signal; client intercepts and redirects to PingOne login (Phase 187 pattern) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-189-01 | Spoofing | Resource button (disabled state) | Accept | Disabled state is UI-only; BFF layer-zero gate enforces auth server-side |
| T-189-02 | Tampering | Button click → runAction | Mitigate | BankingAgent.js runAction enforces isLoggedIn check before API call (line 1650) |
| T-189-03 | Repudiation | Balance/transactions fetch | Mitigate | BFF logs all /api/banking/* calls with user ID + timestamp; audit trail in CHANGELOG/Phase 187 |
| T-189-04 | Information Disclosure | Resource data on /marketing | Accept | Data only visible to logged-in users; need_auth flow (Phase 187) ensures token exchange before any API hit |
| T-189-05 | Denial of Service | Button spam | Mitigate | BankingAgent.js request debouncing + BFF rate-limit per phase plan (not Phase 189 scope but noted) |
| T-189-06 | Elevation of Privilege | Actor/act claim bypass | Mitigate | Phase 187 RFC 8693 validation ensures only user's own act claim used; scope audit in Phase 104 |

</threat_model>
