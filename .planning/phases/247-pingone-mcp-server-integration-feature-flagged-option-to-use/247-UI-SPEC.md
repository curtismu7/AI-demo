# Phase 247 — UI Design Contract

> Visual and interaction contract for the admin MCP mode toggle and status chip.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope of UI Changes

This phase adds exactly **two UI surfaces** to the existing admin area:

1. **Feature flag entry** — a new card in `FeatureFlagsPage.js` under the "MCP Server" category  
   Toggle: `mcp_use_pingone_server` — switches the BFF to route tool calls via `pingidentity/pingone-mcp-server` stdio binary instead of the custom MCP gateway.

2. **Status chip in `McpGatewayConfig.jsx`** — a mode indicator in the existing header badge area  
   Shows: `Custom Gateway` (default) or `PingOne MCP Server` (when flag is on).

No new pages, no new routes, no nav changes. Both surfaces extend existing components and CSS classes already established in the codebase.

## Component Specifications

### 1. Feature Flag Card — `mcp_use_pingone_server`

**Location:** `banking_api_server/routes/featureFlags.js` — `FLAG_REGISTRY` array, "MCP Server" category section (insert after `mcp_use_legacy_protocol`).

**Flag object:**

```js
{
  id:           'mcp_use_pingone_server',
  name:         'MCP — Use PingOne MCP Server (stdio)',
  category:     'MCP Server',
  description:
    'When **ON**, the BFF spawns the `pingidentity/pingone-mcp-server` stdio binary and routes ' +
    'all agent tool calls through its stdio transport via an adapter layer, bypassing the custom ' +
    'MCP gateway. The PingOne MCP Server must be installed (npx or local binary) and configured ' +
    'with `PINGONE_MCP_SERVER_CMD` env var. ' +
    'When **OFF** (default), the existing custom MCP gateway continues to handle all tool calls.',
  impact:
    'OFF (default) = custom MCP gateway active (all Phase 243 auth, RFC 9728, PingOne Authorize). ' +
    'ON = PingOne MCP Server stdio mode; custom gateway bypassed. Requires valid PingOne credentials ' +
    'in env. The MCP Gateway Config panel shows active mode chip.',
  type:         'boolean',
  defaultValue: false,
  warnIfEnabled: true,
}
```

**Rendering:** Uses existing `FlagCard` component — no new JSX needed. Renders as standard toggle card under "MCP Server" category group.

**Warning banner copy (warnIfEnabled = true):**  
> "⚠️ Enabling this flag may reduce security. Use with care."  
*(uses existing `ff-card__warn` class/copy from `FlagCard` component)*

### 2. Mode Status Chip in `McpGatewayConfig.jsx`

**Location:** Inside the existing `<div className="mgc-header-badge">` — add a second badge below the existing `<StatusBadge>`.

**New component:**

```jsx
function McpModeChip({ usePingOneServer }) {
  if (usePingOneServer) {
    return (
      <span className="mgc-badge mgc-badge--pingone-mode">
        🔵 PingOne MCP Server
      </span>
    );
  }
  return (
    <span className="mgc-badge mgc-badge--custom-mode">
      🛡️ Custom Gateway
    </span>
  );
}
```

**New CSS classes** (append to `McpGatewayConfig.css`):

```css
.mgc-badge--custom-mode   { background: #fff3cd; color: #856404; }
.mgc-badge--pingone-mode  { background: #cfe2ff; color: #0a58ca; }
```

**Data source:** `McpGatewayConfig` fetches `/api/admin/mcp-gateway/config`. The BFF route must include `mcpMode: 'custom' | 'pingone'` in the response JSON (driven by the feature flag). The component reads `data.mcpMode` and passes `usePingOneServer={data.mcpMode === 'pingone'}` to `McpModeChip`.

**Placement in header:**

```jsx
<div className="mgc-header-badge">
  <StatusBadge running={…} devBypass={…} enabled={…} />
  {data && <McpModeChip usePingOneServer={data.mcpMode === 'pingone'} />}
</div>
```

## Interaction Contract

### Feature flag toggle
- Click toggle → `PATCH /api/admin/feature-flags` with `{ mcp_use_pingone_server: <newValue> }`
- Uses existing `onToggle` handler in `FeatureFlagsPage.js` — no custom handler needed
- Saving spinner: existing `ff-card__saving` "Saving…" overlay
- On success: card re-renders with updated value (existing optimistic update pattern)

### McpModeChip data loading
- Chip renders only when `data` is non-null (guard: `{data && <McpModeChip …/>}`)
- No loading skeleton — chip simply absent during initial fetch (< 100ms in normal operation)
- On fetch error: chip absent (error banner from existing `setError` path handles visibility)
- Chip is **read-only** — clicking it does nothing; it reflects state set by the feature flag

### Feature flag → chip refresh
- The two UI surfaces are **independent** — the chip reads from the MCP gateway config endpoint, not from FeatureFlagsPage state
- After toggling the flag, user must navigate to or refresh McpGatewayConfig to see chip update
- This is acceptable UX for an admin-only dev tool page; no cross-component state sync needed
