## Goal

## Real PingOne Authorize response shapes (from docs)

### PERMIT (HTTP 200)
```json
{
  "id": "<uuid>",
  "createdAt": "<iso8601>",
  "completedAt": "<iso8601>",
  "duration": 12,
  "status": "SUCCESS",
  "result": {
    "decision": "PERMIT",
    "weight": 1.0
  },
  "statements": [],
  "obligations": []
}
```

### DENY (HTTP 200)
```json
{
  "id": "<uuid>",
  "createdAt": "<iso8601>",
  "completedAt": "<iso8601>",
  "duration": 14,
  "status": "SUCCESS",
  "result": {
    "decision": "DENY",
    "weight": 1.0
  },
  "statements": [],
  "obligations": []
}
```

### Step-up (HTTP 200 with obligation, or 428 if BFF converts)
PingOne returns 200 with an obligation; the BFF converts to 428.
```json
{
  "id": "<uuid>",
  "status": "SUCCESS",
  "result": { "decision": "PERMIT", "weight": 1.0 },
  "obligations": [
    { "id": "step_up_mfa", "type": "IDENTITY_REQUIREMENT", "detail": { "acr": "Multi_Factor" } }
  ]
}
```

## Tasks

### T1 — Read simulatedAuthorizeService.js
Read `banking_api_server/services/simulatedAuthorizeService.js` and compare its response envelope to the PingOne shapes above.

### T2 — Patch simulatedAuthorizeService.js response envelope
Ensure the simulated service returns exactly the documented PingOne shape.

Key fields to add/fix if missing:
- `id` — `crypto.randomUUID()`
- `createdAt` — `new Date().toISOString()` (set before evaluation starts)
- `completedAt` — `new Date().toISOString()` (set after evaluation)
- `duration` — `completedAt - createdAt` in ms (integer)
- `status` — `"SUCCESS"`
- `result.decision` — `"PERMIT"` | `"DENY"`
- `result.weight` — `1.0`
- `statements` — `[]` (empty array)
- `obligations` — `[]` for permit/deny, or obligation object for step-up

The BFF (`transactionAuthorizationService.js` or `pingOneAuthorizeService.js`) then inspects this response and converts the obligation to HTTP 428 — that conversion should NOT change.

### T3 — Patch simulated request body to match PingOne Authorize POST format
Read `pingOneAuthorizeService.js` to see what body it sends to PingOne. Ensure the simulated path receives and logs an identical request body format:
```json
{
  "parameters": {
    "transactionAmount": 5000,
    "userId": "...",
    "accountId": "...",
    "transactionType": "transfer"
  }
}
```
The simulated handler should accept this same shape (not a different custom format).
