# Phase 269.1 Plan 01: Extend vaultLoader.js with runtime unlock + state helpers Summary

Added `unlockVaultAtRuntime` and two state-helper functions to `banking_api_server/services/vaultLoader.js` so admin routes in Plan 02 can unlock the encrypted vault from the web UI without a process restart, while preserving the ¬ß1-protected `loadVaultIntoConfigStore` startup contract byte-identically (except for a provably side-effect-free 2-line state mirror on its success path).

## What Was Built

### `banking_api_server/services/vaultLoader.js` ‚Äî 3 precise additions

| Location | Change | Why |
|----------|--------|-----|
| Top of file (after `DEFAULT_VAULT_PATH`) | `let _unlocked = false; let _entriesLoaded = 0;` + 3-line comment | Module-scoped state flags read only by the new state helpers; reads no other state |
| Inside `loadVaultIntoConfigStore` success path (immediately before the existing `logger.log('[vault] loaded ...')` line) | `_unlocked = true; _entriesLoaded = entryCount;` | Mirrors successful-startup state so `GET /api/admin/vault/status` reports unlocked correctly after a normal BFF boot. Writes only to flags introduced in this plan |
| After `loadVaultIntoConfigStore`, before `module.exports` | New `async function unlockVaultAtRuntime(opts)` + `function isVaultUnlockedThisProcess()` + `function vaultEntryCountThisProcess()` | Sibling runtime-unlock function and read-only state helpers |
| `module.exports` line | Expanded to 5 exports (was 2) | Plan 02 imports the new functions |

**Behavior preservation contract** (REGRESSION_PLAN ¬ß1 row "Vault BFF startup"):

- `loadVaultIntoConfigStore` signature unchanged
- Return value shape `{loaded, entries, reason?}` unchanged
- `configStore.setRaw(data, { persist: false })` call signature unchanged
- `fs.existsSync` gate unchanged
- `delete process.env.VAULT_PASSWORD` in finally unchanged (still occurs exactly once, only in the startup loader)
- `vault.close()` in finally unchanged
- 5-state reason matrix (vercel / no_vault_file / missing_password / open_failed / loaded) unchanged
- All `logger.log` / `logger.error` call sites unchanged
- Only modification: 2 lines on the success path that write to module-scoped flags introduced in this plan; nothing reads those flags outside `isVaultUnlockedThisProcess` / `vaultEntryCountThisProcess` (which did not exist before this plan)

The 14/14 existing startup tests in `bff-startup.test.js + serverless.test.js` continue to pass ‚Äî that is the empirical proof of behavior preservation.

### `unlockVaultAtRuntime({password, vaultPath, configStore, vaultLib, logger})`

Differences vs. the startup loader:
- `password` is **required** (no `process.env.VAULT_PASSWORD` fallback) ‚Äî caller supplies via `req.body`
- Does **not** consult `process.env.VERCEL` (route handler in Plan 02 enforces the 503)
- Does **not** call `delete process.env.VAULT_PASSWORD` (the runtime password never came from env)
- Throws `Error` with `code === 'VAULT_PASSWORD_MISSING'` synchronously for non-string / empty-string password (before any Argon2id work)
- Throws `Error` with `code === 'VAULT_FILE_NOT_FOUND'` synchronously when vault file is absent

Same as the startup loader:
- Calls `configStore.setRaw(data, { persist: false })` batched once with lowercased names
- Calls `vault.close()` in `finally` ‚Äî KEK + DEKs always zeroed
- Logs only `err.message` on open failure (never `err.stack` ‚Äî no argon/kek/dek leak)
- Rethrows `VaultAuthError` / `VaultIntegrityError` from `lib/vault.openVault` verbatim (opaque error ‚Üí no enumeration oracle)

### `isVaultUnlockedThisProcess()` / `vaultEntryCountThisProcess()`

Two synchronous helpers that return the module-scoped flags. Used by `GET /api/admin/vault/status` (Plan 02) to answer "is the vault unlocked?" without enumerating `configStore`. Returns `false` / `0` from boot until the first successful unlock (startup OR runtime); flips `true` / `N` after either path succeeds.

## REGRESSION_PLAN ¬ß1 Preservation Statement

**Protected row touched:** `Vault BFF startup` (REGRESSION_PLAN.md line 72) ‚Äî added by Phase 269 Plan 03.

**Why the 2-line state-mirror modification is behavior-preserving:**

The two lines `_unlocked = true; _entriesLoaded = entryCount;` are inserted immediately before `logger.log('[vault] loaded ' + entryCount + ' entries from ' + vaultPath);` on the function's success path. The lines:

1. **Read no external state** ‚Äî they only write to module-scoped `let` bindings that did not exist before this plan
2. **Mutate only module-scoped flags introduced in this plan** ‚Äî nothing reads `_unlocked` or `_entriesLoaded` outside the new `isVaultUnlockedThisProcess` / `vaultEntryCountThisProcess` helpers
3. **Do not change the function's return value** ‚Äî `return { loaded: true, entries: entryCount };` is unchanged
4. **Do not change the function's `configStore.setRaw` mutation** ‚Äî the upstream batch call signature is unchanged
5. **Do not change the function's `vault.close()` finally behavior** ‚Äî the cleanup block runs the same way
6. **Do not change the function's `delete process.env.VAULT_PASSWORD`** ‚Äî that line still runs in finally exactly once
7. **Do not change the function's 5-state reason-code matrix** ‚Äî `vercel`, `no_vault_file`, `missing_password`, `open_failed`, and `loaded` branches all return the same shapes

By construction, no existing test, route handler, server.js IIFE, or downstream caller can observe the difference. The 14/14 existing tests in `bff-startup.test.js + serverless.test.js` passing after the change is the empirical proof.

## Threat Model Disposition (vs Plan 01's STRIDE table)

| Threat | Disposition | How verified |
|--------|-------------|--------------|
| T-269.1-03 (info disclosure via err.stack in logger) | mitigated | `logger.error('[vault] runtime open failed:', err.message)` ‚Äî only `err.message`; Test 4 asserts joined logger output does NOT match `/argon\|kek\|dek/i` and does NOT contain the caller-supplied password |
| T-269.1-04 (spoofing via empty/non-string password) | mitigated | `typeof password !== 'string' \|\| password.length === 0` synchronous gate BEFORE `openVault` ‚Äî no Argon2id work for invalid input (CPU + timing window closed); Tests 1 + 2 |
| T-269.1-05 (tampering via configStore persistence) | mitigated | `configStore.setRaw(data, { persist: false })` enforced ‚Äî Test 7 asserts exact `{persist:false}` opts; same contract as Phase 269 Plan 03 (T-269-06 extends to runtime path) |
| T-269.1-08 (password in req.body / function scope) | accept (deferred) | Password lives in function-local scope only; not stored on the module-scoped object. Route handler in Plan 02 will be responsible for not retaining `req.body.password` |
| T-269.1-09 (vaultPath tampering) | mitigated | `vaultPath` defaults to `process.env.VAULT_PATH \|\| DEFAULT_VAULT_PATH`; Plan 02 route handler MUST NOT accept a vaultPath from `req.body` (enforced in Plan 02 acceptance criteria) |
| T-269-08 inherited (KEK/DEK lifetime) | mitigated | `vault.close()` in finally ‚Äî Tests 5 + 6 assert `close` is invoked on success AND on `configStore.setRaw` error; lib/vault's close zeroes KEK + DEKs |
