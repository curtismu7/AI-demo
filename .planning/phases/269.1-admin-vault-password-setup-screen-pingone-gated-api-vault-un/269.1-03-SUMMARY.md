# Phase 269.1 Plan 03: Admin Vault Page (operator UX) Summary

React admin page at `/admin/vault` that operators use to unlock and rotate the vault without restarting the BFF. Three sections: status indicator, unlock form, rotate form. Uses `apiClient` only (no raw axios or fetch). One eager import + one Route added to `App.js`; one text-only entry added to `AdminSideNav.jsx` under System Tools. Together with Plan 269.1-02 this closes Phase 269.1's user-facing scope.

## What Was Built

### `banking_api_ui/src/components/AdminVaultPage.jsx` — three sections, one default export

| Section | Action | UX rules |
|---------|--------|----------|
| Status card | `apiClient.get('/api/admin/vault/status')` on mount + manual refresh | Renders vaultFilePresent (✅/❌), basename(vaultPath), unlocked (✅/❌), entriesLoaded |
| Unlock form | `apiClient.post('/api/admin/vault/unlock', { password })` | Disabled button while empty / submitting. Success: clear input + green banner with N entries. Failure: red banner with server's opaque message; input retained for retry |
| Rotate form | `apiClient.post('/api/admin/vault/rotate', { currentPassword, newPassword })` | Client-side validate empty → mismatch → length<12 → same-as-current BEFORE any API call. Success: clear all three inputs + green banner including the ⚠️ "Update VAULT_PASSWORD before next BFF restart" warning |

State shape:

| State | Type | Notes |
|---|---|---|
| `status` | `{ loading } \| { loading: false, unlocked, entriesLoaded, vaultFilePresent, vaultPath } \| { loading: false, error }` | mount-fetched + manually refreshable |
| `unlockPassword` | `string` | cleared on success only |
| `currentPassword` / `newPassword` / `confirmNewPassword` | `string` | all three cleared on rotate success |
| `banner` | `null \| { kind: 'ok'\|'err', text }` | text is static success message OR `err.response?.data?.message` — never the typed password |
| `submitting` / `rotating` | `boolean` | per-form double-submit guard |

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking issue] Plan said wrap in `<AdminLayout>`; actual codebase already mounts the sidebar globally**
- **Found during:** Pre-implementation read of App.js
- **Issue:** The plan instructed `<AdminLayout>{...}</AdminLayout>`. App.js line 943 renders `<AdminSideNav user={user} />` outside the Routes element for every admin path. AdminLayout (`banking_api_ui/src/components/AdminLayout.jsx`) also renders `<AdminSideNav />`. Wrapping the page in AdminLayout would produce **two sidebars side-by-side**.
- **Fix:** Followed the existing admin-page pattern (LlmConfigPage = `<div className="page-container">`, AuthorizeConfigPage = plain root `<div>`). My page wraps content in `<div className="admin-vault-page">` only.
- **Files modified:** banking_api_ui/src/components/AdminVaultPage.jsx
- **Commit:** 9a757582

**2. [Rule 3 — Blocking issue] Plan's CSS spec included `monospace` font-family; UI regression test bans it**
- **Found during:** Full UI test suite verification (after AdminVaultPage tests passed)
- **Issue:** `src/__tests__/uiRegression.test.js` line 226 asserts "no CSS file uses monospace font" — with an explicit 6-file allowlist. My new file triggered the rule on 3 lines (`.__status code`, `.__rotate-hint code`, `input[type="password"]`).
- **Fix:** Removed `font-family: ui-monospace, ...` declarations. `<code>` elements continue to render monospace via the browser user-agent stylesheet default. Visual outcome: identical for the `<code>` blocks; password inputs use the browser default UI font (acceptable — they're password inputs, contents are masked).
- **Alternative considered:** Adding `AdminVaultPage.css` to the regression test's allowlist. Rejected — it expands a global test rule, exceeds the plan's `files_modified` list, and is unnecessary because `<code>` already renders monospace by default.
- **Files modified:** banking_api_ui/src/components/AdminVaultPage.css
- **Commit:** 9a757582

**3. [Rule 1 — Lint bug] Pre-commit hook (testing-library/no-unnecessary-act) rejected `act(...)` wrapper around `fireEvent.click`**
- **Found during:** Initial commit attempt
- **Issue:** First test-12 implementation wrapped `fireEvent.click` in `await act(async () => { ... })`. The project's ESLint rule `testing-library/no-unnecessary-act` flagged this — React 18+ auto-wraps fireEvent in act, making the explicit wrapper redundant. The `lint-staged` pre-commit hook blocked the commit.
- **Fix:** Removed the `act()` wrapper; `fireEvent.click(...)` followed by `await waitFor(...)`. Removed the unused `act` import from `@testing-library/react`.
- **Files modified:** banking_api_ui/src/components/__tests__/AdminVaultPage.test.jsx
- **Commit:** 9a757582 (the linter also Prettier-reformatted import lines on its way through; non-functional)

## STRIDE Threat Register — Disposition Evidence

| Threat | Disposition | Mitigation Evidence |
|--------|-------------|---------------------|
| T-269.1-02 Tampering (CSRF) | accept (existing posture) | apiClient uses `withCredentials: true` + `httpOnly` session cookie + `sameSite` per env + admin gate per handler — same posture as every other `/api/admin/*` route in this SPA |
| T-269.1-03 Info Disclosure (password leak via DOM/autocomplete) | mitigate | All four password fields use `<input type="password">`; `autoComplete="current-password"` for the unlock + rotate-current fields, `autoComplete="new-password"` for the new + confirm fields. Test 9 asserts `type === 'password'` on all four fields. State always cleared on success |
| T-269.1-04 Spoofing (brute force) | mitigate (server-side) | Server's express-rate-limit (5/5min keyed by sub) caps damage. UI surfaces the 429 message verbatim via `err.response.data.message` |
| T-269.1-08 Info Disclosure (password retained longer than needed) | mitigate | useState scope only — cleared on success. Not in localStorage, sessionStorage, or any Context provider. Component unmount drops state |
| T-269.1-11 Misconfiguration (UI accessible on Vercel where backend is 503) | accept (graceful degrade) | Backend returns 503; UI's catch block surfaces the documented message in the red banner |
| T-UI-01 (new) Info Disclosure via banner text | mitigate | Banner always uses static success strings OR `err.response.data.message` from the server. Test 5 uses a SENTINEL password string and asserts the banner.textContent does not contain it |
| T-UI-02 (new) Tampering — mismatch bypass | mitigate (defense in depth) | Both client-side check (`if (newPassword !== confirmNewPassword) bail`) AND server-side rotate handler still re-verifies currentPassword via openVault. Test 6 asserts `apiClient.post('/api/admin/vault/rotate', ...)` was NOT called when client validation fails |
