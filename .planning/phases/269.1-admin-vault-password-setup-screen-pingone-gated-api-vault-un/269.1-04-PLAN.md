    Content covers:
    - When to use this (vs. the CLI). Bullet points:
      - Use the CLI (`npm run vault:rotate`) when you can restart the BFF — simpler, fewer moving parts.
      - Use `/admin/vault` when you must rotate without downtime, OR when the operator only has browser access to the running BFF.
    - Prerequisites: admin login session (PingOne-authenticated session OR `banking:admin` scope token); vault file present at `VAULT_PATH || repo-root/secrets.vault`.

    - Audit trail: every unlock/rotate writes a NDJSON line to `<vaultPath>.audit.log` with `caller:"adminVault"`. CLI uses `caller:"vault.js"`. Both rotate-via-CLI and rotate-via-web therefore appear in the same audit log, distinguished by the caller field.
    - Vercel: `/admin/vault` routes return 503 with `error:"vault_disabled_serverless"`. Vercel deployments use Encrypted Environment Variables instead — see existing "Serverless treatment" section of this doc.
    - Rate limit: unlock is throttled to 5 attempts / 5-minute window per admin session sub. 6th attempt returns 429 with `Retry-After` header. Wait 5 minutes or rotate from a different admin session.
    - Mutex: rotate is serialized via a process-local mutex. Two concurrent rotate calls → one returns 200, the other returns 409 `rotate_in_progress`.

    Section 2 — H2 `## After rotating: update VAULT_PASSWORD before next restart`:

    Content covers (REQ-VAULT-ADMIN-13 — verbatim warning shape from RESEARCH §"Common Pitfalls" Pitfall 1):
    - Critical warning: after `/admin/vault rotate` returns 200, the vault FILE is encrypted with the new password but the BFF's startup environment still has the OLD `VAULT_PASSWORD`. The next BFF restart will fail-fast with `[vault] startup load failed; refusing to start.`
    - Numbered checklist of places to update VAULT_PASSWORD (mirror RESEARCH lines 287-292):
      1. `banking_api_server/.env` (or wherever your shell sources from).
      2. Your pm2 ecosystem file (`ecosystem.config.js` env block) if you use pm2.
      3. Your secret manager (1Password / SOPS / direnv `.envrc`) if you use one.
      4. Any CI/CD environment that runs the BFF (Vercel envs don't apply because Vercel skips the vault — see §"Serverless treatment").
    - "How to verify": run `VAULT_PASSWORD=<new-password> npm run vault:list` from `banking_api_server/` — should print the entry name list. If it fails with `vault: open failed`, your VAULT_PASSWORD copy is wrong somewhere.
    - "Recovery if you forget the new password": you cannot. Re-provision from source — see the existing "Recovery" section of this doc (forgotten-password = re-create from origin secrets).

    Section 3 (sub-section under Section 1 — H3 `### MCP Gateway desync caveat (T-269.1-07)`):
    - Phase 269.1's runtime rotate ONLY changes the BFF's view of the vault. The MCP Gateway (`banking_mcp_gateway`) loads the vault at ITS OWN startup using ITS OWN VAULT_PASSWORD env. After web-rotate, the gateway is still operating with the OLD password in its env; on next gateway restart it WILL fail to open the vault.
    - Operator workflow after web-rotate:
      1. Confirm web banner success: "Vault password rotated".
      2. Update `VAULT_PASSWORD` in every BFF + every gateway env (see Section 2 checklist).
      3. Restart MCP Gateway: `./run-bank.sh stop && ./run-bank.sh` (or just the gateway service if you have selective restart).
    - The gateway does NOT have a `/admin/vault` equivalent in this phase — it's intentionally out of scope; the gateway is a small, restart-tolerant service and adding an admin web surface there has diminishing returns.
