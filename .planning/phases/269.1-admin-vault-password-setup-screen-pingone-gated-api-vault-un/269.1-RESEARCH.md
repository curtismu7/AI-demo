# Phase 269.1: Admin vault-password setup screen ŌĆö Research

**Researched:** 2026-05-14
**Domain:** BFF admin route + React admin page + runtime vault unlock/rotate extension to Phase 269's startup-only loader
**Confidence:** HIGH (all critical findings verified against in-repo source; one design-tradeoff discussion flagged)

## Summary

Phase 269 (shipped today) loads the vault **only at BFF startup** ŌĆö `services/vaultLoader.js` reads `secrets.vault`, copies entries into `configStore` via `setRaw(data, {persist:false})`, calls `vault.close()` to zero the KEK, then `delete process.env.VAULT_PASSWORD`. There is no in-process way to unlock the vault after that point. Operators today must `npm run vault:rotate` on the host and restart the BFF to swap passwords.

269.1 closes the runtime gap with three HTTP endpoints (`POST /api/vault/unlock`, `POST /api/vault/rotate`, `GET /api/vault/status`) and a `/admin/vault` React page. **All existing 269 invariants are preserved**: vault values still bypass SQLite (`persist:false`), KEK/DEKs still zero on close, the password never appears in URL/query/log/audit. The only Phase 269 file extended is `services/vaultLoader.js` ŌĆö adding a sibling function `unlockVaultAtRuntime(...)` rather than mutating `loadVaultIntoConfigStore(...)` keeps the ┬¦1-protected startup path byte-identical.

**Primary recommendation:** Reuse the canonical admin gate `authenticateToken + requireAdmin` (verified in `routes/admin.js` line 31, 78, etc.); mount routes at `/api/admin/vault/*` (consistent with `/api/admin/config`, `/api/admin/mcp-gateway`, `/api/admin/management`); add a NEW `unlockVaultAtRuntime({password, vaultPath, configStore, vaultLib, logger})` function (no env-var coupling) rather than refactoring `loadVaultIntoConfigStore`; use the existing `express-rate-limit` package for unlock throttling (already a dep at `^7.1.5`); the React page imports `apiClient` (the same axios instance every other admin page uses) and lives at `components/AdminVaultPage.jsx`. Issue a 503 on `VERCEL=1`.

**Notable design tension (call out for the planner):** Phase 269 RESEARCH.md line 205 explicitly listed *"Putting the vault behind admin auth in the UI"* as an **anti-pattern** (*"Tempting, never appropriate"*) with the rationale that the UI doesn't need to see vault contents ŌĆö only the BFF does, at startup. The user's 269.1 brief consciously overrides that recommendation. This research treats the override as a locked decision (operator UX win > pure-isolation principle), but the planner should note the prior recommendation exists so the SUMMARY can record it as a deliberate deviation.

### Pattern 1: Sibling function ŌĆö `unlockVaultAtRuntime`

**What:** Add a new export to `services/vaultLoader.js`. Shape mirrors `loadVaultIntoConfigStore` but:
- `password` is REQUIRED (no `process.env.VAULT_PASSWORD` fallback)
- Does NOT call `delete process.env.VAULT_PASSWORD` (caller decides)
- Does NOT short-circuit on `VERCEL=1` (caller decides ŌĆö the route handler returns 503 before reaching this function on Vercel)
- DOES still call `vault.close()` in finally (KEK zeroing is non-negotiable)
- DOES still call `configStore.setRaw(data, {persist:false})` (Plan 01 contract)

**When to use:** Called only from the unlock route handler. The startup loader still calls `loadVaultIntoConfigStore` byte-identically.

**Example (illustrative ŌĆö planner refines):**
```javascript
// banking_api_server/services/vaultLoader.js
// EXISTING loadVaultIntoConfigStore stays exactly as it is.
// ADD:
async function unlockVaultAtRuntime({ password, vaultPath, configStore, vaultLib, logger }) {
  if (!password) {
    const err = new Error('vault: password required');
    err.code = 'VAULT_PASSWORD_MISSING';
    throw err;
  }
  vaultPath  = vaultPath  || process.env.VAULT_PATH || DEFAULT_VAULT_PATH;
  configStore = configStore || require('./configStore');
  vaultLib    = vaultLib    || require('../lib/vault');
  logger      = logger      || console;

  if (!fs.existsSync(vaultPath)) {
    const err = new Error('vault: file not found');
    err.code = 'VAULT_FILE_NOT_FOUND';
    throw err;
  }
  const vault = await vaultLib.openVault(vaultPath, password);  // throws VaultAuthError/VaultIntegrityError opaquely
  let entryCount = 0;
  try {
    const data = {};
    for (const name of vault.list()) {
      data[name.toLowerCase()] = vault.read(name);
      entryCount++;
    }
    if (entryCount > 0) await configStore.setRaw(data, { persist: false });
  } finally {
    try { vault.close(); } catch (_) {}
    // DELIBERATELY do NOT delete process.env.VAULT_PASSWORD ŌĆö this is runtime, not startup,
    // and caller's req.body.password was never in the env to begin with.
  }
  return { loaded: true, entries: entryCount };
}

module.exports = { loadVaultIntoConfigStore, unlockVaultAtRuntime, DEFAULT_VAULT_PATH };
```

**Why this shape:** Plan 03's `loadVaultIntoConfigStore` body is byte-identical to what shipped today; nothing in ┬¦1 row "Vault BFF startup" is at risk. The new function is a clean addition with its own test file.

### Pattern 2: Rotate route ŌĆö require unlock first + re-verify currentPassword

**What:** Rotate is a two-condition gate:
1. Vault must already be unlocked (i.e., `configStore` has vault-supplied entries cached). Otherwise: 423 `{error:'vault_locked', message:'Unlock the vault before rotating.'}`
2. `currentPassword` in the request body must successfully `openVault(path, currentPassword)` ŌĆö this is the re-verify. Then `handle.rotate(newPassword)` + `handle.save()`.

**Why both:** Plan 01's `handle.rotate(newPassword)` only re-wraps DEKs with the new KEK; it does NOT verify the caller knows the current password (it trusts the caller because they already opened the vault). By requiring an explicit `openVault(currentPassword)` in this route, we make rotate **independently authenticated against the file** ŌĆö even if the in-memory unlocked state is somehow stale, the file-level check catches it. This is defense-in-depth at trivial cost.

**Validation:** `newPassword.length >= 12`, `newPassword !== currentPassword`, both must be present strings. Don't impose stronger strength rules; Argon2id makes weak-passphrase brute-force already economic-only against very weak passphrases, and the operator is the only one who'll ever see this UI.

```javascript
// Pseudocode ŌĆö planner refines for routes/adminVault.js
router.post('/rotate', requireAdmin, async (req, res) => {
  const { currentPassword, newPassword } = req.body || {};
  // Input validation
  if (typeof currentPassword !== 'string' || typeof newPassword !== 'string') {
    return res.status(400).json({ error: 'bad_request', message: 'currentPassword and newPassword required' });
  }
  if (newPassword.length < 12) return res.status(400).json({ error: 'weak_password', message: 'new password must be Ōēź 12 chars' });
  if (currentPassword === newPassword) return res.status(400).json({ error: 'same_password', message: 'new password must differ from current' });

  // Concurrency guard (single async mutex)
  if (rotateInProgress) return res.status(409).json({ error: 'rotate_in_progress' });
  rotateInProgress = true;
  try {
    const vault = await vaultLib.openVault(vaultPath, currentPassword);  // re-verify
    try {
      await vault.rotate(newPassword);
      await vault.save();
    } finally { vault.close(); }
    recordAudit(vaultPath + '.audit.log', { op: 'rotate', key: null, caller: 'adminVault', result: 'ok' });
    res.json({ ok: true, message: 'Vault password rotated. Update VAULT_PASSWORD before next BFF restart.' });
  } catch (err) {
    // Opaque error for VaultAuthError + VaultIntegrityError; specific 500 for I/O
    const isAuth = err.name === 'VaultAuthError' || err.name === 'VaultIntegrityError';
    recordAudit(vaultPath + '.audit.log', { op: 'rotate', key: null, caller: 'adminVault', result: isAuth ? 'bad_password' : 'io_error' });
    if (isAuth) return res.status(401).json({ error: 'unauthorized', message: 'vault: open failed (bad password or tampered file)' });
    return res.status(500).json({ error: 'io_error', message: 'rotate failed' });
  } finally { rotateInProgress = false; }
});
```

### Pattern 3: Audit log reuse ŌĆö no new fields

**What:** `lib/vault/audit.js` accepts ONLY `{op, key, result, caller}` (verified line 24: `const ALLOWED = new Set(['op', 'key', 'result', 'caller']);` ŌĆö anything else throws). The audit writer ALSO automatically adds `ts`, `pid`, `host`. We cannot add `admin_sub` or `source_ip` to the audit line itself without changing `audit.js`.

**Recommendation: do NOT modify audit.js.** Instead:
- `op` values: `unlock`, `rotate` (new), plus existing `open`, `read`, `set`, `delete`, `save`, `close`
- `key` is `null` for unlock/rotate (whole-vault ops, like existing `rotate` already does)
- `result` values: `ok`, `bad_password`, `tampered`, `io_error`, `locked` (when rotate hit pre-unlock gate)
- `caller` is `'adminVault'` (analogous to `'vault.js'` for the CLI) ŌĆö this is how forensic analysis distinguishes CLI rotation from web rotation

If the user later wants admin_sub captured: log it via `logger.info(LOG_CATEGORIES.AUTHORIZATION, ...)` in the route handler ŌĆö that's the **separate, structured BFF log** (which already scrubs tokens per `bff-sessions` skill). The vault audit log stays narrow because that's its security invariant.

### Pattern 4: Concurrent-rotate mutex ŌĆö module-scoped boolean

**What:** A single `let rotateInProgress = false` flag at module scope of `routes/adminVault.js`. Set before `openVault`, cleared in finally. The route is admin-only and demo-scale; a real distributed lock would be over-engineering. Two admins clicking Rotate simultaneously will get one success + one 409 ŌĆö the human can retry.

**Why not `async-mutex` or similar:** Adding a dep for a single boolean is exactly the kind of over-engineering CLAUDE.md "minimum code" prohibits. The flag is process-local; multi-process BFF deployments don't exist in this repo today.

### Anti-Patterns to Avoid

- **Mutating `loadVaultIntoConfigStore` to accept a `{runtime: true}` flag.** Breaks REGRESSION_PLAN ┬¦1 row "Vault BFF startup" by changing the function's body. Add a sibling instead.
- **Adding `admin_sub` to the vault audit log line by extending `ALLOWED`.** That changes the audit module's invariant ("only 4 fields, ever") and is a one-way door. Log to the BFF structured logger instead.
- **CSRF token middleware just for these routes.** No other admin route in this repo uses CSRF tokens; they all rely on `sameSite=lax` + `httpOnly` + same-origin SPA fetches. Adding one for vault routes alone is inconsistent and signals false confidence.
- **Returning the password back to the SPA in the unlock response.** Even for "echo so the user can verify they typed it right." Never. The response is `{ok:true, entriesLoaded:N}` ŌĆö nothing more.
- **Letting the rotate handler log `req.body`.** The password is in there. Strip before any `logger.info(...)` / `console.log(...)`. Easiest: never log the body at all.
- **Falling back to `localhost` in docs/examples.** Use `https://api.ping.demo:4000/admin/vault` everywhere.
- **Hardcoding the vault path inside the route.** Use `DEFAULT_VAULT_PATH` from `vaultLoader.js` (or `process.env.VAULT_PATH || DEFAULT_VAULT_PATH` for runtime override) ŌĆö same source of truth Plan 03 + Plan 05 use.
- **Calling `delete process.env.VAULT_PASSWORD` from the runtime route handler.** Phase 269's startup loader already deleted it. The runtime password came from `req.body`, not env. There's nothing to delete; pretending otherwise muddles the security model.

## Common Pitfalls

### Pitfall 1: Forgetting to update `VAULT_PASSWORD` env var after rotating via UI

**What goes wrong:** Operator rotates via `/admin/vault`, walks away, BFF restarts overnight (deploy / pm2 restart / VM reboot). Startup tries the now-stale `VAULT_PASSWORD` from `.env` / pm2 ecosystem / shell ŌĆö `loadVaultIntoConfigStore` throws `VaultAuthError`, BFF exits 1, demo is down.

**Why it happens:** The UI rotates the **file**, not the **env var**. There's no way for the BFF to update `.env` for the operator without writing to disk (which expands the attack surface and conflicts with `secrets.vault` being the canonical at-rest format).

**How to avoid:** Plan 04 / 05 update `docs/vault.md` with an explicit post-rotate checklist:
> After `/admin/vault` reports rotation success, update `VAULT_PASSWORD` in **all three** places that might run the BFF:
> 1. Your shell's `.env` file (`banking_api_server/.env`)
> 2. Any pm2 ecosystem file (`ecosystem.config.js` env block) if you use pm2
> 3. Your secret manager (1Password / SOPS / direnv `.envrc`) if you use one
>
> The post-rotation success banner in the UI must restate this verbatim.

**Warning signs:** Operator reports "BFF won't start" within 24 hours of using the rotate feature.

### Pitfall 2: Two admins rotating simultaneously

**What goes wrong:** Both pass their respective `currentPassword` (same value); both call `openVault`; both call `handle.rotate(newPassword_A)` and `handle.rotate(newPassword_B)` interleaved; last-write-wins on `handle.save()`; one admin gets `{ok:true}` but their password doesn't work.

**Why it happens:** No mutex.

**How to avoid:** Module-scoped `rotateInProgress` boolean. Second call returns 409. Trivial; covered by integration test "two parallel rotates ŌåÆ one 200 one 409."

**Warning signs:** Audit log shows two `rotate ok` entries within milliseconds; one admin can't log in afterward.

### Pitfall 3: Wrong-password unlock generating a timing oracle

**What goes wrong:** Wrong-password unlock takes ~60-300ms (Argon2id KDF). Wrong-payload (bad JSON) returns in <1ms. An attacker timing the response distinguishes "tried but failed" from "didn't reach openVault" ŌĆö useful for probing whether the rate limit is engaged.

**Why it happens:** Argon2id is intentionally slow.

**How to avoid:** Accept it. The legitimate use case is one admin typing a password; 60-300ms is invisible to humans. The rate limit (5 attempts / 5 min) caps damage. **Do not** try to "normalize" timing ŌĆö every "fix" leaks more information.

**Warning signs:** PR adding `setTimeout` around the openVault call.

### Pitfall 4: Audit log not flushed on opaque-error path

**What goes wrong:** Wrong password ŌåÆ `openVault` throws BEFORE the route handler can call `recordAudit`. The audit log is missing the failed-unlock event.

**Why it happens:** Plan 01's `openVault` audits its OWN failures (line 124 records `result: 'bad_password'`; line 162 records `result: 'integrity_failed'`). So the vault-library-level audit covers this BUT the `caller` field will be `'vault.js'` ŌĆö looking like CLI activity, not web.

**How to avoid:** In the route handler's catch block, ALSO call `recordAudit(..., {op:'unlock', caller:'adminVault', result:'bad_password'})` so the entry is attributed to the web surface. Two audit lines per failed web-unlock (one from `openVault` with caller `vault.js`, one from the route with caller `adminVault`) is fine ŌĆö it shows the propagation path.

**Warning signs:** Forensic analyst can't tell whether a `bad_password` event came from CLI or web. (Solution: the `adminVault` caller line will exist.)

### Pitfall 5: `requireAdmin` requires `authenticateToken` to run first

**What goes wrong:** Mount `app.use('/api/admin/vault', vaultAdminRoutes)` WITHOUT `authenticateToken` in front; routes inside use `requireAdmin`; `req.user` is undefined; `requireAdmin` throws `AUTHENTICATION_REQUIRED` 401 even for legit admin sessions.

**Why it happens:** `requireAdmin` reads `req.user` (line 858); `authenticateToken` is what sets it (line 765-788 / 644-660). The mount order matters.

**How to avoid:** Mount exactly like the existing admin route:
```javascript
// server.js
app.use('/api/admin/vault', authenticateToken, require('./routes/adminVault'));
```

This matches `app.use('/api/admin', authenticateToken, adminRoutes)` at line 896.

**Warning signs:** First integration test fails with 401 even when the test sets a valid session.

### Pitfall 6: Returning `{entries: vault.list()}` in the status response

**What goes wrong:** Status endpoint enumerates entry names. Even names like `HELIX_API_KEY` reveal what the BFF has secrets for, which informs reconnaissance.

**Why it happens:** "Let me show the operator what got loaded so they can verify."

**How to avoid:** Return only `{unlocked: bool, entriesLoaded: <number>, vaultFilePresent: bool}`. Number-of-entries is fine (operationally useful, attacker-irrelevant). Names are not.

**Warning signs:** `vault.list()` referenced in any GET status handler.

```javascript
// banking_api_server/routes/adminVault.js
'use strict';
const express = require('express');
const rateLimit = require('express-rate-limit');
const path = require('node:path');
const fs = require('node:fs');
const { requireAdmin } = require('../middleware/auth');
const configStore = require('../services/configStore');
const { unlockVaultAtRuntime, DEFAULT_VAULT_PATH } = require('../services/vaultLoader');
const vaultLib = require('../lib/vault');
const { recordAudit } = require('../lib/vault/audit');

const router = express.Router();

// Reject all vault routes on Vercel (serverless uses Encrypted Environment Variables instead)
router.use((req, res, next) => {
  if (process.env.VERCEL === '1') {
    return res.status(503).json({
      error: 'vault_disabled_serverless',
      message: 'Vault is disabled on Vercel. Use Encrypted Environment Variables.',
    });
  }
  next();
});

const unlockLimiter = rateLimit({
  windowMs: 5 * 60 * 1000,
  max: 5,
  standardHeaders: true,
  legacyHeaders: false,
  keyGenerator: (req) => req.user?.sub || req.ip,
  message: { error: 'too_many_requests', message: 'Too many unlock attempts. Wait 5 minutes.' },
});

const resolveVaultPath = () => process.env.VAULT_PATH || DEFAULT_VAULT_PATH;

let rotateInProgress = false;

router.get('/status', requireAdmin, (req, res) => {
  const vaultPath = resolveVaultPath();
  const vaultFilePresent = fs.existsSync(vaultPath);
  // "unlocked" heuristic: vault file exists AND at least one vault-derived key
  // has been seen by configStore in this process. Planner decides the canonical
  // probe ŌĆö likely a flag set by unlockVaultAtRuntime + loadVaultIntoConfigStore.
  res.json({
    unlocked: configStore.isVaultUnlockedThisProcess?.() ?? false,
    entriesLoaded: configStore.vaultEntryCountThisProcess?.() ?? 0,
    vaultFilePresent,
    vaultPath: path.basename(vaultPath),  // basename only ŌĆö don't leak full path
  });
});

router.post('/unlock', requireAdmin, unlockLimiter, async (req, res) => {
  const { password } = req.body || {};
  if (typeof password !== 'string' || password.length === 0) {
    return res.status(400).json({ error: 'bad_request', message: 'password (string) required' });
  }
  const vaultPath = resolveVaultPath();
  try {
    const result = await unlockVaultAtRuntime({
      password,
      vaultPath,
      configStore,
      vaultLib,
    });
    recordAudit(vaultPath + '.audit.log', {
      op: 'unlock', key: null, caller: 'adminVault', result: 'ok',
    });
    res.json({ ok: true, entriesLoaded: result.entries });
  } catch (err) {
    const isAuth = err.name === 'VaultAuthError' || err.name === 'VaultIntegrityError';
    const isMissing = err.code === 'VAULT_FILE_NOT_FOUND';
    recordAudit(vaultPath + '.audit.log', {
      op: 'unlock', key: null, caller: 'adminVault',
      result: isAuth ? 'bad_password' : (isMissing ? 'not_found' : 'io_error'),
    });
    if (isMissing) return res.status(404).json({ error: 'vault_file_not_found' });
    if (isAuth) return res.status(401).json({
      error: 'unauthorized',
      message: 'vault: open failed (bad password or tampered file)',
    });
    return res.status(500).json({ error: 'io_error', message: 'unlock failed' });
  }
});

router.post('/rotate', requireAdmin, async (req, res) => {
  // (See Pattern 2 above)
});

module.exports = router;
```

## Open Questions

1. **Should the new sidebar nav include a "Vault" entry?**
   - What we know: every other admin page is reachable via `AdminSideNav`. Direct URL access works regardless.
   - What's unclear: Adding it to the nav puts vault management in front of every admin who logs in ŌĆö including admins who shouldn't ordinarily touch vault.
   - Recommendation: Plan 03 (UI) adds it under "System Tools ŌåÆ Vault" (the existing System Tools section is admin-only). Don't add to a top-level entry. If planner prefers no nav entry, that's acceptable ŌĆö the URL is enough for the operator who needs it.

2. **Does rotate need to "re-load" entries into configStore?**
   - What we know: After rotate, the file is re-encrypted with the new KEK, but the in-memory configStore cache still has the values from when the vault was unlocked. The values themselves haven't changed.
   - What's unclear: If rotate happens BEFORE unlock (we forbid this in REQ-VAULT-ADMIN-02 by returning 423), this question is moot. If rotate happens AFTER unlock (the standard flow), no re-load is needed because nothing changed in the configStore.
   - Recommendation: Rotate must NOT call `setRaw` again. Document explicitly in the route's docstring: "rotate only re-wraps DEKs in the file; configStore cache is untouched because entry values are unchanged."

3. **Should the unlock route also accept `VAULT_PATH` override via body?**
   - What we know: Plan 01's CLI accepts `VAULT_PATH` env var override. Plan 03's loader honors it.
   - What's unclear: Should the web UI let admins specify a non-default path?
   - Recommendation: NO. The path comes from `process.env.VAULT_PATH || DEFAULT_VAULT_PATH` ŌĆö same as everywhere else. Letting the web UI specify arbitrary file paths is a path-traversal vector. If an operator runs the BFF against a non-default vault, they set `VAULT_PATH` in the environment, same as before.

4. **Does the React page need a "logout" / "lock" button to drop the unlocked state?**
   - What we know: There's no API to "re-lock" today. configStore can't selectively forget vault-derived entries without re-implementing the loader's reverse.
   - What's unclear: Operator UX after rotate ŌĆö do they want a clear way to "wipe" the in-memory cache?
   - Recommendation: Plan 03 (UI) adds NO lock button. The only way to drop unlocked state is BFF restart, which is the expected operator flow after a sensitive event. Add a doc note: "To re-lock: restart the BFF."

### Known Threat Patterns for Express + React + cookie-session

| Threat ID | Pattern | STRIDE | Standard Mitigation | Status in this phase |
|-----------|---------|--------|---------------------|----------------------|
| T-269.1-01 | Unauthenticated POST to `/api/admin/vault/unlock` | Spoofing / Elevation of Privilege | Reuse `authenticateToken + requireAdmin` (already validates PingOne JWT + role) | Mitigated by route mount in `server.js` |
| T-269.1-02 | CSRF on unlock POST (attacker site posts to the BFF using the admin's cookie) | Tampering | `sameSite=lax` cookie + admin-only auth + same-origin SPA fetch. Modern browsers refuse cross-site `lax` cookies on non-GET. JSON-body content-type defeats `<form>` submission CSRF | Mitigated by existing posture; **no new middleware required** |
| T-269.1-03 | Password leaked in URL/query/log/audit | Information Disclosure | POST body only; route handlers never log `req.body`; vault audit log allows only 4 specific fields and rejects the rest | Mitigated by design (REQ-VAULT-ADMIN-05) |
| T-269.1-04 | Brute-force unlock via repeated guesses | Spoofing | `express-rate-limit` 5 attempts / 5 min per session sub (or IP if anonymous, which should be impossible behind admin gate) + Argon2id slow KDF (~60-300ms per attempt) | Mitigated by REQ-VAULT-ADMIN-08 |
| T-269.1-05 | Rotate without re-verifying current password (in-memory unlock state stale or compromised) | Elevation of Privilege / Tampering | Rotate route requires `currentPassword` in body AND re-opens the vault file with it before calling `handle.rotate(newPassword)`. Defense-in-depth even when caller already has `req.user.role === 'admin'` | Mitigated by REQ-VAULT-ADMIN-02 + Pattern 2 |
| T-269.1-06 | Concurrent rotate by two admins ŌåÆ last-write-wins, one admin's new password silently lost | Tampering | Module-scoped `rotateInProgress` boolean ŌĆö second concurrent call gets 409; admin retries | Mitigated by REQ-VAULT-ADMIN-10 + Pattern 4 |
| T-269.1-07 | Vault state desync ŌĆö BFF unlocked, MCP Gateway still locked | Information Disclosure (false sense of unlocked-ness) | Document explicitly that this phase does NOT propagate unlock to MCP Gateway. Gateway still uses Phase 269 Plan 04's startup-load from its OWN env. Operator-restart of gateway is still required after rotate | Documented; out of scope for this phase |
| T-269.1-08 | Vault password retained in `req.body` long after handler return | Information Disclosure (memory dump) | Express discards `req` references after response end; no explicit retention. Don't store `req.body.password` on any longer-lived object (e.g., session, module-scope cache) | Mitigated by handler discipline (REQ-VAULT-ADMIN-05) |
| T-269.1-09 | Path traversal via `VAULT_PATH` env override | Tampering / Information Disclosure | `VAULT_PATH` comes from environment only (operator-trusted); never from request body. Route handler uses `process.env.VAULT_PATH \|\| DEFAULT_VAULT_PATH` exclusively | Mitigated by design (REQ-VAULT-ADMIN-04 docstring + Open Question 3) |
| T-269.1-10 | Timing oracle on wrong vs. nonexistent vault file | Information Disclosure | Both paths log opaquely via the audit module; HTTP responses are distinct (401 vs 404) but `npm view` of audit log is admin-only too. Acceptable tradeoff: 404 is operationally useful for "did you create the vault yet?" | Accepted; documented |

**STRIDE summary:** Spoofing (T-01, T-04) ŌåÆ handled by existing auth + new rate limit. Tampering (T-02, T-05, T-06, T-09) ŌåÆ handled by sameSite + currentPassword re-verify + mutex + env-only path source. Repudiation ŌåÆ all unlock/rotate events go to the vault audit log AND the BFF structured logger (admin sub + IP captured separately). Information Disclosure (T-03, T-07, T-08, T-10) ŌåÆ password never leaves the route handler, no entry-name enumeration in status, gateway-desync documented. Denial of Service ŌåÆ rate limit caps unlock; rotate is admin-only and single-tenant. Elevation of Privilege (T-01, T-05) ŌåÆ admin gate + currentPassword re-verify.
