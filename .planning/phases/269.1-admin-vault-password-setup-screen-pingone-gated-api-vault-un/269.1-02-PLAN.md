## Middleware ordering (CRITICAL — drives 401 vs 403 vs 503 ordering)

The mount line in server.js is:
```
app.use('/api/admin/vault', authenticateToken, require('./routes/adminVault'));
```

Express runs middleware in registration order. For ANY request to `/api/admin/vault/*`:
1. `authenticateToken` (from server.js mount) runs FIRST. Unauthenticated requests → 401, never reach the router.
2. The vault router's `router.use(vercelGuard)` runs SECOND. So an authenticated request hits the Vercel guard before any handler.
3. Per-handler middleware (`requireAdmin`, `unlockLimiter`, then handler) runs THIRD.

Net result on Vercel:
- Unauthenticated probers get **401** (from outer `authenticateToken`) — no decryption oracle, no leak about whether the vault routes exist.
- Authenticated non-admin probers get **503 vault_disabled_serverless** (from router-level guard before requireAdmin). Acceptable — they already proved login, no oracle.
- Authenticated admin probers get **503 vault_disabled_serverless** (same guard). Operator sees the documented error.
This ordering is intentional and is asserted by regression test 16 (which sets req.user via the test helper, so reaches the vercel guard).

## HTTP Contracts (implement EXACTLY)

`GET /api/admin/vault/status`:
- 200: `{ unlocked: boolean, entriesLoaded: number, vaultFilePresent: boolean, vaultPath: string }`
  - `vaultPath` is `path.basename(...)` ONLY — never the full path (T-269.1-09)
  - NEVER include entry names (Pitfall 6)
- 401: AUTHENTICATION_REQUIRED (authenticateToken)
- 403: INSUFFICIENT_SCOPE (requireAdmin)
- 503: `{ error: 'vault_disabled_serverless', message: '...' }` when `process.env.VERCEL === '1'`

`POST /api/admin/vault/unlock` body `{ password: string }`:
- 200: `{ ok: true, entriesLoaded: number }` — password NEVER echoed
- 400: `{ error: 'bad_request', message: 'password (string) required' }`
- 401 (vault open failed): wrong password OR tampered file — `{ error: 'unauthorized', message: 'vault: open failed (bad password or tampered file)' }` — IDENTICAL message both cases
- 403: non-admin
- 404: `{ error: 'vault_file_not_found' }` (vault file absent)
- 429: rate-limited (6th attempt in 5-min window per req.user.sub)
- 500: `{ error: 'io_error', message: 'unlock failed' }`
- 503: Vercel

`POST /api/admin/vault/rotate` body `{ currentPassword: string, newPassword: string }`:
- 200: `{ ok: true, message: 'Vault password rotated. Update VAULT_PASSWORD before next BFF restart.' }`
- 400: `{ error: 'bad_request' | 'weak_password' | 'same_password', message: '...' }`
- 401: wrong currentPassword (same opaque message as unlock)
- 403: non-admin
- 409: `{ error: 'rotate_in_progress' }` (mutex held)
- 423: `{ error: 'vault_locked', message: 'Unlock the vault before rotating.' }` (isVaultUnlockedThisProcess() === false)
- 500: io_error
- 503: Vercel

## Audit Lines (NDJSON to `<vaultPath>.audit.log`)

Each handler calls `recordAudit` once per outcome. `result` values: `ok` | `bad_password` | `not_found` | `io_error` | `locked`. `op` values: `unlock` (new) | `rotate` (reused from CLI). `caller` is ALWAYS literal string `'adminVault'`. GET /status is NOT audited (read-only metadata; per RESEARCH §"Audit log reuse").
