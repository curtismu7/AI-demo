### WR-02: vault status returns entriesLoaded as integer (minor inventory disclosure)

**Files modified:** `banking_api_server/routes/adminVault.js`
**Commit:** 0357eae8
**Applied fix (Option B — accepted, with comment):**

- Added a 6-line block comment immediately above the `router.get('/status', ...)` handler documenting why the integer count is intentional rather than being downgraded to a boolean:
  - Operators need "vault loaded N entries as expected" confirmation after unlock/rotate (operator UX).
  - Any caller reaching this endpoint is already authenticated AND admin (requireAdmin) — inside the trust boundary.
  - Entry names are still never returned; only the count.
  - Disposition: `accepted (admin-only; no enumeration of names)`.
- No code or behavior change — comment-only edit on the `/status` handler. UI, tests, and contract are byte-identical.

Option B was chosen over Option A (boolean downgrade) per reviewer's note that "either is acceptable" and per the user's recommendation that "admins seeing entry count is a feature (operator UX confirmation that vault loaded correctly)." This is the minimal-diff path that closes the threat model gap without churning the UI or its 12 RTL tests.
