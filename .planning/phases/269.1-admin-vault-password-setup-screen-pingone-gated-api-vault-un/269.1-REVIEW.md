# Phase 269.1: Code Review Report

**Reviewed:** 2026-05-14T00:00:00Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Phase 269.1 adds a runtime vault unlock/rotate admin surface that complements
Phase 269's startup-only contract. The implementation is unusually careful
about the documented threat model:

- **Token custody preserved.** Vault password is held only in component-local
  `useState`, cleared on success, sent only via `apiClient.post` to the BFF.
  No localStorage, no URL, no query params, no global state.
- **Opaque error UX is verifiably byte-identical** for `VaultAuthError` and
  `VaultIntegrityError` (the constant string
  `'vault: open failed (bad password or tampered file)'` is hard-coded in
  both `routes/adminVault.js` and `services/vaultLoader.js`; the regression
  test #8 asserts byte-equality across the two error paths).
- **Rate limiter is correctly post-auth keyed** to `req.user?.sub || req.ip`
  with the `requireAdmin` middleware running BEFORE the limiter, so the IP
  fallback only matters if `req.user` is missing — which it cannot be after
  `requireAdmin` accepts the request.
- **Rotate mutex** (`rotateInProgress` boolean in `try/finally`) is correctly
  scoped and clearable on every code path (including the early-return paths
  before `rotateInProgress = true` is set).
- **Audit allowlist** (4-field `{op, key, result, caller}`) is unchanged from
  Phase 269; the new route only ever calls `recordAudit` with the
  `caller:'adminVault'` discriminator. The audit module physically cannot
  serialize a password because the allowlist throws on unexpected fields
  before `JSON.stringify` runs.
- **Minimal-diff discipline preserved.** `server.js` is +1 line at line 899
  (the new `app.use('/api/admin/vault', ...)` mount). The Phase 269 vault
  loader IIFE around `.listen` is byte-identical. `vaultLoader.js`'s existing
  `loadVaultIntoConfigStore` body gains exactly 2 state-mirror lines on its
  success path (`_unlocked = true; _entriesLoaded = entryCount;`).
- **REGRESSION_PLAN.md edit is verified append-only.** `git diff
  b3a31c51^ b3a31c51 -- REGRESSION_PLAN.md | grep '^-[^-]'` returns 0 — no
  deletion lines, only the one diff header. New §1 row + §4 entry added at
  the correct positions.
- **UI conformance.** Page uses `apiClient` exclusively (no raw axios/fetch);
  `type="password"` on all four inputs; only ⚠️ ✅ ❌ emojis appear in
  rendered output; RTL tests assert the emoji rule against the entire
  Unicode misc-symbols + pictographs ranges.
- **Test coverage strong.** 12 vaultLoader-runtime tests + 18 mocked route
  tests + 6 real-vault integration tests + 12 RTL UI tests = 48 new tests
  exercising the threat model.

The findings below are quality items and one minor security hygiene issue
related to the SPA traffic inspector, not functional defects.

## Warnings

### WR-01: rotate request body fields not in apiTrafficStore redaction list — admin Traffic Inspector will show currentPassword and newPassword in cleartext

**File:** `banking_api_ui/src/services/apiTrafficStore.js:69-72` (existing constant) — surfaced by `banking_api_ui/src/components/AdminVaultPage.jsx:137-140`
**Issue:** `apiClient` records every request body through `redactBody()`, which has a fixed allowlist of redacted keys:

```js
const REDACT_BODY_KEYS = new Set([
  'access_token', 'refresh_token', 'id_token', 'client_secret', 'password', 'token',
  'client_credentials', 'code_verifier',
]);
```

The `/unlock` endpoint sends `{password: ...}` which IS redacted. However, `/rotate` sends `{currentPassword, newPassword}` — neither key is in the allowlist. The resulting traffic entry stored in `apiTrafficStore` and rendered on the admin Traffic Inspector page (`/api-traffic` or wherever the SPA surfaces this store) will contain both passwords in cleartext, visible to anyone who can subsequently load the inspector in the same browser session — including the next admin who logs in if the store is not cleared, or anyone who can see the screen. This is a meaningful local-disclosure regression because banking_api_ui already redacts `password` for `/unlock` but not its sibling key names for `/rotate`.

The threat model document T-UI-01 says "banner text never echoes typed passwords" — that is true, but the traffic inspector is a separate surface that should also not echo them.

**Fix:** Add the two new key names to `REDACT_BODY_KEYS` (or switch to a case-insensitive substring match on `/password/i`):

```js
const REDACT_BODY_KEYS = new Set([
  'access_token', 'refresh_token', 'id_token', 'client_secret', 'password', 'token',
  'client_credentials', 'code_verifier',
  'currentpassword', 'newpassword',  // Phase 269.1 — /api/admin/vault/rotate body
]);
```

Note that `REDACT_BODY_KEYS.has(String(k).toLowerCase())` is already case-insensitive, so lowercase entries are correct. Alternatively, switch the comparison to a `Set` of regexes or do a substring check on `'password'`. Add a one-line RTL test under `AdminVaultPage.test.jsx` that simulates a rotate and asserts the traffic-store entry has `***` rather than the typed password.

### WR-02: vaultEntryCountThisProcess() leaks the number of secrets in the vault to any admin via /status — minor information disclosure

**File:** `banking_api_server/routes/adminVault.js:99` + `banking_api_server/services/vaultLoader.js:192`
**Issue:** `GET /api/admin/vault/status` returns `entriesLoaded: vaultEntryCountThisProcess()` to any authenticated admin. The §1 row in REGRESSION_PLAN.md explicitly forbids `GET /status` from returning entry names, but the entry COUNT is still a small inventory disclosure: a compromised admin session learns whether the vault has, say, 7 secrets (the demo's canonical 9-entry allowlist minus 2 not-set values) or only 1 (typo / partial migration). This is a low-severity item — the count is already inferable by anyone with /admin login because they can also call any of the existing /admin/config endpoints to enumerate effective-config keys — but if the threat model wants strict minimization, the count is information that does not need to leave the BFF.

**Fix:** Either:

1. **Document the disclosure as accepted** in `docs/vault.md` "Endpoint reference" table with a `T-269.1-XX` ID and disposition `accepted (admin-only; no enumeration of names)`. The endpoint already redacts names; documenting the count carve-out closes the gap.
2. **Or replace the integer with a boolean** (`entriesLoaded: vaultEntryCountThisProcess() > 0`) — the UI uses the value only as "Entries loaded: 5" decoration; reducing it to "Entries loaded: yes/no" preserves the operational signal without leaking the cardinality.

Option 1 is the minimal-diff choice; option 2 is the security-tightening choice. Either is acceptable; both close the threat model gap.

## Info

### IN-01: AdminVaultPage refresh button has no busy state — admin can spam clicks during in-flight /status

**File:** `banking_api_ui/src/components/AdminVaultPage.jsx:187-193`
**Issue:** The "Refresh status" button has no `disabled={status.loading}` attribute, so a fast clicker can fire many parallel `apiClient.get('/api/admin/vault/status')` requests. Each request fires its own spinner increment/decrement pair; the last response wins in `setStatus`. No correctness issue (status is idempotent), but the UX is uneven.

**Fix:**
```jsx
<button
  type="button"
  className="admin-vault-page__refresh"
  onClick={refresh}
  disabled={status.loading}
>
  Refresh status
</button>
```

### IN-02: AdminVaultPage status error path loses the loaded values — admin sees "Failed to load" even after one prior success

**File:** `banking_api_ui/src/components/AdminVaultPage.jsx:48-55`
**Issue:** When `refresh()` fails after a prior success, `setStatus({ loading: false, error: msg })` discards the previously-loaded `unlocked`, `entriesLoaded`, `vaultFilePresent`, `vaultPath`. The error UI then shows only "❌ <msg>" with no fallback to the last-known state. For a long-running admin session this is a minor regression in visible information — the prior section just disappears.

**Fix:** Spread the prior state and overlay only the error:
```jsx
} catch (err) {
  const msg = err.response?.data?.message || err.response?.data?.error || err.message || 'Failed to load vault status';
  setStatus((prev) => ({ ...prev, loading: false, error: msg }));
}
```
Then in the render, show the cached fields alongside the banner if both are present. Optional; not a correctness defect.

### IN-03: Eager import of AdminVaultPage costs ~6KB in the main bundle for non-admin users

**File:** `banking_api_ui/src/App.js:26`
**Issue:** Phase 269.1 Plan 03 explicitly locked the import to eager (not `React.lazy`) — verified at `import AdminVaultPage from "./components/AdminVaultPage";` on line 26. The decision is documented in the plan as intentional (simpler, no Suspense fallback flash, avoids code-split chunk hash drift in tests). For a customer who never visits `/admin/vault`, this adds the ~6KB component + the CSS to the initial bundle. Not a defect — the plan made this trade explicitly. Recording here only so the trade-off remains visible if a future bundle audit revisits CRA chunking.

**Fix:** Leave as-is — the lazy alternative was rejected in the plan. If a future phase code-splits the admin pages as a group, AdminVaultPage should ride along.

### IN-04: AdminVaultPage rate-limit (429) does not display Retry-After to the admin

**File:** `banking_api_ui/src/components/AdminVaultPage.jsx:88-94`
**Issue:** When the BFF returns 429 with a `Retry-After` header (express-rate-limit ^7 emits it on `standardHeaders: true`), `extractMessage(err)` falls through to `err.response?.data?.error || err.message`. The user sees "❌ too_many_requests" (or "❌ Too many unlock attempts. Wait 5 minutes." per the server's static message) without a countdown. This is fine for a demo — `docs/vault.md` documents the 5-attempts-per-5-min window — but a small UX win is to surface the header value.

**Fix:** Read `err.response?.headers?.['retry-after']` when status is 429 and append "Try again in N seconds" to the banner. One-line change, not blocking.
