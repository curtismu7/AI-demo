# Plan 237-04 — RFC 9728 Live Metadata & Token Audience Chain Diagram

## Objective

RFC 9728 is the most concrete, testable RFC in the system — every running service
responds at `/.well-known/oauth-protected-resource`. Make the UI fetch and display
the real response, not a static example. Also add a visual audience chain diagram
so the olb-resource → mcp-gw → mcp-olb/mcp-invest flow is immediately clear.

## Tasks

### T1 — RFC 9728 live metadata fetch

**BFF route** (`banking_api_server`):
- `GET /api/rfc9728/all` already exists or create it — returns metadata from:
  - BFF own endpoint: `/.well-known/oauth-protected-resource`
  - Proxied fetch from mcp-server, mcp-gateway, mcp-invest (server-side, avoids CORS)
- Return as:
  ```json
  {
    "bff":     { ...metadata... },
    "mcp_olb": { ...metadata... },
    "mcp_gw":  { ...metadata... },
    "mcp_invest": { ...metadata... }
  }
  ```
- Each entry should include a `_status` field: `"ok"` | `"unreachable"` | `"error"`

**UI** (`education/RFC9728Content.js` or `enhancedRFC9728Content.js`):
- Replace static example JSON with a "Fetch Live Metadata" button
- On click: `GET /api/rfc9728/all` → render each service's response in a collapsible card
- Each card shows:
  - Service name + port
  - Response fields with per-field annotations (required vs optional per RFC 9728 §2)
  - `_status` badge (green/yellow/red)
- Link to RFC 9728: `<RfcLink rfc="RFC_9728" />`
- Merge/deduplicate the two existing RFC 9728 content files (`RFC9728Content.js` and
  `enhancedRFC9728Content.js`) into one — remove the weaker one

**Required RFC 9728 fields to annotate:**
| Field | Required? | Meaning |
|---|---|---|
| `resource` | REQUIRED | Resource server URI |
| `authorization_servers` | RECOMMENDED | List of AS URIs |
| `bearer_methods_supported` | OPTIONAL | `["header"]` |
| `scopes_supported` | OPTIONAL | Scopes this RS accepts |
| `resource_documentation` | OPTIONAL | Link to API docs (OpenAPI) |

### T2 — Token audience chain diagram component

Create `src/components/education/TokenAudienceChain.js`:

Visual diagram (CSS-based, no external library):
```
┌─────────────────────────────────────────────────────────────┐
│  TOKEN AUDIENCE CHAIN                          [RFC 8693]   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [Login]         [Exchange #1]       [Exchange #2]          │
│  User Token   ──────────────►  GW Token  ──────────────►  Backend Token  │
│  aud: olb-                     aud: mcp-gw                aud: mcp-olb   │
│  resource                      act: agent1                act: agent1    │
│  .bxf.com                                                 (or mcp-invest)│
│                                                                           │
│  sub=user preserved throughout ──────────────────────────────────────►   │
│  act chain added at Exchange #1, preserved at #2                          │
└─────────────────────────────────────────────────────────────────────────┘
```

Each node is clickable — clicking a token node opens its decoded claims example.
The two exchange arrows show "RFC 8693" as a badge link.

Embed this component:
- In the RFC 8693 education panel (new "Audience Chain" tab)
- In `TokenChainDisplay.js` as a collapsible "How this works" section above the live chain
