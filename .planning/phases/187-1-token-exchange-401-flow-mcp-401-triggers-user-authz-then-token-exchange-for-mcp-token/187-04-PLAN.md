    **Content must include:**
    1. When this flow applies: user is authenticated (has access token in session) but the MCP server returns HTTP 401 because the user token is not a valid MCP token (wrong audience, missing scopes, or plain access token without exchange)
    2. Flow steps:
       - User agent sends request to MCP server using raw user access token
       - MCP server returns 401 (Bearer token required / wrong aud)
       - BFF detects 401 → sets `need_auth: true` in response
       - Client (BankingAgent.js) sees `need_auth: true` → triggers PingOne customer login (PKCE flow)
       - After re-auth, BFF performs RFC 8693 1-token exchange: user access token (subject only) → MCP token with MCP audience
       - Retry: MCP server accepts the exchanged token → request completes
    3. Comparison note: distinguish from Phase 184/186 dual-token flows:
       - Phase 184: user access token (subject) + agent CC token (actor) → MCP token with `act` claim (delegation proof)
       - Phase 186: user ID token (subject) + agent CC token (actor) → MCP token
       - Phase 187: user access token (subject only, no actor) → MCP token; simpler, no delegation chain; triggered by live 401 not a proactive exchange
    4. When to use: lightweight scenarios where the MCP server only needs to know "which user" without requiring a full delegation (act) chain; or as the fallback recovery path when a tool call fails with 401
    
    Match the formatting style of existing sections in the file.
    
    Per D-07: Do NOT modify `PINGONE_MAY_ACT_TWO_TOKEN_EXCHANGES.md`.
