## Transport Target

| Option | Description | Selected |
|--------|-------------|----------|
| h2c (plaintext) (Recommended) | http2.createServer + allowHTTP1, no cert | ✓ |
| h2 (TLS) | http2.createSecureServer with mkcert |  |
| Both — h2c locally, h2 via env var | Maximum flexibility, more config to test |  |

**User's choice:** h2c (plaintext)
**Notes:** Matches existing `http://localhost:8080`, bridge client already supports h2c via `allowHTTP1: true`. Cost: no realistic-production TLS story — captured in Deferred Ideas.

## WebSocket Coexistence Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Shared port via ALPN downgrade | http2.createServer({ allowHTTP1: true }) + ws.Server on the same 8080 |  |
| Split ports — 8080 stays WS, new port for h2 (Recommended) | Add 8083 for h2 only; WS untouched | ✓ |
| Drop WS entirely from MCP server | All clients move to h2 |  |

**User's choice:** Split ports — 8080 stays WS, new port for h2
**Notes:** Avoids the unsolved question of whether `ws.Server` cleanly coexists with `http2.createServer` on the same port across Node versions. Zero risk to existing WS clients (langchain_agent, the BFF's current WS path).

### Follow-up: Port and config

| Option | Description | Selected |
|--------|-------------|----------|
| 8083 + new `MCP_SERVER_HTTP2_URL` env var (Recommended) | New env var, WS stays at MCP_SERVER_URL | ✓ |
| 8083 + reuse `MCP_SERVER_URL` with `http://` prefix | Single env var, scheme switches transport |  |
| 8081-range — keep close to MCP_INVEST (8081) | 8083 stays in the existing port-range cluster | (8083 falls in this range — covered) |

**User's choice:** 8083 + new `MCP_SERVER_HTTP2_URL` env var
**Notes:** Mirrors the existing `MCP_GATEWAY_HTTP_URL` opt-in pattern at `server.js:1727`. Doesn't risk regressing existing `MCP_SERVER_URL` semantics — important because §1 REGRESSION_PLAN rows already reference it.

## Default Transport for the Demo

| Option | Description | Selected |
|--------|-------------|----------|
| Keep WS as default; h2 opt-in via env var (Recommended) | run-bank.sh doesn't set the new env var | ✓ |
| Make h2 the default | run-bank.sh sets MCP_SERVER_HTTP2_URL=http://localhost:8083 |  |
| Default depends on Node version | Conditional in run-bank.sh — Node 22→h2, Node 24→WS |  |

**User's choice:** Keep WS as default; h2 opt-in via env var
**Notes:** Zero-regression default. Audiences see WS in Token Chain like today. Conference presenters who want to demo h2 set the env var explicitly. This is consistent with the broader "v1.0 at 97%, don't add demo-blocker risk" stance from the discussion that led to this backlog item.

## Node Version Pin + Crash Hedging

| Option | Description | Selected |
|--------|-------------|----------|
| Verify on current Node first; gate behind try/catch fallback if it reproduces (Recommended) | Empirical, scoped, no Node constraint | ✓ |
| Add Node version range check (>=22 <25) | Forces nvm; conflicts with CLAUDE.md "20/22/24 supported" |  |
| Switch to spdy or fastify-http2 library | Adds dep + review surface |  |
| Ship and let users hit it | No mitigation |  |

**User's choice:** Verify on current Node first; gate behind try/catch fallback if it reproduces
**Notes:** The Node 24.3 crash hit on the BFF was triggered by an outbound PingOne 401 cycling through the compat layer. The MCP server makes no equivalent outbound HTTPS call at startup, so the crash may not reproduce. Plan must include reproduction step before designing the fallback. Avoids the trap of premature mitigation.
