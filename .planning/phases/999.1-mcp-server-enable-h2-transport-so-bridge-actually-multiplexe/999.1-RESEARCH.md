## User Constraints (from CONTEXT.md)

### Locked Decisions

**Transport flavor**
- **D-01:** Use **h2c (plaintext HTTP/2)** via `http2.createServer({ allowHTTP1: true })`. Matches existing `http://localhost:8080` setup, no cert needed for MCP server, simplest local-dev path. BFF h2 bridge already supports h2c via `http2.connect(..., { allowHTTP1: true })`.

**WebSocket coexistence**
- **D-02:** **Split ports.** WebSocket transport stays on port 8080 (unchanged, no risk to existing WS clients). HTTP/2 endpoint runs on a new port. This avoids the unsolved question of whether `ws.Server` cleanly coexists with `http2.createServer` on the same port across Node versions.
- **D-03:** New port is **8083**. New env var **`MCP_SERVER_HTTP2_URL`** (e.g. `http://localhost:8083`). The existing `MCP_SERVER_URL` env var keeps its semantics (WebSocket transport default) and is untouched.

**Default transport**
- **D-04:** **WebSocket remains the default.** `run-bank.sh` does NOT set `MCP_SERVER_HTTP2_URL` by default. The h2 path is **opt-in** — conference presenters or developers who want to demo h2 set the env var explicitly. Zero risk to existing demos. The BFF logic at `server.js:1730` already branches on `mcpUrl.startsWith('http')` — extend it to prefer `MCP_SERVER_HTTP2_URL` when set, else fall back to `MCP_SERVER_URL` (WS).

**Node version + crash hedging**

## Summary

The bridge at `banking_api_server/services/http2McpBridge.js` is already a working h2 *client* — it correctly calls `http2.connect(connectUrl, { allowHTTP1: true })` against any `http://` upstream. The reason multiplexing is theoretical is purely server-side: `banking_mcp_server/src/server/BankingMCPServer.ts:108` builds the listener with `http.createServer((req, res) => this.handleHttpRequest(req, res))`, so Node negotiates DOWN to HTTP/1.1 even when the client asks for h2. The cleanest fix is to add a **second** listener — `http2.createServer({ allowHTTP1: true })` on port 8083 — that delegates all `/mcp`, `/.well-known/oauth-protected-resource`, `/audit` requests to the existing `HttpMCPTransport.handleRequest()`. Because `HttpMCPTransport` is built against the raw `IncomingMessage`/`ServerResponse` types (not Express's wrapped req/res), Node's http2 compat layer presents drop-in shapes — **no transport refactor is required**.

Three risks dominate the plan:

1. **Node 24 compat-layer instability** — there's a known `IncomingMessage._read` crash where `this.socket` is undefined under specific conditions (CITED: nodejs/node#46538, expressjs/express#5347). Empirically the crash has surfaced on this codebase's BFF when an outbound PingOne call cycled through the compat layer. The MCP server has no equivalent outbound HTTPS at startup, so the predicate suggests it won't reproduce — but D-05 mandates verification before shipping, with a try/catch fallback if it does.

2. **`ws` and `http2.createServer` on the same port** — would force us to handle the HTTP/1.1 Upgrade frame manually because `http2.createServer` does not emit `'upgrade'` the way `http.createServer` does. D-02 sidesteps this by using a second port; this research confirms that decision is sound.

3. **Educational surfacing scope creep** — D-06 specifies both a customer-visible Token Chain pill AND an admin inspector. These are two distinct UI surfaces touching three files (`TokenChainDisplay.js`, `tokenChainService.js`, a new or extended admin route). The planner should sequence these as separate tasks so the transport flip (AC-01..03) can ship and be observed via logs before the UI work lands.

**Primary recommendation:** Ship the h2c listener on 8083 as a small, fully additive change to `BankingMCPServer.startServer()` (one extra `http2.createServer(...).listen()` call gated by `MCP_HTTP2_PORT`), then extend the BFF h2 bridge to detect `session.alpnProtocol` and emit a `transport` field into the existing `tokenEvents` array. UI work follows the same pill component pattern already used for `StatusBadge` and `SpecRefPill` in `TokenChainDisplay.js`.

### Pattern 1: Dual-server colocation (one Node process, two listeners)

**What:** Run `http.createServer` (with `ws.Server` attached) on 8080 AND `http2.createServer({ allowHTTP1: true })` on 8083 from the same `BankingMCPServer.startServer()` method. Both delegate routing to the same `HttpMCPTransport.handleRequest()` and `MCPMessageHandler` instances.

**When to use:** When you need two transports speaking the same underlying protocol (here, MCP 2025-11-25) without forcing clients onto a new one. Matches the established repo pattern of env-gated optional listeners.

**Example (target shape, derived from current code):**
```typescript
// banking_mcp_server/src/server/BankingMCPServer.ts (extending startServer)
// [CITED: project code at BankingMCPServer.ts:108]

async startServer(): Promise<void> {
  // ... existing httpServer setup on this.config.port (8080) ...
  this.httpServer = createServer((req, res) => this.handleHttpRequest(req, res));
  this.server = new WebSocket.Server({ server: this.httpServer, ... });
  await new Promise<void>((resolve, reject) => {
    this.httpServer!.listen(this.config.port, this.config.host, ...);
  });

  // NEW: opt-in h2c listener (D-03)
  const h2Port = parseInt(process.env.MCP_HTTP2_PORT || '', 10);
  if (Number.isFinite(h2Port) && h2Port > 0) {
    try {
      const http2 = await import('http2');  // dynamic import = isolates the failure mode (D-05)
      this.http2Server = http2.createServer({ allowHTTP1: true }, (req, res) => {
        // Node http2 compat layer presents IncomingMessage/ServerResponse shape;
        // HttpMCPTransport.handleRequest is already typed against these from 'http'.
        this.handleHttpRequest(req as any, res as any);
      });
      this.http2Server.on('error', (e) => console.error('[BankingMCPServer] h2 error:', e));
      await new Promise<void>((resolve, reject) => {
        this.http2Server.listen(h2Port, this.config.host, (err?: Error) => err ? reject(err) : resolve());
      });
      console.log(`[BankingMCPServer] HTTP/2 (h2c) listener enabled on :${h2Port}`);
    } catch (err) {
      // D-05 fallback: do NOT crash the demo. Log loudly, leave WS on 8080 working.
      console.warn(`[BankingMCPServer] ⚠️ Failed to start h2 listener on :${h2Port}: ${(err as Error).message}`);
      console.warn('[BankingMCPServer] Continuing with WebSocket transport only (h2 educational pill will not surface).');
    }
  }
}
```

### Pattern 2: Negotiated-protocol detection on the client side

**What:** After `http2.connect()` resolves, inspect `session.alpnProtocol` (string `'h2'` for HTTP/2, `'http/1.1'` for h1 fallback). For h2c without ALPN, the property may be the empty string or `'h2c'`; treat anything that isn't `'http/1.1'` as h2 for the pill label.

**When to use:** When you need to label the actual negotiated transport, not the configured intent. D-06 mandates this.

**Example (target shape; current `createHttp2Session` does not yet emit this):**
```javascript
// banking_api_server/services/http2McpBridge.js — extension to createHttp2Session
// [CITED: project code at http2McpBridge.js:65]

const session = http2.connect(connectUrl, options);

let negotiated = null; // 'h2' | 'http/1.1' | null (still negotiating)
session.on('connect', () => {
  // session.alpnProtocol is set on 'connect' (HTTP/2 secure) OR on the underlying socket for h2c.
  // For h2c with allowHTTP1: true: alpnProtocol is '' but the session.type === http2.constants.NGHTTP2_SESSION_CLIENT
  // and pings will work. Heuristic: ask the server what it negotiated.
  negotiated = session.alpnProtocol || 'h2c';
  // Push event for inspector + token chain pill
  appEventService.logEvent('mcp', 'info', `[http2McpBridge] alpnProtocol=${negotiated}`, {
    tag: 'mcp/transport',
    metadata: { url: mcpServerUrl, transport: negotiated },
  });
});
```

**Important caveat — h2c and alpnProtocol:** ALPN is a TLS-layer mechanism. h2c (cleartext h2) does NOT use ALPN; `session.alpnProtocol` is typically the empty string for h2c sessions. The reliable test is **whether the response arrived via an h2 stream at all** — the bridge's existing `session.request(reqHeaders)` returning a `ClientHttp2Stream` proves h2 succeeded. The simplest distinguishing fact: if the server is a `http.createServer`, `session.request()` rejects with `ERR_HTTP2_GOAWAY_SESSION` immediately. With a real `http2.createServer`, streams succeed. **Recommended:** detect by capturing the first stream's response headers — if `:status` arrives via the h2 stream pseudo-header API (`responseHeaders[':status']`, not `responseHeaders['statusCode']`), it's h2. The current bridge already uses `responseHeaders[':status']` (line 226), so detection is implicit; we just need to record the result.

### Pattern 3: Env-var-gated optional listener (project convention)

**What:** Mirror the existing `MCP_GATEWAY_HTTP_URL` pattern — when env var is set, the optional path is active; when unset, default behavior is unchanged.

**When to use:** Demo features that are opt-in and add risk to existing flows. D-04 maps directly to this pattern.

**Example (BFF side, target shape):**
```javascript

const gatewayHttpUrl = mcpGatewayClient.getMcpGatewayHttpUrl();
```

### Anti-Patterns to Avoid

- **Replacing the WS server with the h2 server.** D-02 explicitly forbids this. Demos and external WS clients (Inspector, LangChain agent) depend on `ws://localhost:8080`.
- **Mounting `Express` onto `http2.createServer`.** [CITED: expressjs/express#5347, expressjs/express discussions/6583] — Express + http2 compat layer is the documented source of the `IncomingMessage._read` crash. `HttpMCPTransport` does NOT use Express (it operates on raw `IncomingMessage`/`ServerResponse`), and this is **load-bearing** — it sidesteps the crash class entirely.
- **Using `http2.createSecureServer` for local dev.** Would require cert plumbing for an http-only loopback service. D-01 defers TLS.
- **Probing transport with `try { session.request() } catch`.** Slow, racy. Use `session.alpnProtocol` snapshot in the `'connect'` event plus the existing `:status` pseudo-header check.
- **Exposing bearer tokens in the admin h2 inspector.** Pool keys already use a 16-char token *prefix* (`http2McpBridge.js:30`), and the inspector should mirror that — never the full token.

## Common Pitfalls

### Pitfall 1: Node 24 http2 compat-layer crash (`IncomingMessage._read` socket undefined)

**What goes wrong:** Under Node 22.x / 23.11 / 24.2+ (including the local 24.3.0 measured by `node --version`), code paths that go through the `http2` compat layer can crash with:
```
TypeError: Cannot read properties of undefined (reading 'readable')
  at IncomingMessage._read (node:_http_incoming:211:19)
  at Readable.read (node:internal/streams/readable:739:12)
  at resume_ (node:internal/streams/readable:1257:12)
```

**Why it happens:** The compat layer's `IncomingMessage._read` accesses `this.socket.readable`, but `this.socket` is `undefined` in certain teardown / re-entry paths. The publicly documented trigger is **Express on http2** (CITED: expressjs/express#5347, expressjs/express#6580). The Express + http2 binding via `http2-express-bridge` lost its hook when Express 5 dropped `application.lazyrouter` (CITED: expressjs/express discussions/6583). Other reproducible triggers have been observed in this codebase's BFF when an outbound PingOne call cycled through the compat layer.

**How to avoid:**
1. **Don't introduce Express into the h2 server path.** `HttpMCPTransport.handleRequest()` works on raw `IncomingMessage`/`ServerResponse` shapes — keep it that way. [VERIFIED: read of `HttpMCPTransport.ts:170-233`]
2. **Don't make outbound HTTPS calls from within the h2 request handler.** The MCP server's `/mcp` POST handler doesn't make outbound HTTPS today — confirmed by reading `HttpMCPTransport.handlePost()`. PingOne calls happen via `authManager.validateAgentToken()` which goes through the session manager, not a fresh HTTPS request per call. The crash is *less likely* to reproduce here than it did on the BFF.
3. **Wrap the h2 listener startup in try/catch with a fallback.** Per D-05: if `http2.createServer(...).listen()` throws or if any request handler within 30 seconds of startup throws an unhandled `IncomingMessage._read` TypeError, log a warning and disable the h2 listener for that process. WS on 8080 continues to serve. (See Pattern 1 above for the exact try/catch shape.)
4. **For D-05 verification:** the plan must include an explicit smoke test that fires ≥10 sequential `tools/call` requests via the h2 bridge with `MCP_SERVER_HTTP2_URL=http://localhost:8083` set, and asserts no `_http_incoming:211` line appears in the MCP server log. If it appears, fallback is engaged and the phase ships with the warning rather than ignoring it.

**Warning signs:**
- A `_http_incoming:211` stack trace in `/tmp/bank-mcp.log` after the h2 listener starts.
- The h2 server accepts connections but every `POST /mcp` returns `ERR_HTTP2_STREAM_ERROR` after the first request.
- The BFF h2 bridge's `session.alpnProtocol` is reported but the pool entry's `pendingStreams` never decrements.

[CITED: https://github.com/nodejs/node/issues/46538 — long-running issue, not closed-with-fix in any release we could verify; status as of 2026-05-14 is "needs more info / closed as not planned" per WebFetch attempt]
[CITED: https://github.com/expressjs/express/issues/5347 — h2 insecure server crash]

### Pitfall 2: `ws` and `http2.createServer` on the same port

**What goes wrong:** Attaching a `WebSocket.Server` (with `server: http2Server`) to an `http2.createServer` instance silently fails — `ws.Server` listens for the HTTP/1.1 `'upgrade'` event, but `http2.createServer` emits `'unknownProtocol'`/`'streamError'` for non-h2 framing. WS clients fail to connect.

**Why it happens:** `ws@8` was designed against `http.Server`. There's no first-class adapter for h2.

**How to avoid:** D-02 already locks the answer: **use two separate listeners, two ports.** This research confirms that decision — there's no clean cross-version-stable way to share the port. Don't try.

**Warning signs:** `ws://localhost:8083` returns ECONNRESET immediately on handshake.

### Pitfall 3: Port-sweep arrays in `run-bank.sh` missing 8083

**What goes wrong:** `run-bank.sh` has TWO port-sweep arrays (lines 320 and 332) used by `start`/`stop`/`status` to clean stale processes. They list every project port (3001 4000 8080 8888 8889 8890 3005 3006 3009 8081 8082). If 8083 is omitted, `run-bank.sh stop` leaves a hung h2 server, and the next `run-bank.sh` fails on EADDRINUSE.

**How to avoid:** When the plan touches `run-bank.sh` (it must, even minimally), update **both** arrays AND the comment at line 412 (`echo "   Sweeping ports..."`) to include 8083. Also add a one-line documentation comment near line 753 (where MCP Invest is started on :8081) noting that 8083 is an optional h2 listener side-by-side with 8080.

**Warning signs:** `EADDRINUSE :8083` on a second `run-bank.sh start` after stopping the demo.

### Pitfall 4: BFF h2 bridge falls back to h1 silently, no pill change

**What goes wrong:** If the MCP server's h2 listener fails to start (Pitfall 1) but the BFF still has `MCP_SERVER_HTTP2_URL=http://localhost:8083` set, `http2.connect('http://localhost:8083', { allowHTTP1: true })` will negotiate down to h1 — the *current* behavior the phase is trying to eliminate. Without an explicit "transport actually used" emit, the Token Chain pill will incorrectly show "HTTP/2" while the wire protocol is h1.

**How to avoid:**
1. The pill label MUST come from the bridge's runtime observation (`session.alpnProtocol` or `:status` pseudo-header presence), NOT from "the env var is set so it must be h2."
2. The admin inspector MUST log fallback events explicitly. Suggested: emit `appEventService.logEvent('mcp', 'warning', 'h2 negotiation fell back to h1', { tag: 'mcp/transport', metadata: {...} })` on detection.
3. The Token Chain pill SHOULD use three distinct labels: `HTTP/2` (h2c success), `HTTP/1.1` (configured h2 but negotiated down), and `WebSocket` (default path) — D-06 already names all three.

**Warning signs:** Token Chain pill says "HTTP/2" but the bridge is opening one stream per request (h1 has no multiplexing — the pool's `pendingStreams` count stays low, but bridge throughput is identical to the pre-phase baseline).

### Pitfall 5: `HttpMCPTransport.acceptHeaderIsValid` rejects browser-style `*/*` Accept

**What goes wrong:** The MCP server's POST /mcp handler is strict about Accept headers (line 783–788 of `HttpMCPTransport.ts`): it requires both `application/json` AND `text/event-stream`, OR `*/*`. The BFF h2 bridge currently sends `Accept: application/json` (h2McpBridge.js:133). Switching to h2 doesn't change the bridge's headers — but if anyone manually `curl`s the h2 endpoint without `--header "Accept: application/json, text/event-stream"`, they'll get a 400 and conclude h2 is broken.

**How to avoid:** Document the verification command precisely. See "Verification commands for plan acceptance criteria" below — `curl -v --http2-prior-knowledge -H 'Accept: application/json, text/event-stream' -H 'Content-Type: application/json' http://localhost:8083/mcp -d '...'`.

**Warning signs:** `curl` returns 400 with `Accept header MUST include both "application/json" and "text/event-stream"` — server is responding (h2 negotiation worked!) but the request shape is wrong.

### Pitfall 6: Existing bridge docstring is stale after this phase

**What goes wrong:** The lead comment of `http2McpBridge.js` (lines 5–11) currently says the bridge "negotiates DOWN to HTTP/1.1." This was correct as of commit 42082c76 — but it's load-bearing documentation. If the docstring isn't updated when h2 actually negotiates up, future readers will be misled.

**How to avoid:** Plan task list must include "update lead docstring in `http2McpBridge.js` to reflect post-phase reality." Suggested replacement language is in CONTEXT.md `<specifics>` section.

### Example 2: Detecting negotiated protocol on the BFF h2 bridge

```javascript
// banking_api_server/services/http2McpBridge.js
// Replace lines 65-77 (createHttp2Session bottom half)
// [CITED: project code at http2McpBridge.js:41-78]

const session = http2.connect(connectUrl, options);

// Phase 999.1 (D-06): record the negotiated protocol for the Token Chain pill
// and admin inspector. session.alpnProtocol is reliable for h2 over TLS; for
// h2c (no ALPN) we infer from the absence of failure on the first stream.
let negotiatedTransport = 'pending';
session.once('connect', () => {
  // For h2 over TLS, alpnProtocol is 'h2'. For h2c (no ALPN), alpnProtocol is
  // the empty string but session.type === NGHTTP2_SESSION_CLIENT and streams
  // will open. We label h2c as 'h2c' so the inspector can distinguish.
  if (session.alpnProtocol === 'h2') {
    negotiatedTransport = 'h2';
  } else if (parsed.protocol === 'http:' && !session.destroyed) {
    // h2c success path — confirmed by the bridge's :status pseudo-header
    // arriving via http2 stream API at h2Post() line 226.
    negotiatedTransport = 'h2c';
  } else {
    negotiatedTransport = 'http/1.1';
  }
});
session.once('error', () => { negotiatedTransport = 'error'; });

session.on('error', (err) => {
  console.error(`[http2McpBridge] Session error for ${parsed.host}: ${err.message}`);
  pool.delete(key);
});

session.on('close', () => {
  pool.delete(key);
});

// Extend the pool entry so the admin inspector can read it:
pool.set(key, {
  session,
  lastUsed: Date.now(),
  pendingStreams: 0,
  // NEW for Phase 999.1:
  createdAt: Date.now(),
  connectUrl,
  get negotiatedTransport() { return negotiatedTransport; },
});
return session;
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `http.createServer` with manual HTTP/1.1 chunked transfer | `http2.createServer({ allowHTTP1: true })` — h2 with h1 fallback in one server | Node 8.4 (2017) onwards | The exact server pattern this phase adopts |
| ALPN-only h2 detection | h2c (no TLS) via `--http2-prior-knowledge` in curl, no ALPN | Stable since Node 10 | Aligns with D-01 (cleartext, no cert needed) |
| `http2-express-bridge` for Express compat | Avoid Express on h2 entirely; use raw `IncomingMessage`/`ServerResponse` | Express 5 (2024) broke the bridge | `HttpMCPTransport.handleRequest()` is already raw — fortunate prior decision |
| WebSocket-only MCP transport | MCP 2025-11-25 Streamable HTTP transport | spec 2025-11-25 | `HttpMCPTransport.ts` is already shipped (Phase D). h2 is just transport-layer optimization beneath it |

**Deprecated/outdated:**
- `node:http2-express-bridge` — abandoned, broken on Express 5. [CITED: expressjs/express discussions/6583]
- `http2.createSecureServer` without `allowHTTP1` — fine in production, but for a demo with HTTP-only loopback, h2c is simpler. [CITED: D-01]

### Known Threat Patterns for {Node h2c + JWT bearer + admin inspector}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Token exfiltration via admin inspector | Information Disclosure | The inspector returns pool keys (URL + 16-char token prefix), NOT full tokens. Verified by reading `http2McpBridge.js:28-30`. |
| Admin route un-authenticated | Elevation of Privilege | `app.use('/api/admin/h2', authenticateToken, ...)` — exact pattern of §1-protected admin mounts at server.js:899. Tests in Wave 0 assert 401 unauthenticated. |
| Cleartext bearer over loopback h2c | Information Disclosure (local) | h2c is loopback-only (8083 bound to 127.0.0.1 in dev). Same threat surface as the existing WS on 8080. No regression. |
| h2 stream amplification (HTTP/2 Rapid Reset, CVE-2023-44487) | Denial of Service | Node patched in 20.x and forward. [CITED: Node release notes, broadly known]. Our request volume is demo-scale (≤1 request/sec), not exploitable. |
| Connection-pool key collision (two users get the same h2 session) | Information Disclosure | Pool key is `{url}::{first-16-chars-of-token}` — two users have different tokens, different keys. Verified at `http2McpBridge.js:30-32`. The 16-char prefix is high entropy for distinct PingOne JWTs (post-header substring varies per token). |
| Node 24 http2 compat-layer crash (Pitfall 1) | Denial of Service | D-05 try/catch fallback ensures MCP server stays up on WS even if h2 listener crashes. |
