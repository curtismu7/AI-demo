# Phase 999.1: MCP server — enable h2 transport so http2McpBridge actually multiplexes - Context

**Gathered:** 2026-05-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Make `banking_api_server/services/http2McpBridge.js` actually negotiate HTTP/2 to the MCP server. Today the bridge calls `http2.connect(..., { allowHTTP1: true })` against `banking_mcp_server`, which uses plain `http.createServer` — so Node silently downgrades to HTTP/1.1 and the "multiplexing" claim in the bridge docstring is theoretical.

**In scope:**
- Enable HTTP/2 (h2c) on the banking_mcp_server.
- Provide a way for the BFF h2 bridge to actually exercise the h2 path against it.
- Add educational surfacing so demo audiences can see when h2 is in use.
- Verify the Node 24 http2-compat crash hit on the BFF is or isn't reproducible on the MCP server, and hedge accordingly.

**Out of scope:**
- Changing transport on banking_mcp_gateway (audit concluded h2 is the wrong fit there — WebSocket upgrade and synchronous POST /mcp dominate the traffic).
- Changing transport on banking_api_server (BFF) — Node 24.3 http2-compat regression observed there, deferred.
- Retiring WebSocket transport from the MCP server (it stays as the default).

</domain>

<decisions>
## Implementation Decisions

### Transport flavor
- **D-01:** Use **h2c (plaintext HTTP/2)** via `http2.createServer({ allowHTTP1: true })`. Matches existing `http://localhost:8080` setup, no cert needed for MCP server, simplest local-dev path. BFF h2 bridge already supports h2c via `http2.connect(..., { allowHTTP1: true })`.

### WebSocket coexistence
- **D-02:** **Split ports.** WebSocket transport stays on port 8080 (unchanged, no risk to existing WS clients). HTTP/2 endpoint runs on a new port. This avoids the unsolved question of whether `ws.Server` cleanly coexists with `http2.createServer` on the same port across Node versions.
- **D-03:** New port is **8083**. New env var **`MCP_SERVER_HTTP2_URL`** (e.g. `http://localhost:8083`). The existing `MCP_SERVER_URL` env var keeps its semantics (WebSocket transport default) and is untouched.

### Default transport
- **D-04:** **WebSocket remains the default.** `run-bank.sh` does NOT set `MCP_SERVER_HTTP2_URL` by default. The h2 path is **opt-in** — conference presenters or developers who want to demo h2 set the env var explicitly. Zero risk to existing demos. The BFF logic at `server.js:1730` already branches on `mcpUrl.startsWith('http')` — extend it to prefer `MCP_SERVER_HTTP2_URL` when set, else fall back to `MCP_SERVER_URL` (WS).

### Node version + crash hedging
- **D-05:** **Empirical verification first, fallback if needed.** Plan must include:
  1. Reproduce or rule out the Node 24.3 http2-compat crash (`TypeError: Cannot read properties of undefined (reading 'readable') at IncomingMessage._read (node:_http_incoming:211:19)`) against the MCP server. The crash hit on the BFF was triggered by an outbound PingOne call cycling through the compat layer. The MCP server does no equivalent outbound HTTPS call at startup, so the crash may not reproduce.
  2. If it reproduces: wrap `http2.createServer` startup in try/catch with explicit fallback to `http.createServer` + log a warning that h2 is disabled on this Node version. Demo doesn't break; h2 educational value lost gracefully.
  3. If it doesn't reproduce: ship as-is. No `engines.node` constraint added (CLAUDE.md currently says Node 20/22/24 all supported — keep that contract).

### Educational surfacing
- **D-06:** **Both Token Chain pill and admin inspector.**
  - Token Chain panel: small pill/badge on the BFF→MCP step shows `HTTP/2` / `WS` / `HTTP/1.1` so presenters and audiences see which transport ran.
  - Admin inspector (dev panel): show actual h2 stream IDs from the bridge pool, connection reuse count, and h1 fallback events. This is the "look under the hood" surface for technical audiences.
  - Touches `banking_api_ui/src/components/TokenChainDisplay.js`, `banking_api_server/services/tokenChainService.js`, and admin route(s). Planner must scope this carefully — it's UI + service work, not just transport.

### Claude's Discretion
- Exact label text for the transport pill ("HTTP/2", "h2", "HTTP/2 mux" — planner picks).
- Where in `/admin` the inspector lives (new tab, sub-section of an existing admin page).
- How to detect h1 fallback from the bridge — likely log line or `session.alpnProtocol`. Implementation choice.
- Whether to expose h2 stream IDs in the customer-visible Token Chain or only in admin. Default: admin only.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`HttpMCPTransport`** (`banking_mcp_server/src/server/HttpMCPTransport.ts`) — spec-compliant POST /mcp and GET /mcp (SSE) handlers. Currently wired into the same `http.createServer` instance via `handleHttpRequest`. **Reuse unchanged** for the h2 path — the handler reads/writes via `IncomingMessage`/`ServerResponse`, which Node's http2 compat layer also provides.
- **`http2McpBridge`** client pool — already supports h2c via `allowHTTP1: true`. Will automatically negotiate h2 once the server speaks it. No client-side code change required for the basic transport flip.
- **`appEventService`** (`banking_api_server/services/appEventService.js`) — emits structured events with categories. Add a new category or a tag (`mcp/transport`) to log "h2 stream opened", "h1 fallback", "session evicted" — that data flows into the admin inspector.

### Established Patterns
- **Env-var-gated optional features**: `MCP_GATEWAY_HTTP_URL` already follows this pattern at `server.js:1727` — when set, the BFF routes via gateway; when unset, falls back. `MCP_SERVER_HTTP2_URL` mirrors that contract exactly.
- **Token Chain "pill" UI**: `banking_api_ui/src/components/TokenChainDisplay.js` already renders pills for `act` claim status, `aud` validation, etc. The transport pill is one more in the same visual family — planner should reuse the existing pill component.

### Integration Points
- BFF h2 path decision at `server.js:1730` (`useHttp2 = !useGateway && (mcpUrl.startsWith("http") ...)`) — extend to read `MCP_SERVER_HTTP2_URL` first.
- MCP server startup at `BankingMCPServer.ts:108` — replace `createServer` import + call with conditional `http2.createServer` based on `MCP_HTTP2_PORT` env var; if env var set, listen on a second server on that port (don't replace the WS-bearing server).
- `run-bank.sh` — add an optional comment-out line for `MCP_SERVER_HTTP2_URL` to make h2 discoverable.

</code_context>

<deferred>
## Deferred Ideas

- **TLS variant (h2 over TLS, not h2c).** Conscious deferral. Once BFF↔MCP runs over h2c successfully, a follow-up phase can add `http2.createSecureServer` with mkcert and flip the default to TLS to match realistic production. Currently out of scope per D-01.
- **Retire WebSocket from the MCP server.** Conscious deferral. Doable once h2 has been demoed at conferences and there are no remaining WS-only consumers. Would let `BankingMCPServer.ts` drop the `WebSocket.Server` import entirely.
- **Apply same h2 transport pattern to MCP Invest (8081) and gateway upstream paths.** Conscious deferral. The gateway audit in this session already concluded h2 is not the right fit at the gateway layer; revisit if BFF→gateway switches from WS to HTTP.
- **PostHog event for transport-path selection** — emit `mcp_transport_path = h2 | h1 | ws` on every BFF→MCP call so we can measure adoption in the demo deployment. Probably a tiny follow-on; planner can fold it into D-06 if cheap.
