### ID Token Exchange (Phase 186)

#### When to Use

Use ID token in dual-token exchange when:
- User identity assertion is more important than capability grants
- You want to demonstrate identity-based delegation (vs. capability-based via access token)
- MCP Gateway needs to know the user's identity directly from the subject claim
- Tokens expire frequently and you want to refresh identity independently from capability

**Compare to Phase 184 (Access Token):**
| Aspect | Access Token (Phase 184) | ID Token (Phase 186) |
|--------|-------------------------|----------------------|
| **Purpose** | Grants access to resource APIs | Asserts user identity |
| **Claims** | `scope`, `aud`, `resource` | `sub` (user ID), `aud`, `iss` |
| **Lifetime** | Longer (1 hour typical) | Shorter (15-30 min typical) |
| **Refresh** | Via refresh_token grant | Via new auth/token request |
| **Use Case** | Continuous resource access | One-time identity assertion |

#### Flow Diagram

\`\`\`
Agent       App BFF         PingOne         MCP Gateway
  |           |               |               |
  |--- MCP request -------->  |               |
  |                           | (MCP 401)     |
  |<--- 401 Unauthorized ---- |               |
  |                           |               |
  |                           |--- OIDC auth code exchange
  |                           | (with PKCE)
  |                           |--- Returns: ID token + access token + refresh token
  |                           |
  |                           |--- Token Exchange (RFC 8693)
  |                           |    subject_token = ID token
  |                           |    actor_token = Agent CC token
  |                           |    audience = MCP resource
  |                           |
  |                           |--- Returns: MCP token
  |                           |    (claims: sub=user, act=agent)
  |                           |
  |--- MCP request (retry) --->                ----->|
  |<--- MCP response --------                <-----|
  |
\`\`\`

#### Code Example

**Backend (Node.js + express):**

\`\`\`javascript
// Step 1: Catch 401 from MCP
const response = await fetch('https://mcp-gateway/tool/call', {
  headers: { Authorization: \`Bearer \${mcpToken}\` }
});

if (response.status === 401) {
  // Step 2: Redirect user to OIDC auth
  const authUrl = oauthService.getAuthIdTokenRedirectUrl(
    returnTo: '/api/mcp/retry',  // Return here after auth
    scope: 'openid profile email'  // Get ID token
  );
  // Frontend redirects to authUrl
}

// Step 3: After auth callback, exchange tokens
const { idToken, accessToken, refreshToken } = session.oauthTokens;
const { mcpToken } = await oauthService.performTokenExchangeWithIdToken(
  idToken,                    // Subject: User identity
  MCP_SERVER_AUDIENCE,        // Audience: MCP resource
  ['banking:read', 'banking:write']  // Scopes
);

// Store agent CC token for actor claim
const agentCCToken = session.agentToken;

// Exchange with actor claim for delegation
const result = await oauthService.performTokenExchangeWithActor(
  idToken,                    // subject_token (user identity)
  agentCCToken,               // actor_token (agent CC)
  MCP_SERVER_AUDIENCE,        // audience
  ['banking:read', 'banking:write']  // scopes
);
// result.mcpToken now has claims: { sub: 'user-id', act: 'agent-id', aud: 'mcp-gateway' }

// Step 4: Retry original MCP request
const retryResponse = await fetch('https://mcp-gateway/tool/call', {
  headers: { Authorization: \`Bearer \${result.mcpToken}\` }
});
\`\`\`

#### Security Considerations

1. **ID Token Lifetime**: ID tokens are typically short-lived (15-30 min). Each 401 incident requires new auth.
2. **Token Leak**: ID tokens stored in httpOnly session cookie only; never exposed to JavaScript frontend.
3. **Signature Validation**: PingOne validates ID token signature at exchange time; validate at token endpoint recipient.
4. **Scope Limiting**: ID token carries claims; ensure scopes in auth request are minimal (just \`openid profile email\`).

\`\`\`markdown
### Phase 186: ID Token Exchange (401 → Auth → Exchange)

**Flow:**
1. Agent calls MCP server
2. MCP returns 401 (unauthenticated)
3. App catches 401 and initiates OIDC user auth
4. User redirects to PingOne, authenticates
5. PingOne returns auth code; app exchanges code for ID token + access + refresh
6. App exchanges agent CC token + user ID token at PingOne for MCP token
7. App retries original agent call with MCP token
8. Result returned to agent

**When to use:**
- When user identity assertion is critical (vs. capability-based access)
- When MCP Gateway needs to verify user directly from subject claim
- When demonstrating identity delegation patterns (educational value)

**Parallel to:**
- Phase 184 (Dual-Token Exchange with Access Token) — same flow, but uses access token instead of ID token

**See also:** [Detailed documentation](./PINGONE_TWO_TOKEN_EXCHANGES.md#id-token-exchange-phase-186)
