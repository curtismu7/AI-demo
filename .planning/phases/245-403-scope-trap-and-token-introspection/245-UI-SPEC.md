## Copywriting Contract

### Scope upgrade — automatic path (new behavior in Phase 245)

The agent now silently upgrades the token and retries. The user ONLY sees a message if:
(a) the upgrade succeeded and the tool result is returned, or (b) the upgrade failed.

| Element | Copy |
|---------|------|
| Upgrade in-progress (toast, loading) | `Upgrading token scope — retrying tool call…` |
| Upgrade succeeded (inline chat message) | `Scope upgraded automatically (RFC 8693) — re-ran \`{toolName}\` with write access.` |
| Upgrade failed — still 403 (inline chat message, error) | `Scope upgrade failed: your token does not have \`{missingScope}\`. Approve manually below to request write access.` |
| Upgrade not attempted — non-scope 403 (inline chat message) | `Access denied: \`{toolName}\` returned 403 (not a scope error). Check delegation configuration.` |

### Existing scope modal (unchanged — keep current copy)

The existing `scopeErrorModal` text in BankingAgent.js lines 5090–5140 is NOT changed.
It activates only when the automatic upgrade also fails (SCOPE-02 path).

### Token introspection consolidation — no user-visible copy changes

INTROSPECT-01 and INTROSPECT-02 are backend-only. No new copy required.
The existing `ConfigTokenValidation.tsx` introspection UI is untouched.

### Empty states

| Scenario | Copy |
|----------|------|
| No tool call in progress | (no empty state — agent chat is always present) |

### Error states

| Scenario | Copy |
|----------|------|
| Token exchange API unreachable | `Could not reach the token exchange endpoint. Check BFF connectivity and retry.` |
| Exchange returned invalid token | `Scope upgrade exchange completed but returned an unusable token. Contact your administrator.` |
| Retry loop guard triggered (should never appear if code is correct) | `Internal error: scope upgrade attempted more than once. Please reload and try again.` |

### Destructive actions

None. This phase has no delete or revoke actions. The "Approve scope upgrade" CTA is non-destructive.

## Interaction Contracts

### 1. Automatic scope upgrade flow (SCOPE-01)

```
User sends message → agent dispatches MCP tool → BFF returns mcp_scope_denied
  ↓
BankingAgent.js intercepts BEFORE setting scopeErrorModal
  ↓
Show toast: "Upgrading token scope — retrying tool call…" (loading, indeterminate)
  ↓
BFF performs RFC 8693 exchange → returns upgraded token
  ↓
Retry tool call with upgraded token
  ↓
SUCCESS: dismiss toast, append inline chat message (see copy above)
FAIL:    dismiss toast, show scopeErrorModal (existing component, unchanged)
```

State used: add `_scopeUpgradeInProgress: boolean` to local scope (not React state — synchronous flag
inside `callMcpTool()` on the BFF side; UI sees the result, not the intermediate state).

### 2. Tool-chip status during upgrade

While the automatic upgrade is running, the existing tool-chip for the affected tool must show
`status: "running"` (already handled by `.ba-tool-chip-status--running` CSS). No new chip states needed.

After success, chip shows `status: "success"`. After failure, chip shows `status: "error"`.

### 3. No new modals

Phase 245 does NOT add new modal components. The existing `scopeErrorModal` handles the manual
fallback path (SCOPE-02). Adding a new modal would duplicate UX already defined in Phase 211.

### 4. Introspection consolidation — zero UI changes

INTROSPECT-01 and INTROSPECT-02 are pure backend consolidation. No UI components are added,
modified, or removed. `ConfigTokenValidation.tsx` remains as-is.
