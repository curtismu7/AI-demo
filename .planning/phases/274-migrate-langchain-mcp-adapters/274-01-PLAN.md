LangChain-AI. The MCP server already exposes an HTTP Streamable endpoint at
POST /mcp (port 8080) which maps directly to the library's `streamable_http` transport.

Specific changes:

1. ADD import at top (after existing langchain imports):
   `from langchain_mcp_adapters.client import MultiServerMCPClient`

2. Keep the existing import line:
   `from .mcp_tool_provider import MCPToolProvider, build_auth_popup_message`
   Change to:
   `from .mcp_tool_provider import build_auth_popup_message, _current_tracer`
   (MCPToolProvider is no longer needed; _current_tracer ContextVar is preserved in stub)

3. In `__init__`, remove:
   `self.mcp_tool_provider = MCPToolProvider(mcp_client_manager, auth_manager, self.conversation_memory)`
   
   Add instead a helper to build the MultiServerMCPClient config from registered servers:
   ```
   self._mcp_client_manager = mcp_client_manager  # kept for _session_challenges dict
   self._auth_manager = auth_manager
   ```

4. Replace `initialize_tools()`:
   - Build `server_configs` dict from `self._mcp_client_manager._server_configs`
   - For each server config, derive the HTTP URL: if endpoint starts with `ws://` or `wss://`,
     replace the scheme with `http://` or `https://` respectively. The MCP server listens
     on the same port for both WebSocket (legacy) and Streamable HTTP (`POST /mcp`).
   - Obtain agent token via `self._auth_manager.get_client_credentials_token(additional_scopes=["ai_agent"])`
   - Construct the MultiServerMCPClient config dict with:
     `"transport": "streamable_http"`, `"url": f"{http_url}/mcp"`,
     `"headers": {"Authorization": f"Bearer {token.token}"}`
   - Open the client as an async context manager, call `await client.get_tools()`,
     store tools in `self._tools`. Close the client after getting tools (the adapter
     caches the schema; tools themselves are stateless BaseTool wrappers).
   - If no server configs exist, log warning and set `self._agent_executor = self._create_basic_agent()`.

5. Replace `set_session_context` calls in `process_message_with_tracing` and
   `process_message`:
   - Remove `await self.mcp_tool_provider.set_session_context(session_id)` calls.
   - Instead, before calling `_get_agent_executor_for_session`, refresh tools with the
     current session's agent token by re-running the MultiServerMCPClient context manager
     with updated auth headers. Store refreshed tools in `self._tools`.
   - This replaces the per-session token injection that MCPTool._current_agent_token was
     doing — the library handles auth at connection time via headers.

6. Replace `self.mcp_tool_provider.set_tracer(tracer)` calls:
   - Use `_current_tracer.set(tracer)` directly (the ContextVar is now imported from stub).

7. Replace `self.mcp_tool_provider.retry_pending_tool_call(session_id, tracer)`:
   - Move the retry logic inline (it references `self._mcp_client_manager._session_challenges`
     and calls `self._mcp_client_manager.execute_tool` directly). The pending challenge dict
     is on MCPClientManager which is preserved.
   - The retry path must obtain a fresh agent token, build a MultiServerMCPClient with that
     token, and call the tool directly via `client.get_tools()` then find and invoke the
     matching tool by name.

8. Replace `self.mcp_tool_provider.mcp_client_manager._session_challenges` references:
   - Change to `self._mcp_client_manager._session_challenges` directly.

9. Replace `self.mcp_tool_provider.get_tool_info()` in `get_agent_status()`:
   - Return tool names from `[t.name for t in self._tools]` directly.

Key constraint: the `_session_challenges` dict on MCPClientManager must continue to
be populated when auth challenges are returned from tool execution. Since the library's
BaseTool._arun now runs the actual tool, the challenge detection happens when the tool
raises or returns an auth challenge string. The existing post-processing logic in

response must remain unchanged.

Per D-01 from todos/pending: keep the transition minimal — do not refactor unrelated
