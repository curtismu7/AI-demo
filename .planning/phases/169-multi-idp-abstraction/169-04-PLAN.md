Create service `roleClaimResolver.js` to abstract role extraction:

```javascript
const configStore = require('./configStore');

/**
 * Extract role from token claims based on configuration.
 * 
 * Handles:
 * - String claims: population_id = 'admin_uuid' → role = 'admin'
 * - Array claims: app_roles = ['admin', 'user'] → role = 'admin'
 * - URI claims: roles = ['https://example.com/roles/admin'] → match by suffix
 */
function getRoleFromClaims(claims) {
  const claimName = configStore.getEffective('oauth_role_claim_name') || 'population_id';
  const claimValue = claims[claimName];
  
  if (!claimValue) {
    // Fallback legacy PingOne population_id check (backward compat)
    if (claimName === 'population_id') {
      const adminPopId = configStore.getEffective('admin_population_id'); 
      if (adminPopId && claims.population_id === adminPopId) return 'admin';
    }
    return 'customer'; // default role
  }
  
  // Check if claim is array or string
  const isArray = ['true', '1', true].includes(configStore.getEffective('oauth_role_claim_is_array'));
  const adminValue = configStore.getEffective('oauth_role_claim_value_admin');
  const customerValue = configStore.getEffective('oauth_role_claim_value_customer');
  
  if (isArray) {
    // Array-based role (e.g., Azure AD app_roles, Auth0 roles)
    const roleArray = Array.isArray(claimValue) ? claimValue : [claimValue];
    
    // Check for exact matches or suffix matches (for URIs)
    for (const role of roleArray) {
      if (adminValue && (role === adminValue || role.endsWith('/' + adminValue))) return 'admin';
      if (customerValue && (role === customerValue || role.endsWith('/' + customerValue))) return 'customer';
    }
  } else {
    // String-based role (e.g., PingOne population_id, simple role string)
    if (adminValue && claimValue === adminValue) return 'admin';
    if (customerValue && claimValue === customerValue) return 'customer';
  }
  
  return 'customer'; // default
}

module.exports = { getRoleFromClaims };
```

Tests:

- Azure AD app_roles array match → 'admin'
- Auth0 roles array URI suffix match → 'admin'
- Missing claim → 'customer'
- Multiple roles in array → picks first admin match

# Role Claim Mapping Configuration

Configure how the BFF determines admin vs customer role based on token claims.

## Default (PingOne)

```bash
OAUTH_ROLE_CLAIM_NAME=population_id
OAUTH_ROLE_CLAIM_VALUE_ADMIN=d02d2305-f445-406d-82ee-7cdbf6eeabfd  # Admin population UUID
OAUTH_ROLE_CLAIM_IS_ARRAY=false
```

## Azure AD (Enroll as Array)

```bash
OAUTH_ROLE_CLAIM_NAME=app_roles
OAUTH_ROLE_CLAIM_VALUE_ADMIN=admin
OAUTH_ROLE_CLAIM_VALUE_CUSTOMER=user
OAUTH_ROLE_CLAIM_IS_ARRAY=true
```

Token example:
```json
{
  "app_roles": ["admin", "user", "viewer"],
  "oid": "user-uuid"
}
```

## Auth0 (Roles as URI)

```bash
OAUTH_ROLE_CLAIM_NAME=roles
OAUTH_ROLE_CLAIM_VALUE_ADMIN=admin
OAUTH_ROLE_CLAIM_VALUE_CUSTOMER=customer
OAUTH_ROLE_CLAIM_IS_ARRAY=true
```

Token example:
```json
{
  "roles": [
    "https://banking-demo.auth0.com/roles/admin",
    "https://banking-demo.auth0.com/roles/customer"
  ]
}
```

(Resolver matches by URI suffix, so 'admin' matches '.../roles/admin')

## Okta (Groups)

```bash
OAUTH_ROLE_CLAIM_NAME=groups
OAUTH_ROLE_CLAIM_VALUE_ADMIN=banking-admins
OAUTH_ROLE_CLAIM_VALUE_CUSTOMER=banking-users
OAUTH_ROLE_CLAIM_IS_ARRAY=true
```

Token example:
```json
{
  "groups": ["banking-admins", "all-users"],
  "sub": "user@example.com"
}
```

For custom IDPs, check your token claims and map accordingly:

```bash
OAUTH_ROLE_CLAIM_NAME=<claim-name>
OAUTH_ROLE_CLAIM_VALUE_ADMIN=<admin-value>
OAUTH_ROLE_CLAIM_VALUE_CUSTOMER=<customer-value>
OAUTH_ROLE_CLAIM_IS_ARRAY=<true|false>
```

Inspect token claims in browser DevTools or Config UI debug panel.

# Federate Migration Guide

Complete instructions for swapping from PingOne to PingFederate.

## Prerequisites

- PingFederate instance running (e.g., https://my-federate.example.com:9031)
- OAuth 2.0 server configured
- Two OAuth clients: one for admin, one for user

## Step 1: Create OAuth Clients in Federate

### Admin Client
1. Grant Types: Authorization Code, Refresh Token
2. Client Authentication: Client Secret Basic
3. Redirect URIs: `https://your-app.example.com/oauth2/callback`
4. Scopes: `openid profile email offline_access`
5. Note Client ID and Client Secret

### User Client
1. Same as admin, different Client ID/Secret
2. Redirect URI: `https://your-app.example.com/oauth2/callback`

## Step 2: Configure BFF Environment

```bash
# OAuth Endpoints (Federate uses fixed paths)
export OAUTH_ISSUER=https://my-federate.example.com:9031
export OAUTH_DISCOVERY_ENABLED=true
# (or explicit endpoints if discovery not available)
export OAUTH_AUTHORIZATION_ENDPOINT=https://my-federate.example.com:9031/oauth2/authorize
export OAUTH_TOKEN_ENDPOINT=https://my-federate.example.com:9031/oauth2/token
export OAUTH_USERINFO_ENDPOINT=https://my-federate.example.com:9031/oauth2/userinfo
export OAUTH_JWKS_URI=https://my-federate.example.com:9031/oauth2/JWKS

# Callback Paths
export OAUTH_ADMIN_CALLBACK_PATH=/oauth2/callback
export OAUTH_USER_CALLBACK_PATH=/oauth2/callback

# Admin OAuth Client
export PINGONE_ADMIN_CLIENT_ID=<admin-client-id>
export PINGONE_ADMIN_CLIENT_SECRET=<admin-client-secret>
export PINGONE_ADMIN_REDIRECT_URI=https://your-app.example.com/oauth2/callback

# User OAuth Client
export PINGONE_USER_CLIENT_ID=<user-client-id>
export PINGONE_USER_CLIENT_SECRET=<user-client-secret>
export PINGONE_USER_REDIRECT_URI=https://your-app.example.com/oauth2/callback

# Role Mapping (configure based on Federate claims)
export OAUTH_ROLE_CLAIM_NAME=groups  # or roles, custom_role, etc.
export OAUTH_ROLE_CLAIM_VALUE_ADMIN=federate-admins
export OAUTH_ROLE_CLAIM_VALUE_CUSTOMER=federate-users
export OAUTH_ROLE_CLAIM_IS_ARRAY=true

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Token claims → Role assignment | If role claim is spoofed/modified, user could get escalated role |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-169-04-01 | Tampering | Role claim in token | Mitigate | Token is JWT-signed; any claim modification invalidates signature; tokenValidationService must verify before roleResolver reads |
| T-169-04-02 | Spoofing | Claim value misidentification | Mitigate | Exact match required for role values; no fuzzy matching; case-sensitive |
| T-169-04-03 | Elevation of Privilege | Wrong role assigned | Mitigate | Only admin role grants access to /admin routes; even if role resolver is buggy, BFF must re-verify role at each protected route |
| T-169-04-04 | Information Disclosure | Role claim names in logs | Accept | Configuration is non-secret; OK to log (e.g., "[role-resolver] Using claim: population_id") |

Mitigations implemented:
- tokenValidationService signature check (existing, no change)
- Exact match + case-sensitive role values
- No wildcard or regex matching
- Role resolver is final arbiter; middleware must enforce perms per role
</threat_model>
