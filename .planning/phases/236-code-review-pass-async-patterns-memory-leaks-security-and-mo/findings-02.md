# Findings — Batch 02: server.js and Key Routes

## Part 1: server.js (~1639 lines)

| Severity | Category | File:Line | Issue | Fix Snippet |
|----------|----------|-----------|-------|-------------|
| **Major** | Async Patterns | server.js:980 | `POST /api/mcp/tool` is a ~555-line async handler with **no outer try/catch**. Lines 983–1080 (setup of `_appEvents`, `userAgent`, `emit`, `endFlowTraceOnce`, flowTrace guard) run before any `try` block. A synchronous throw here (e.g. from `mcpFlowSseHub.ensurePostTrace`) produces an **unhandled promise rejection** in Express 4. | Wrap the full handler body in one outer `try { … } catch (err) { next(err); }` |
| **Major** | Async Patterns | server.js:466–470 | `oauthService.revokeToken()` is called twice in `GET /api/auth/logout` **without `await` and without `.catch()`**. If the promise rejects, it fires an unhandledRejection. Comment says "non-fatal" but the rejection is never captured. | Add `.catch(err => console.warn('[logout] revoke error:', err.message))` to each call. |
| **Major** | Memory Leaks | server.js:889 / oauthMonitor.js:205 | `oauthMonitor.startPeriodicHealthCheck()` calls `setInterval()` but the **return handle is never stored**. No `SIGTERM`/`SIGINT` handler clears this interval. Timer leaks on any graceful shutdown. | Store: `this._intervalId = setInterval(…)`. Add `stop() { clearInterval(this._intervalId); }` and call on SIGTERM. |
| **Major** | Memory Leaks | server.js (whole file) | **No SIGTERM/graceful shutdown handler** exists in server.js. The HTTP/HTTPS server handle returned by `app.listen()` / `https.createServer().listen()` is never stored, so `server.close()` is never called on termination. In-flight requests are cut abruptly and the oauthMonitor interval leaks. | `const server = app.listen(…); process.on('SIGTERM', () => { server.close(); oauthMonitor.stop(); });` |
| **Major** | Async Patterns | server.js (whole file) | **No `unhandledRejection` or `uncaughtException` handler** is registered anywhere in the server entrypoint. In Express 4, async handlers that throw without try/catch leave dangling promises. Without this handler, Node silently swallows errors (pre-v15) or crashes with no structured log (v15+). | `process.on('unhandledRejection', (reason) => { console.error('[unhandledRejection]', reason); process.exit(1); });` |
| **Minor** | Security | server.js:156–191 | `helmet()` **PRESENT and well-configured**: HSTS 2yr, CSP, X-Frame-Options DENY, noSniff, referrer policy, hidePoweredBy. No defaults disabled. `unsafe-inline` in scriptSrc is acceptable given CRA requirement. CLEAN. | — |
| **Minor** | Security | server.js:205–212 | CORS **is not a wildcard `*` with credentials**. Production uses explicit `CORS_ORIGIN` env var or `false`. Development uses an explicit allowlist. CLEAN — no Critical wildcard+credentials issue. | — |
| **Minor** | Security | server.js:243–293 | Rate limiting applied globally and specifically to all four auth endpoints (login + callback for both admin and user flows). CLEAN. Generous demo defaults are intentional and documented. | — |
| **Minor** | Security | server.js:682–687 | `req.query.category`, `req.query.severity`, and `req.query.since` are passed to `appEventService.getEvents()` without allowlist validation. Low risk for an in-memory store, but worth hardening if the service ever evolves. | Validate `category` and `severity` against an allowlist array before passing. |
| **Minor** | Security | server.js:988–989 | `console.log` prints the full `req.sessionID` string. If logs are shipped to an external aggregator, a session ID in plaintext is a low-level information disclosure risk. | Truncate to `req.sessionID?.slice(0, 8) + '…'` in log output. |
| **Minor** | Modern JS | server.js:354 | `configStore.ensureInitialized().then(() => next()).catch(next)` — a `.then()/.catch()` chain inside `app.use()`. Breaks the async/await style used everywhere else. | `app.use(async (req, res, next) => { try { await configStore.ensureInitialized(); next(); } catch (err) { next(err); } });` |
| **Minor** | Modern JS | server.js:662,681,983,1270 | `require()` calls **inside route handlers** (e.g. `const _appEvents = require('./services/appEventService')` at line 983). Node caches these so there is no runtime cost after first call, but it obscures the module's dependency graph and looks like a per-request load. | Move all `require()` calls to the top of the file with the other imports. |
| **Major** | Maintainability | server.js:980–1535 | `POST /api/mcp/tool` handler is **~555 lines** and mixes: token resolution, MCP authorize gate, token introspection, remote MCP dispatch, HTTP/2 dispatch, auth-challenge interception, local fallback (three separate code paths), and SSE event publication. This is the largest single maintainability risk in the codebase. | Extract into named async functions: `resolveToolToken()`, `runMcpAuthorizeGate()`, `runTokenIntrospection()`, `dispatchMcpTool()`. Each becomes independently testable. |
| **Minor** | Maintainability | server.js:249,286 | Rate limiter max values `20000`, `8000`, `500`, `300` are inline magic numbers. | Define named constants at the top of the rate-limit section: `const DEV_GLOBAL_LIMIT = 20000; const PROD_GLOBAL_LIMIT = 8000;` etc. |
| **Minor** | Maintainability | server.js:260–276 | `shouldSkipGlobalRateLimit()` contains a list of hardcoded path strings that will silently diverge from actual routes as new endpoints are added. No test guards it. | Move to a named constant `const RATE_LIMIT_SKIP_PATHS = new Set([…])` defined alongside the limiter, or generate it from route registrations. |
| **Minor** | Maintainability | server.js:456–495 | Logout logic (`GET /api/auth/logout`) and the route-level `GET /logout` in `routes/oauth.js` are near-duplicates: both revoke tokens, destroy session, clear auth cookie, and redirect to PingOne signoff. Minor but dangerous divergence: `server.js` uses `configStore.getEffective(…)` for envId/region while `oauth.js` reads stale `process.env.*` directly. | Shared helper `buildSignoffRedirect(req, res, tokens)` eliminates the duplication and the env-read inconsistency. |
| **Minor** | Maintainability | server.js:647–655 | The inline admin-role check for `/api/mcp/audit` is ad-hoc session logic duplicated from the `requireSession`/`authenticateToken` pattern. | Extract `requireAdminSession` middleware and reuse across the three or four places this pattern appears. |

## Batch 02 — Part 2: transactions.js, tokenChain.js, oauth.js routes

| Severity | Category | File:Line | Issue | Fix Snippet |
|----------|----------|-----------|-------|-------------|
| **Major** | Async Patterns | transactions.js:152–173 | `POST /consent-challenge` is an `async (req, res)` handler with **no outer try/catch**. `restoreAccountsFromSnapshot()` is awaited — if it throws beyond its internal warn-and-return, the rejection propagates unhandled (Express 4). | Add outer `try { … } catch (err) { next(err); }` and include `next` in handler signature. |
| **Major** | Async Patterns | transactions.js:176–192 | `POST /consent-challenge/:challengeId/confirm` — async handler, `txConsent.confirmChallenge()` awaited. **No outer try/catch.** | Same fix as above. |
| **Minor** | Async Patterns | tokenChain.js:10,17 | `getTokenChain()` and `getMCPToolCalls()` are awaited **sequentially** but are independent. Sequential await adds unnecessary latency. | `const [tokenChain, mcpToolCallsChain] = await Promise.all([getTokenChain(req.user.id), getMCPToolCalls(req.user.id)]);` |
| **Minor** | Async Patterns | oauth.js:499–503 | `POST /refresh` calls `req.session.save((err) => { if (err) console.error(…); })` then **immediately** returns `res.json(…)` without awaiting the save. If the session store fails after the response is sent, the refreshed token is lost; the next request will see the old (possibly expired) token. | Await the save before responding, or document the optimistic pattern explicitly. |
| **Critical** | Security | transactions.js:343 | **Type mismatch in balance check**: `fromAccount.balance < amount` compares a numeric balance against `amount`, which is the original `req.body`-destructured value (a **string**). `req.body.amount` was mutated on line 285, but the destructured `const { …, amount, … }` binding (line 254) was **not updated**. JS coerces the string for `<` in most cases, but: (a) the check is semantically wrong — it should compare against the validated numeric `parsedAmount`, and (b) non-numeric input that somehow passed earlier validation could produce `NaN < balance === false`, silently allowing the transaction to proceed. | Replace `amount` with `parsedAmount` on line 343: `if (fromAccount.balance < parsedAmount)`. |
| **Minor** | Security | transactions.js:227–245 | **IDOR check — PASS.** `GET /:id` verifies `transaction.userId !== req.user.id` before returning data. Non-admin users cannot read another user's transaction by guessing an ID. | — |
| **Minor** | Security | transactions.js:252–609 | Amount validation is thorough: `parseFloat`, `Number.isFinite`, positive check, $1M cap, configStore hard limit, 2-decimal rounding. Ownership enforcement for both `fromAccountId` and `toAccountId` is present for non-admin. CLEAN. | — |
| **Minor** | Security | tokenChain.js:1–53 | No raw token values logged or returned at the route level. CLEAN. | — |
| **Minor** | Security | oauth.js:183–196 | **State parameter — PASS.** Validated against session value with PKCE cookie fallback. Mismatch triggers auto-retry redirect with structured logging. | — |
| **Minor** | Security | oauth.js:80–81,216–228 | **Nonce — PASS.** Generated via `crypto.randomBytes(16)`, stored in session + PKCE cookie, validated against ID token claim on callback. Mismatch triggers error redirect and `logAppEvent`. | — |
| **Minor** | Security | oauth.js:66–67,199,213 | **PKCE verifier — PASS.** Generated, stored in session + PKCE cookie, passed to `exchangeCodeForToken()`. | — |
| **Minor** | Security | oauth.js:291–297 | Session regenerated before token storage (session fixation prevention). Failure is fatal — login aborted. CLEAN. | — |
| **Minor** | Security | oauth.js:382–384 | `GET /logout` calls `oauthService.revokeToken()` twice **without `.catch()`**. Same pattern as the server.js logout — unhandled rejection risk. | `.catch(err => console.warn('[oauth/logout] revoke error:', err.message))` on each call. |
| **Minor** | Modern JS | transactions.js:285,343 | `req.body.amount` is mutated in place after destructuring (`req.body.amount = Math.round(…)`), but the destructured `amount` binding (line 254) is stale. This shadowed-mutation pattern is the root cause of the Critical type-mismatch finding. | Avoid mutating `req.body`. Use `const normalizedAmount = Math.round(parsedAmount * 100) / 100;` throughout. |
| **Minor** | Modern JS | transactions.js:534,581 | `console.log` lines include emoji (`💰`). Emoji in log lines can cause encoding issues in some log aggregators and is inconsistent with the rest of the codebase's plain-text log style. | Replace with `[Transaction]` prefix. |
| **Minor** | Maintainability | transactions.js:252–609 | `POST /` is **357 lines long**, mixing seven concerns: input validation, account re-hydration, HITL consent gate, session check, step-up MFA gate, Authorize policy evaluation, and transfer/deposit/withdrawal execution. | Extract: `validateTransactionInput()`, `runHitlConsentGate()`, `runStepUpGate()`, `runAuthorizePolicyGate()`, `executeTransfer()`, `executeTransaction()`. |
| **Minor** | Maintainability | transactions.js:395–434 | Step-up MFA gate logic is **repeated twice** (once for `STEP_UP_WITHDRAWALS_ALWAYS`, once for the general threshold check) with near-identical `res.status(428).json(…)` payloads differing only in message and `amount_threshold`. | Extract `buildStepUpResponse(description, acr, method, threshold, isHITL)` and call from both branches. |
| **Minor** | Maintainability | oauth.js:375–413 vs server.js:456–495 | Logout logic is **near-duplicated** across two files. If config is changed at runtime, `oauth.js` logout will use the wrong PingOne signoff endpoint. | Shared `buildSignoffRedirect(req, res, tokens)` helper eliminates duplication; `oauth.js` should use `configStore.getEffective(…)` consistently. |
| **Minor** | Maintainability | transactions.js:505–549,583–597 | `sendTransactionConfirmation()` is called inside an unnecessary `{}` block in both the transfer and non-transfer paths.
