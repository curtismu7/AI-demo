<objective>
Wire PingOne MFA (`/api/auth/mfa/challenge`) into the Phase 174 step-up modal so that OTP verification goes through PingOne's deviceAuthentications API instead of the current client-only stub.

Purpose: The current `handleOtpSubmit` just retries the action without actually verifying the OTP with PingOne. This plan makes MFA real by using the existing `mfaService.js` + `routes/mfa.js` backend.

Strategy: When `step_up_method=p1mfa` (from backend 428 response or configStore), the modal initiates a PingOne deviceAuthentication, shows the user's enrolled devices, collects OTP (or triggers FIDO2/push), and only retries the action after PingOne returns COMPLETED. The current stub behavior is preserved for `step_up_method=email` as a fallback.

Output: OtpStepUpModal enhanced with P1MFA multi-step flow, BankingAgent wired to initiate/complete P1MFA challenges.
</objective>

<!-- P1MFA Backend API (already implemented in routes/mfa.js) -->

POST /api/auth/mfa/challenge
  → { daId: string, status: 'DEVICE_SELECTION_REQUIRED', devices: Array<{ id, type, ... }> }

PUT /api/auth/mfa/challenge/:daId
  Body: { deviceId }        → select device → { daId, status: 'OTP_REQUIRED'|'ASSERTION_REQUIRED'|'PUSH_CONFIRMATION_REQUIRED', completed: false }
  Body: { deviceId, otp }   → submit OTP    → { daId, status: 'COMPLETED', completed: true }
  Body: { assertion }       → submit FIDO2  → { daId, status: 'COMPLETED', completed: true }

GET /api/auth/mfa/challenge/:daId/status
  → { daId, status, completed, publicKeyCredentialRequestOptions? }

<!-- PingOne MFA Status Flow -->
DEVICE_SELECTION_REQUIRED → (select device) → OTP_REQUIRED | ASSERTION_REQUIRED | PUSH_CONFIRMATION_REQUIRED
OTP_REQUIRED → (submit OTP) → COMPLETED | FAILED
ASSERTION_REQUIRED → (submit assertion) → COMPLETED | FAILED
PUSH_CONFIRMATION_REQUIRED → (poll status) → COMPLETED | PUSH_CONFIRMATION_TIMED_OUT

<!-- On COMPLETED: backend sets req.session.stepUpVerified = Date.now() + 5min TTL -->

Allowed values: 'email' (current stub), 'ciba', 'p1mfa' (new PingOne MFA)

Refactor OtpStepUpModal to support two modes controlled by a `mode` prop:

**mode="stub" (default — preserves current behavior exactly):**
- Current OTP input → onSubmit(otp) → done. No changes.

**mode="p1mfa" (new PingOne MFA flow):**
- Accept new props: `daId`, `devices`, `apiBase`, `onP1MfaComplete`, `onP1MfaError`
- Internal state machine: `step` = 'pick-device' | 'otp' | 'push' | 'fido' | 'error'
- **Step: pick-device** — Show list of enrolled devices (name, type icon). User clicks one → PUT `/api/auth/mfa/challenge/:daId` with `{ deviceId }` → transition to next step based on response status.
- **Step: otp** — Show current OTP input UI (reuse existing). On submit → PUT `/api/auth/mfa/challenge/:daId` with `{ deviceId, otp }`. On completed=true → call `onP1MfaComplete()`. On error → show inline error, allow retry.
- **Step: push** — Show "Push notification sent to your device" with spinner. Poll `GET /api/auth/mfa/challenge/:daId/status` every 3 seconds (max 60s). On completed → `onP1MfaComplete()`. On timeout → show error with "Try another method" button.
- **Step: fido** — Show "Use your passkey" prompt. Use `publicKeyCredentialRequestOptions` from status response to call `navigator.credentials.get()`. On success → PUT with `{ assertion }`. On completed → `onP1MfaComplete()`.
- **Step: error** — Show error message with "Try another method" (returns to pick-device) and Cancel button.

For all P1MFA API calls, use `fetch` with `credentials: 'include'` (session cookie). API base from `process.env.REACT_APP_API_URL || ''`.

Keep all existing CSS classes. Add minimal new classes for device picker only:
- `.otp-step-up-modal__device-list` — flexbox column
- `.otp-step-up-modal__device-item` — button per device (border, hover, icon + label)

Error handling: On any fetch failure (network, 410 challenge_expired, 401 session_expired), transition to 'error' step with appropriate message. On 410 (challenge expired), message should say "MFA session expired — please try again".

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Frontend → BFF MFA routes | User-supplied OTP/assertion crosses trust boundary |
| BFF → PingOne deviceAuthentications API | Worker token + user token used for P1MFA calls |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-174-04-01 | Spoofing | OTP submission | mitigate | OTP validated server-side by PingOne, not client-side. Session cookie with httpOnly flag prevents token theft. |
| T-174-04-02 | Tampering | Device selection | mitigate | Device IDs come from PingOne response; PUT validates deviceId belongs to the user's deviceAuthentication session server-side. |
| T-174-04-03 | Information Disclosure | Error messages | mitigate | Frontend shows generic "MFA failed" messages, not PingOne error internals. Console.error for debugging only. |
| T-174-04-04 | Denial of Service | Push polling | mitigate | Poll capped at 60s with 3s interval (max 20 requests). No retry loop on failure. |
| T-174-04-05 | Elevation of Privilege | stepUpVerified bypass | accept | stepUpVerified set server-side only on PingOne COMPLETED status. 5-min TTL. Client cannot forge this. |
</threat_model>
