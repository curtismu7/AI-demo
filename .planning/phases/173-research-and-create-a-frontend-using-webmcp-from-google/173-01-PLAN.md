  <files>banking_api_ui/src/services/webMcpClient.js</files>
  <action>
Create `banking_api_ui/src/services/webMcpClient.js` following the patterns from `mcpFlowSseClient.js`.

The service provides three exports:

1. **`listMcpTools()`** — `async function` that fetches `GET /api/mcp/inspector/tools` with `credentials: 'include'`. Returns `{ tools: [...] }` on success, throws on HTTP error with status and body.

2. **`callMcpTool(toolName, params, flowTraceId)`** — `async function` that POSTs to `/api/mcp/tool` with body `{ tool: toolName, params, flowTraceId }` and `credentials: 'include'`. Returns the JSON response.

3. **`openMcpToolStream(traceId, onEvent)`** — Re-export/wrapper around the pattern from `mcpFlowSseClient.js`: opens `EventSource` to `/api/mcp/tool/events?trace=<traceId>`, calls `onEvent(data)` for each message, auto-closes on `stream_end` phase. Returns disconnect function.

The service should:
- Use `fetch` (not axios) for consistency with mcpFlowSseClient.js patterns
- Include proper error extraction (read response body on non-OK status)
- Export named functions (ESM, matching CRA convention)
- Be ~50-70 lines total — thin wrapper over existing BFF endpoints

Per D-02: Use WebSocket transport language in the comments. Per research: the actual transport to MCP server is WebSocket (handled by BFF); the browser communicates with BFF via HTTP+SSE which is the practical browser pattern.
  </action>
