Each segment of a token chain is tagged with its path; the chain renders with a path-specific colour/badge so the user sees three visibly different chains as they exercise the three demo prompts.

**R2 narrative update:** the dual_token tokenEvents now describe a real outbound call to `/api/resource-server/identity` (the gateway no longer terminates dual_token at the marker — it forwards to a real backend, validates the bearer, decodes id_token claims server-side, and returns the payload). The Token Chain narrative reflects this by including segments for "bearer validated" and "id_token decoded server-side at banking_resource_server." Plan 01 emits these synthesized events; this plan ensures they render correctly.

Output: TokenChainContext schema extended with credentialPath; TokenChainDisplay renders path-aware segments; ActivityLogs has new event categories; FloatingTokenChainPanel + TokenChainModal pass field through.

TokenChainContext exposes:
```javascript
setTokenEvents(tool, events)  // events: Array<{ id, label, status, decoded, explanation, ... }>
```
Each event currently has no credentialPath field. We're adding it as an optional field per segment, defaulting to 'oauth_bearer' when absent (backwards compatible — existing callers continue to work).

EVENT_CATEGORIES (appEventService.js:14): currently OAUTH, TOKEN_EXCHANGE, SESSION, JWKS, MCP, AUTH_LIFECYCLE, AGENT, AUTHORIZE, AGENT_PROMPT, DELEGATION, INTROSPECTION, HELIX. We add GATEWAY_PATH.

Banking Agent Service note: `bankingAgentService.callMcpTool` writes the events to TokenChainContext via setTokenEvents (see existing wiring). It receives the gateway response and currently does not look at `_meta.credentialPath`. We will modify it to read `_meta.credentialPath` and stamp it on each tokenEvent passed to setTokenEvents.

**R2 narrative for dual_token (from Plan 01 Task 2):** the gateway emits a 4-segment chain:
- `evt-inbound` — Inbound user bearer received
- `evt-idtoken-fetch` — id_token fetched from BFF session (server-to-server)
- `evt-bearer-validated` — banking_resource_server: bearer validated (authenticateToken middleware)
- `evt-idtoken-decoded` — banking_resource_server: id_token decoded server-side, claims returned (no raw JWT)

All four segments carry `credentialPath: 'dual_token'`. The TokenChainDisplay must render all of them with the teal visual identity.
