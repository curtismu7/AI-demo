<objective>
Handle MCP spec `notifications/cancelled` messages so that when the MCP server signals cancellation of a long-running tool call, the corresponding `_pending` future is resolved with `asyncio.CancelledError` rather than leaking. Also add an explicit `asyncio.CancelledError` guard in `MCPTool._arun` so cancellation propagates correctly instead of being swallowed by the bare `except Exception`.

Purpose: Prevent silent resource leaks in the `_pending` dict when the MCP server cancels tool calls, and ensure asyncio task cancellation works correctly through the agent tool invocation stack.

Output: Modified `connection.py` with a `notifications/cancelled` branch in `_read_loop`; modified `mcp_tool_provider.py` with a `CancelledError` guard; three new tests in `test_mcp_connection_demux.py`.
</objective>

From `langchain_agent/src/mcp/connection.py`:

```python
# _pending: Dict[Any, asyncio.Future]  — keyed by JSON-RPC id (str uuid4)
# Already cleaned up on close via:
#   - disconnect() → _fail_all_pending(MCPConnectionClosedError(...))
#   - _read_loop() finally → _fail_all_pending(MCPConnectionClosedError(...))
# NOT yet handled:
#   - notifications/cancelled frame → params.requestId lookup → CancelledError

def _fail_all_pending(self, error: Exception) -> None:
    pending = self._pending
    self._pending = {}
    for fut in pending.values():
        if not fut.done():
            fut.set_exception(error)
```

Current `_read_loop` id-less branch (line ~286):
```python
msg_id = frame.get("id") if isinstance(frame, dict) else None
if msg_id is None:
    logger.debug(f"Unsolicited/notification frame from ... (no id) — dropped")
    continue
```

`notifications/cancelled` carries no `"id"` at the top level (it is a server-to-client notification). Its structure per MCP spec `2025-03-26`:
```json
{
  "jsonrpc": "2.0",
  "method": "notifications/cancelled",
  "params": { "requestId": "<id-of-the-request-being-cancelled>", "reason": "..." }
}
```

The `requestId` in `params` matches the `id` that was placed in `self._pending` when `_send_request` registered the future.

From `langchain_agent/src/agent/mcp_tool_provider.py`:

The bare `except Exception as e:` at the bottom of `MCPTool._arun` (line ~506) catches `asyncio.CancelledError`. In Python 3.8+, `CancelledError` is a subclass of `BaseException` rather than `Exception` — but in earlier versions it was `Exception`. Because the repo targets Python 3.8+ the safe fix is to add an explicit `except asyncio.CancelledError` guard BEFORE the bare `except Exception` block, return a friendly string, and re-raise so the asyncio task is actually cancelled (not just swallowed).

From `langchain_agent/tests/test_mcp_connection_demux.py`:

`FakeReorderingWebSocket` is the correct test harness. Its `push(frame: dict)` method enqueues frames that `_read_loop` will `recv()`. The `_connected(conn, ws)` helper bypasses handshake and attaches the fake socket with `_start_reader()`. Use this pattern for all three new tests.

`MCPConnectionClosedError` and `MCPRequestTimeoutError` are both importable from `src.mcp.connection`.
