### 2. Email Input + Provision Button Row

**Layout:** `flex-direction: row`, `align-items: flex-end`, `gap: 0.75rem`, `flex-wrap: wrap`.

```
<div className="demo-data-provision-row">
  <label className="demo-data-field demo-data-provision-row__field">
    <span>New user email</span>
    <input
      type="email"
      className="demo-data-input"
      placeholder="demo@example.com"
      value={provisionEmail}
      onChange={...}
      disabled={provisioning}
      autoComplete="email"
    />
  </label>
  <button
    type="button"
    className="demo-data-btn primary demo-data-provision-row__btn"
    onClick={handleProvision}
    disabled={provisioning || !provisionEmail.trim()}
  >
    {provisioning ? 'Provisioning…' : 'Provision'}
  </button>
</div>
```

**New CSS classes:**

```css
/* Row wrapper — input + button side by side */
.demo-data-provision-row {
  display: flex;
  align-items: flex-end;
  gap: 0.75rem;
  flex-wrap: wrap;
  margin-bottom: 1rem;
}

/* Email field takes remaining space, min 240px */
.demo-data-provision-row__field {
  flex: 1 1 240px;
  min-width: 0;
  margin-bottom: 0;   /* override default .demo-data-field margin when in flex row */
}

/* Button aligns to bottom of row (same height as input at 0.55rem padding) */
.demo-data-provision-row__btn {
  flex-shrink: 0;
  align-self: flex-end;
  margin-bottom: 0;   /* remove any inherited block margin */
}
```

**Interaction states:**
- Default: button enabled when `provisionEmail.trim()` is non-empty and `!provisioning`.
- Disabled (no email entered): `disabled` attribute set; `.demo-data-btn.primary:disabled` rule applies — opacity 0.5, cursor not-allowed. Already defined in DemoDataPage.css.
- Disabled (provisioning in progress): same disabled visual. Button label changes to "Provisioning…".
- Input disabled during provisioning: add `disabled` attribute; browser default disabled styling is acceptable.

### 3. Step Log

Rendered below the input row. Visible immediately when provisioning starts (even before first step resolves). Hidden entirely when `provisionSteps` array is empty (initial state, before first Provision click).

**Steps (from CONTEXT.md D-12):**
1. "Create PingOne user"
2. "Set may_act attribute"
3. "Enroll email OTP MFA"

**Step status values:** `pending` | `running` | `success` | `error` | `skipped`

**Icon map:**
| Status | Icon | Color |
|--------|------|-------|
| pending | `○` (U+25CB) | `#9ca3af` (muted gray) |
| running | `…` (ellipsis) | `#6b7280` |
| success | `✓` | `#16a34a` |
| error | `✗` | `#b91c1c` |
| skipped | `⚠` | `#92400e` |

**Layout:**

```
<ol className="demo-data-provision-steps" aria-live="polite" aria-label="Provisioning steps">
  {steps.map(step => (
    <li key={step.name} className={`demo-data-provision-step demo-data-provision-step--${step.status}`}>
      <span className="demo-data-provision-step__icon" aria-hidden="true">{iconFor(step.status)}</span>
      <span className="demo-data-provision-step__label">{step.label}</span>
      {step.detail && (
        <span className="demo-data-provision-step__detail">{step.detail}</span>
      )}
    </li>
  ))}
</ol>
```

**New CSS classes:**

```css

.demo-data-provision-step__icon {

  flex: 1;
}

.demo-data-provision-step__detail {
  font-size: 0.78rem;
  color: var(--app-text-muted, #6b7280);

/* Per-status color overrides */
```

**Error detail display (CONTEXT.md D-14):**
- The `.demo-data-provision-step__detail` renders the error message from the BFF step result.
- For skipped steps: detail text is "Skipped — depends on failed step above."
- For warning (may_act not configured, CONTEXT.md D-09): status stays `success` with detail
  "may_act not set — MCP token exchanger client ID not configured". Icon is `✓` but label color
  remains default (not green), detail uses `#92400e` warning color via inline style or modifier class.

**Warning detail modifier:**

```css

.demo-data-provision-step--warn .demo-data-provision-step__detail { color: #92400e; }
```

## Interaction Contract

| State | Button | Step log | Credential card |
|-------|--------|----------|----------------|
| Initial (no email) | disabled | hidden | hidden |
| Email entered, idle | enabled | hidden | hidden |
| Provisioning in progress | disabled + "Provisioning…" | visible, steps animate pending→running→success/error | hidden |
| All steps succeeded | enabled (can re-provision) | visible, all ✓ | visible |
| Any step errored | enabled (can retry) | visible, shows ✗ on failed step + ⚠ skipped on dependents | hidden |
| Partial success (may_act warn) | enabled | visible, step 2 shows ✓ with warning detail | visible |

**Re-provision:** clicking Provision again resets `provisionSteps` to empty and `provisionResult` to null,
then starts a fresh run. The old credential card disappears immediately on click.

**Form validation:** only `type="email"` HTML5 validation. No custom validation UI required.
