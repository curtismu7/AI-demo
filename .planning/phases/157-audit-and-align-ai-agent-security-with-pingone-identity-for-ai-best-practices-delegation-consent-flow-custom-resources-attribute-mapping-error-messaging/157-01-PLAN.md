**Area 4: Token Exchange Flow (RFC 8693)**

Trace the COMPLETE token flow in the codebase:
1. User sign-on → where is the user token stored? (session? cookie?)
2. How does the agent get its own token? (client_credentials?)
3. Where is token exchange called? (server.js endpoint?)
4. What parameters are sent? (subject_token, actor_token, resource, audience?)
5. Does the exchanged token contain `act` claim?
6. Does the user token contain `may_act` claim?
7. How is the exchanged token used to call MCP server?

Map each step to PingOne guide requirements. Note any deviations.

**Area 5: Error Handling & Messaging**

Check the 6 error scenarios from CONTEXT.md §6:
1. Token exchange fails, act claim null (may_act missing)
2. Token exchange fails (agent ID mismatch)
3. Agent can list tools but not call them (wrong scope)
4. User doesn't see consent prompt (policy issue)
5. Delegated token rejected by backend (act claim malformed)
6. User sends user token to agent endpoint (scope mismatch)

For each: does the current code catch this error? Is the error message educational? Reference Phase 156 work.

**Area 6: Configuration & Scopes**

Audit:
- What scopes are configured for the agent app?
- Are scopes stable across sessions?
- Are both 'agent' and 'test'/'banking' scopes assigned?
- How are scopes validated in middleware?

**Area 7: PingGateway or Equivalent MCP Protection**

Audit:
- Is there an API gateway in front of the MCP server?
- Does the MCP server validate act claims before processing?
- Is token validation done server-side before tool execution?
