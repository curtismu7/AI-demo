# PingAuthorize is the sole authoritative MCP tool gate in the BFF; no local scope-policy decision

**Status:** accepted (2026-05-15)

The BFF LangGraph agent previously ran two authorization checks on MCP tool calls: the authoritative external one (`mcpToolAuthorizationService.evaluateMcpFirstToolGate` at `server.js:~1535`, which asks PingAuthorize / simulated and acts on PERMIT/DENY/HITL on every call), and a redundant *local* scope→tool decision (`agentMcpScopePolicy.isToolPermittedByAgentPolicy`, consulted during token acquisition in `agentMcpTokenService.js`). The local check duplicated authorization policy inside the agent, created a hand-maintained second source of truth that drifted from PingAuthorize and from the MCP server registry, and was authoritative-looking enough that a future reader would extend it rather than the real policy. This violates ARCHITECTURE-TRUTHS T-2 (the authorization decision is external and authoritative; the agent owns intent/orchestration, not authorization).

**Decision (remediation R1):** remove the local authorization-*decision* role entirely. Code tracing during implementation showed `agentMcpScopePolicy`'s only runtime consumer was that redundant veto (the agent's advertised tool list comes from the gateway `tools/list`, not from the local map), so the module `services/agentMcpScopePolicy.js` was deleted rather than retained as catalog data. `evaluateMcpFirstToolGate` (PingAuthorize) is now the single authoritative gate. The unrelated `MCP_TOOL_SCOPES` map in `mcpWebSocketClient.js` survives only as RFC 8693 exchange-scope/advertisement data and is annotated explicitly non-authoritative.

**Consequences:**
- One source of authorization truth (PingAuthorize). No local policy drift possible. This resolved BFF code-review findings WR-01 (scope-map drift, downgraded — the map is no longer a security boundary) and WR-02 (duplicate scope entries, moot — the duplicated array lived only in the deleted module).
- Anyone reintroducing a local scope→tool authorization decision in the BFF is reintroducing the smell. REGRESSION_PLAN.md §1 codifies "PingAuthorize (`evaluateMcpFirstToolGate`) is the SOLE authoritative MCP tool gate in the BFF; `agentMcpScopePolicy` must never return as an authz decision." A regression test asserts PingAuthorize is authoritative (simulated DENY blocks / PERMIT allows a tool the old local map would have vetoed).
- Implemented in commits `8faf114e`, `0fffd3a1`, `52f5044b`, `e62d80e0`. Supersedes the standalone note at `docs/architecture-notes/2026-05-15-agent-local-authz-smell.md` (kept for the investigation trail and the corrected-premise record).
- This ADR is about the **BFF** path specifically. The other agents (banking_agent_service, langchain_agent) reach tools through the gateway, which independently asks PingAuthorize (see ARCHITECTURE-TRUTHS T-2). Same principle, different enforcement point.
