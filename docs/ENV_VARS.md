# Environment Variables ŌĆö Source of Truth

> **This file is the authoritative reference for every environment variable used by the banking demo.**
> Keep it in sync with `demo_api_server/.env.example` and `docs/PINGONE_CONFIG.md`.
>
> **SoT hierarchy:**
> ```
> docs/PINGONE_CONFIG.md          ŌåÉ PingOne entity IDs, audiences, token policies
>       Ōåō bootstrap writes
> demo_api_server/.env            ŌåÉ runtime values (gitignored, never commit)
>       Ōåō seeded into
> demo_api_server/data/persistent/lmdb/   ŌåÉ survives .env loss, encrypted at rest
>       Ōåō resolved by
> configStore.getEffective(key)   ŌåÉ LMDB ŌåÆ .env ŌåÆ FIELD_DEFS default
>       Ōåō overridden via
> /setup page (admin UI)          ŌåÉ writes back to LMDB; highest runtime priority
> ```

---

## How variables are resolved

`configStore.getEffective(key)` is the **only** correct way to read config in route handlers and services. Never use `process.env.X` directly for feature config ŌĆö it bypasses LMDB overrides set via the `/setup` page.

**Resolution order (highest ŌåÆ lowest):**
1. LMDB (values saved via `/setup` admin UI or `configStore.setConfig()`)
2. `.env` file (`process.env` at startup)
3. `FIELD_DEFS` default in `configStore.js`

**Bootstrap-allowlisted keys** (`SESSION_SECRET`, `CONFIG_ENCRYPTION_KEY`, `PINGONE_ENVIRONMENT_ID`, management credentials) are always read from `.env` ŌĆö never cached in LMDB. They are needed before the vault unlocks and must never be stale.

**Canonical configStore key names** are lowercase (e.g. `pingone_mcp_exchanger_client_id`). Env var names are UPPER_SNAKE_CASE. The `envFallbackMap` in `configStore.js` maps every env var to its canonical key, including legacy aliases for backward compatibility.

## Naming conventions

All env vars follow these rules:

| Rule | Example |
|---|---|
| `PINGONE_` prefix for PingOne entity credentials | `PINGONE_ADMIN_CLIENT_ID` |
| `PINGONE_RESOURCE_` prefix for PingOne resource server audiences | `PINGONE_RESOURCE_AGENT_GATEWAY_URI` |
| `MCP_GW_` prefix for MCP Gateway-specific config | `MCP_GW_CLIENT_ID` |
| `HELIX_` prefix for Helix LLM config | `HELIX_API_KEY` |
| `OLLAMA_` prefix for Ollama config | `OLLAMA_BASE_URL` |
| `ff_` prefix (lowercase) for feature flags | `ff_admin_token_exchange` |
| `DEMO_` prefix for demo user credentials | `DEMO_USER_USERNAME` |
| Audience variables end in `_URI` or `_AUDIENCE` | `PINGONE_RESOURCE_MCP_GATEWAY_URI`, `ENDUSER_AUDIENCE` |
| Auth method variables end in `_AUTH` or `_AUTH_METHOD` | `PINGONE_ADMIN_TOKEN_ENDPOINT_AUTH` |

## Startup validation

`run.sh` performs pre-flight checks before launching services:
- Warns if `HELIX_API_KEY` is unset (NL agent will fall back to heuristic)
- Warns if `OLLAMA_BASE_URL` remote host is unreachable
- Exits 1 if vault is present but `VAULT_PASSWORD` is unset

Each service performs its own startup validation:
- `demo_hitl_service`: warns if `HITL_ALLOWED_ORIGINS` is empty
- `demo_mortgage_service`: warns if `MORTGAGE_SERVICE_API_KEY` equals the default placeholder
- `demo_mcp_invest`: warns if `MCP_SERVER_RESOURCE_URI` is unset

See `docs/superpowers/specs/2026-05-24-startup-hardening-design.md` for the full startup sequence and health check design.
