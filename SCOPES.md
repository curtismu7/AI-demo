# Scope reference — Banking demo

> Generated by inspecting every `requireScopes(...)` call in `banking_api_server/routes/`
> and `banking_api_server/middleware/scopeEnforcement.js`.
> Update when you add or rename routes.

This file documents what each PingOne scope actually unlocks in the running app.
Tokens carry scopes; the BFF middleware enforces them at route entry. Apps without
the right scope get **403** with `error_description: insufficient_scope`.

---

## Where the scopes live in PingOne

After `npm run setup:fresh`, all custom scopes are defined on three resource servers:

| Resource server         | Audience              | Holds scopes…           |
|-------------------------|-----------------------|--------------------------|
| `Super Banking API`      | `banking_api_enduser` | banking + admin + users + ai_agent |
| `Super Banking MCP Server` | `https://mcp-server.pingdemo.com` | admin + users + banking (subset) |
| `Super Banking MCP Gateway` | `api.ping.demo` | banking:mcp:invoke |

Scope grants are wired to the OIDC apps as part of provisioning (see `setup-config.md`
after a run for the per-app grant inventory).

## Scope catalog

### `banking:read`
**What it unlocks:** read-only access to banking data — accounts, transactions, balances, user lookups.

| Route | Method | Used to |
|---|---|---|
| `/api/accounts` | GET | List user's own accounts |
| `/api/accounts/:id` | GET | Get account details |
| `/api/accounts/:id/balance` | GET | Read current balance |
| `/api/transactions` | GET | Admin: list all transactions |
| `/api/transactions/:id` | GET | Get a transaction |
| `/api/users` | GET (admin) | List all users |
| `/api/users/:userId` | GET | Get user profile (own or admin) |
| `/api/users/search/:query` | GET (admin) | Search users |

**UI pages that need this:** `/dashboard` (end-user), `/admin` (admin user list),
`/admin/transactions` (all-tx view), `/admin/scope-explorer` (introspection).

**Customer apps assigned this scope:** `Super Banking User App`, `Super Banking Admin App`,
`Super Banking MCP Server`.

### `banking:write`
**What it unlocks:** mutating banking operations — transactions, account create/update/delete, user create/update/delete.

| Route | Method | Used to |
|---|---|---|
| `/api/transactions` | POST | Make a deposit / transfer / withdrawal / payment |
| `/api/transactions/:id` | PUT | Update transaction (admin only via `requireAdmin`) |
| `/api/transactions/:id` | DELETE | Delete transaction (admin only) |
| `/api/accounts` | POST | Create new account (admin / non-delegate) |
| `/api/accounts/:id` | PUT | Update account |
| `/api/accounts/:id` | DELETE | Delete account |
| `/api/accounts/reset-all-demo` | POST | Wipe demo data (admin tool) |
| `/api/users` | POST | Create user (admin) |
| `/api/users/:userId` | PUT | Update user profile (own or admin) |
| `/api/users/:userId` | DELETE | Delete user (admin) |

**UI pages that need this:** `/dashboard` deposit/transfer buttons, `/admin/users` user-edit form, `/admin/accounts` account-management.

**Note:** holders of `banking:write` who happen to be `bankDelegate` have a SECOND gate
(`requireNotBankDelegate`) — they get 403 even with the scope. Demo policy: delegates
can only deposit, not transfer/withdraw/etc. See [bankDelegate restrictions](#delegated-users-bankdelegate).

**Customer apps assigned this scope:** `Super Banking User App`, `Super Banking Admin App`.

### `banking:admin`
**What it unlocks:** administrative actions outside of the account/user lifecycle — currently dynamic OIDC client registration.

| Route | Method | Used to |
|---|---|---|
| `/api/admin/clients/register` | POST | RFC 7591 dynamic client registration (admin tool) |
| `/api/admin/clients` | GET | List dynamically-registered clients |

**UI pages:** `/admin/clients` dynamic-registration tester.

**Customer apps assigned this scope:** `Super Banking Admin App` only.

### `banking:accounts:read`, `banking:transactions:read`
**What it unlocks (planned):** finer-grained read access for ENDUSER-audience tokens. Currently the routes accept the broader `banking:read` instead — see `// ENDUSER_AUDIENCE is set, restore ...` comments in `routes/transactions.js`. These exist on the resource server for forward-compat but aren't actively enforced.

### `banking:accounts`
**What it unlocks (legacy):** "account access and management" — broader scope kept for backward-compat with older Postman collections. Not currently checked by any route.

### `banking:ai:agent:read` (formerly `banking:ai:agent`)
**What it unlocks:** the AI agent's read access when invoked via the MCP gateway. Required for the agent to call read endpoints on behalf of a user.

**Used by:** the LangChain agent + MCP gateway re-exchange flow. Server enforces via the MCP gateway scope check (not via `requireScopes` directly on banking routes — the agent's token already carries `banking:read` if the user delegated it).

**Customer apps assigned this scope:** `Super Banking MCP Server`, `Super Banking User App`.

### `banking:mcp:invoke`
**What it unlocks:** permission to invoke MCP tools through the gateway. Lives on the `Super Banking MCP Gateway` resource (audience `api.ping.demo`).

**Used by:** the agent service in its RFC 8693 token-exchange call. Audience-narrowed: only the MCP Gateway accepts tokens with this scope.

**Customer apps assigned this scope:** `Super Banking Agent`.

### `banking:sensitive` (planned)
**What it unlocks (planned):** PII-class operations (full SSN, full account numbers). Currently not checked by any route. Reserved.

### Admin namespace: `admin:read`, `admin:write`, `admin:delete`
**What it unlocks:** mirror of `banking:read` / `banking:write` but on the **`Super Banking MCP Server`** resource — used for admin tokens that target MCP tools. Same operations, different audience.

| Scope | Operation |
|---|---|
| `admin:read` | Read admin data via MCP tools |
| `admin:write` | Modify admin settings via MCP tools |
| `admin:delete` | Delete users / admin resources via MCP tools |

**Customer apps assigned this scope:** `Super Banking Admin App`, `Super Banking MCP Server`.

### Users namespace: `users:read`, `users:manage`
**What it unlocks:** PingOne user-directory operations — read user profiles, modify user accounts.

These are demo scopes; the BFF currently doesn't have routes that key off them
specifically. The Worker app gets `p1:read:user` / `p1:update:user` (PingOne-issued
scopes, not custom) via roles, which is what actually grants the BFF rights to call
PingOne Management API.

### `ai_agent`
**What it unlocks (legacy):** AI agent identity assertion — predates the more specific `banking:ai:agent:read`. Not currently enforced by any route.

## Delegated users (`bankDelegate`)

`bankDelegate` is a regular user with the `banking:read` + `banking:write` scopes
(same as `bankuser`), BUT carries an additional token claim `is_delegate: "true"`.

The `requireNotBankDelegate(...)` middleware checks that claim and returns 403 on:
- `POST /api/transactions` with `type: 'transfer' | 'withdrawal' | 'payment'`
- `POST /api/accounts` (create)
- `PUT /api/accounts/:id` (update)
- `DELETE /api/accounts/:id` (delete)
- `PUT /api/users/:userId` (profile change)

So `bankDelegate` ends up with **read-everywhere + deposit-only** even though their
*scopes* are the same as a regular user. The delegation check is in addition to scope
enforcement, not a replacement.

**Allowed for `bankDelegate`:**
- Read account balances, transactions, profile
- Make deposits

**Denied for `bankDelegate`:**
- Transfers, payments, withdrawals
- Account create/update/delete
- Profile edits

The delegation flag is wired via the `isDelegate` user attribute (CUSTOM STRING) →
SPEL token claim `${user.isDelegate}` emitted on the Super Banking API resource.

## How to test scope enforcement

```bash
# 1. Log in as bankuser, get the access token
# 2. Call a banking:write route
curl -X POST https://api.ping.demo:4000/api/transactions \
  -H "Authorization: Bearer <bankuser-token>" \
  -H "Content-Type: application/json" \
  -d '{"type":"deposit","toAccountId":"...","amount":50}'
# → 200 OK
# 3. Strip banking:write from the token (use a token from a client that doesn't have it)
# → 403 insufficient_scope: required banking:write
```

For the bankDelegate gate specifically:

```bash
# Same call as above but with bankDelegate's token → 200 (deposit allowed)
# Call POST /api/accounts with bankDelegate's token → 403 forbidden_for_delegate
```
